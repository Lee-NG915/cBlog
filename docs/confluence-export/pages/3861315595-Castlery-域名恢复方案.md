# Castlery 域名恢复方案

> **文档状态**: 待对齐

## 1. 背景 & 现状

`castlery.com` 域名被盗后，我们紧急启用了 `castlery.co` 作为临时域名。目前 `.co` 已全面上线，用户可正常访问和购买。

**现在面临的问题：** `.com` 域名即将归还给我们，我们需要一个**安全、有序**的方案将流量从 `.co` 切回 `.com`。

### 当前架构就绪情况

| 组件 | 是否支持 .com | 备注 |
| CloudFront (CDN) | ✅ | 已配置 .com 作为备用域名 |
| SSL 证书 (AWS 托管) | ✅ | `*.castlery.com` 通配符证书有效，自动续期正常 |
| K8s Ingress | ✅ | 已配置双域名路由 |
| 前端应用代码 | ❌ | 存在 `.co` 硬编码，需要改为环境变量驱动 |
| 后端 API 域名 | ❌ | 已切换至 `.co`（如 `apigw-sg-prod.castlery.co`），需要同步处理 |
| 产品 Feed 数据 | ❌ | 所有 Feed 中的产品 URL 已刷新为 `.co`，后续需重新刷新 |
| DNS 解析 | ❌ | 等待 `.com` 域名归还后配置 |


> **好消息**：基础设施层面（CloudFront、SSL、K8s）已经准备好了，核心工作在于代码改造、后端 API 同步、Feed 数据刷新和有序切换。

### 受影响的域名范围

切换不仅仅是 `www.castlery.com`，以下所有子域名都需要同步处理：

| 域名 | 用途 | 当前状态 |
| `www.castlery.com` | 主站 | 待恢复 |
| `apigw-sg-prod.castlery.com` | 后端 API 网关 (SG) | 已切至 `.co` |
| `apigw-us-prod.castlery.com` | 后端 API 网关 (US) | 已切至 `.co` |
| 其他 API / 内部子域名 | 各服务 | 需各团队确认清单 |

## 2. 核心策略：三阶段渐进式切换

我们**不会**一次性从 `.co` 切到 `.com`，而是分三个阶段渐进切换，确保**任何阶段出问题都可以立刻回滚，用户无感知**。

## 3. Phase 1：流量兜底（拿到域名后立刻执行）

### 3.1 目标

> 让所有访问 `castlery.com` 的用户（Google 搜索、历史书签、外部链接）能正常到达我们的网站，不会看到错误页面。

### 3.2 用户体验

```
sequenceDiagram
    participant U as 用户
    participant G as Google 搜索
    participant CF as CloudFront
    participant S as 网站 (.co)
    
    U->>G: 搜索 "Castlery sofa"
    G->>U: 显示 castlery.com/us/sofas
    U->>CF: 点击访问 castlery.com/us/sofas
    CF->>U: 302 临时跳转 → castlery.co/us/sofas
    U->>S: 自动跳转到 castlery.co/us/sofas
    S->>U: ✅ 正常显示页面
    
    Note over U,S: 用户几乎无感知，跳转发生在毫秒级
```

### 3.3 技术实现

在 CloudFront 上部署一个轻量级 Function，在 CDN 边缘直接处理跳转，不需要请求到达服务器：

> 具体看

```
// CloudFront Function - Phase 1
function handler(event) {
  var request = event.request;
  var host = request.headers.host.value;
  
  if (host.indexOf('castlery.com') !== -1) {
    // 保留完整路径和参数
    var qs = Object.keys(request.querystring).length > 0
      ? '?' + Object.keys(request.querystring).map(function(k) {
          var item = request.querystring[k];
          return item.multiValue 
            ? item.multiValue.map(function(v) { return k + '=' + v.value; }).join('&')
            : k + '=' + item.value;
        }).join('&')
      : '';
    
    var targetHost = host.replace('castlery.com', 'castlery.co');
    
    return {
      statusCode: 302,
      statusDescription: 'Found',
      headers: {
        location: { value: 'https://' + targetHost + request.uri + qs }
      }
    };
  }
  
  return request;
}
```

### 3.4 为什么用 302 而不是 301？

|  | 302 临时跳转 | 301 永久跳转 |
| **浏览器缓存** | ❌ 不缓存 | ✅ 会缓存 |
| **SEO 权重转移** | ❌ 不转移 | ✅ 转移 |
| **可逆性** | ✅ 随时可改 | ❌ 浏览器记住了很难改回来 |


Phase 1 用 302 是因为**我们后面要反转方向**（`.co → .com`），如果用 301，用户浏览器会缓存这个跳转，到 Phase 2 改方向后，部分用户还会按旧的跳转走，造成死循环。

### 3.5 执行清单

| # | 操作 | 负责人 | 预计耗时 |
| 1 | 在 DNS 注册商配置 `castlery.com` 及子域名指向 CloudFront / 各服务 (UK、CA证书重新生成) | DevOps | 10 min |
| 2 | 确认 DNS 生效（`dig www.castlery.com`） | DevOps | 5-30 min |
| 3 | 在主站 CloudFront 部署 302 跳转 Function | Frontend | 5 min |
| 4 | 验证跳转正常工作 | QA / Frontend | 10 min |
| 5 | 启用域名锁定 + 2FA | IT Security | 5 min |


> **Phase 1 总耗时约 30-60 分钟，用户影响为零。**

### 3.6 后端 API 域名处理

后端 API 和前端的处理策略**不同**。前端因为 Cookie 问题不能双域名并存，但 API 不存在这个问题，所以后端应该**同时支持 **`.com`** 和 **`.co`** 两个域名访问**，而不是做跳转。

**为什么 API 要并存而不是跳转？**

- API 调用走 302 跳转会导致部分请求失败（POST/PUT 请求跳转后可能变成 GET）
- 双域名并存零风险：旧页面调 `.co` → 能通；新页面调 `.com` → 也能通
- 消除了前后端部署时间差导致的断裂窗口
**后端团队需要做的：**

1. - 确保 DNS 记录同时指向后端服务（CloudFront / ALB）
2. - K8s Ingress / ALB 配置同时接受 `.com` 和 `.co` 的 Host
3. - CORS 配置同时允许 `.com` 和 `.co` 的 Origin
4. - 不需要部署 CloudFront Function，直接透传请求即可
**当前基础设施已支持双域名：**

- CloudFront：✅ 已配置 `.com` 作为备用域名
- SSL 证书：✅ `*.castlery.com` 通配符有效
- K8s Ingress：(我不确定😂)已配置双域名路由
**各阶段 API 域名策略：**

| 阶段 | 策略 |
| Phase 1 | `.com` 和 `.co` API 域名同时可用，前端代码继续使用 `.co` |
| Phase 2 | 前端代码统一切换为 `.com` API 域名，`.co` 继续保留可用作为兜底 |
| Phase 3 | 观察期后视情况决定是否下线 `.co` API 域名 |


> ⚠️ **需要后端团队确认**：所有 API 子域名的完整清单，以及各自对应的 CloudFront / ALB 配置，确保 DNS 指向后两个域名均可正常响应。

### 3.8 回滚方案

如果出现任何问题，直接在 CloudFront 移除 Function 关联即可，`.com` 恢复到之前的状态（无法访问），`.co` 完全不受影响。回滚耗时 < 2 分钟。

## 5. Phase 3：收尾巩固

### 5.1 Feed 数据刷新（后端团队）

当前所有产品 Feed 中的 URL 已经刷新为 `.co` 格式（如 `www.castlery.co/us/products/xxx`）。Phase 2 切换完成后，**必须重新触发 Feed 刷新**，将所有 URL 更新为 `.com`。

| Feed 类型 | 示例 | 负责团队 | 时间节点 |
| Google Shopping Feed | 产品 URL、图片 URL | Backend / Marketing | Phase 2 完成后立即 |
| Facebook / Meta Product Catalog | 产品 URL | Backend / Marketing | Phase 2 完成后立即 |
| Affiliate Feed | 产品 URL | Backend / BD | Phase 2 完成后立即 |
| 内部搜索索引 | 产品 URL | Backend | Phase 2 完成后立即 |
| 其他第三方 Feed | 需各团队确认 | 各团队 | Phase 2 后一周内 |


> ⚠️ 如果 Feed 不刷新，Google Shopping 广告、Facebook 广告等会指向 `.co` 的 URL。虽然 301 跳转能兜住，但会增加一次跳转延迟，且影响广告质量得分。

### 5.3 安全加固（防止再次被盗）

| 措施 | 说明 | 优先级 |
| 域名注册商锁定 (Registrar Lock) | 防止未授权转移 | 🔴 立即 |
| 注册商账号开启 2FA | 建议使用硬件密钥 (YubiKey) | 🔴 立即 |

## 8. 风险评估

| 风险 | 可能性 | 影响 | 应对措施 |
| DNS 传播延迟 | 中 | 低 | TTL 设为 300s (5min)，使用全球 DNS 检测工具监控 |
| SSL 证书问题 | 低 | 高 | 证书当前有效 ✅，已确认覆盖 `*.castlery.com` |
| 用户 Cookie 丢失 | 中 | 中 | Phase 1 始终跳转到 `.co`，不存在双域名并存 |
| 支付回调失败 | 低 | 高 | Phase 2 前先在 Stripe 添加 `.com` endpoint 并测试 |
| 前后端域名不同步 | 中 | 高 | Phase 2 要求前后端**统一部署窗口**同步切换 |
| SEO 排名短期下降 | 高 | 中 | 正常现象，通过 Change of Address + 301 + sitemap 加速恢复 |
| Feed 数据 URL 过期 | 高 | 中 | Phase 2 后立即触发 Feed 刷新，301 兜底过渡期 |
| CloudFront Function 异常 | 低 | 高 | Function 可在 2 分钟内移除，流量回退到 `.co` |
| 第三方服务遗漏 | 中 | 中 | Phase 2 前逐项检查清单，测试环境先验证 |
