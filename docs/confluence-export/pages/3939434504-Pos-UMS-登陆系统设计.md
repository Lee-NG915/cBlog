## 1. 背景

POS 需要接入 UMS 登录，并使用 UMS 的细粒度权限控制能力来管理用户身份与页面访问权限。

原 POS 登录流程是：

- 用户进入 `/{locale}/login`
- 在同一页面输入账号密码
- 同时选择 Branch
- 登录成功后持久化 `retailId`
- 后续请求通过本地持久化的 `retailId` 注入 `X-Retail-Store-ID`
新的目标方案是：

- 去掉 POS 自身账号密码输入
- 改为 UMS 统一登录
- 保留 Branch 选择，但放到 UMS 登录成功之后
- 用户若之前已经选过 Branch，则自动复用上次选择
- 保留 Region 选择入口，让设计师理解“先选市场，再登录，再初始化 Branch”的完整链路
---

## 2. 目标

### 2.1 业务目标

- 将 POS 登录方式切换为 UMS 登录
- 保持原有“记住上次 Branch”的使用体验
- 将登录流程拆分为“认证”和“门店上下文初始化”两个步骤
- 保持深链接能力，登录后尽量回到用户原始目标页面
### 2.2 设计目标

- 让用户清晰感知当前处于哪一步
- 在 UMS 登录成功但尚未完成 Branch 初始化时，给出明确页面引导
- 对错误状态、空状态、等待状态提供统一且清晰的视觉表达
- 让设计师理解哪些页面是“技术中转页”，哪些页面是“真正用户可操作页”
---

## 3. 当前关键技术事实

当前完整登录链路涉及 5 个关键页面：

- `/`
- `/{locale}`
- `/{locale}/login`
- `/{locale}/auth/callback`
- `/{locale}/auth/logout-callback`
当前本地关键状态：

- 登录态：`isLoggedIn`
- UMS 模式标记：`authMode`
- Branch：`retailId`
- 当前销售：`posSalesId`
当前规则：

- `/`

- 用户先选择 Region
- 若本地 `isLoggedIn=1`，进入 `/{region}/discover`
- 否则进入 `/{region}/login`
- `/{locale}`

- 若缺少登录态或缺少 `retailId`，进入 `/{locale}/login`
- 若二者都存在，进入 `/{locale}/discover`
- `auth/callback` 成功后

- 若本地已有 `retailId`，直接去 `callbackUrl` 或 `/{locale}/discover`
- 若没有 `retailId`，回到 `/{locale}/login`
- `/{locale}/login`

- 若未登录，自动跳转 UMS
- 若已登录且无 Branch，展示 Branch 选择
- 若已登录且已有 Branch，自动进入目标页

## 4. 信息架构

### 4.1 页面清单

- `/`
- `/{locale}`
- `/{locale}/login`
- `/{locale}/auth/callback`
- `/{locale}/auth/logout-callback`
### 4.2 页面职责

| 页面 | 职责 |
| `/` | Region 选择入口，决定用户进入哪个市场 |
| `/{locale}` | 市场根路由，判断是否已有登录态与 Branch |
| `/{locale}/login` | 登录入口、登录状态检查、Branch 选择、进入目标页 |
| `/{locale}/auth/callback` | 处理 UMS 回调，决定回 login 还是直接进入业务页 |
| `/{locale}/auth/logout-callback` | 清理本地 OIDC 用户与 POS 上下文，并回登录入口 |

## 7. 页面 PRD

### 7.1 `/`

![](attachment)



#### 页面目标

- 作为 POS 的市场选择入口
- 帮助用户明确先进入哪个 Region
#### 页面状态

- 默认态：展示 Region 下拉与 Next 按钮
- 提交后：

- 已有登录态，进入 `/{locale}/discover`
- 无登录态，进入 `/{locale}/login`
#### 交互要求

- Region 为必选
- 页面无需承担登录动作
- 这是“市场选择页”，不是“认证页”
### 7.2 `/{locale}/login`

![](attachment)

#### 页面目标

- 作为新的正式登录入口页承接整个认证入口
- 承担两段式流程：

- UMS 登录
- Branch 初始化
#### 页面状态

状态 A：Checking

- 页面加载后立即检查 OIDC 用户状态
- UI 可以极简
状态 B：未登录

- 不展示单独登录按钮
- 系统自动跳转到 UMS 登录页
- 如果跳转失败，使用 modal 提示错误
状态 C：已登录但未选择 Branch

- 展示 Branch 选择表单
- 字段：

- Branch 下拉框
- 操作：

- Continue
- Sign out from UMS
状态 D：已登录且已有 Branch

- 页面不需要停留
- 自动跳转至 `callbackUrl` 或 `discover`
#### 交互要求

- Branch 为必填
- 未选择时点击 Continue，应有明确字段错误提示
- Continue 后应展示 loading 状态，避免重复点击
- 登录流程错误、初始化错误、跳转错误使用统一 modal 提示
### 7.3 `/{locale}/auth/callback`

**产品备注可使用空白页**

#### 页面目标

- 承接 UMS 回调
- 处理 code 兑换、用户恢复、桥接登录态、决定跳转目标
- 这是技术中转页，不是主要交互页

## 8. User Stories

### 8.1 核心登录场景

**US-01 首次进入 POS，先选择 Region**

作为 POS 用户，  
我希望先选择要进入的 Region，  
以便系统将我导向正确的市场环境。

验收标准：

- 进入 `/` 时展示 Region 选择页
- 选择 Region 后进入对应 `/{locale}` 或 `/{locale}/login`
**US-02 首次进入 POS，尚未登录**

作为 POS 用户，  
我希望进入系统时自动被引导到 UMS 登录，  
以便我使用公司统一账号登录。

验收标准：

- 进入 `/{locale}/login` 时，若未登录，自动跳转到 UMS
- 不需要用户额外点击登录按钮
**US-03 登录成功但没有 Branch**

作为已完成 UMS 登录的用户，  
我希望系统提示我选择 Branch，  
以便系统知道我的业务上下文。

验收标准：

- 登录回到系统后，如果本地无 `retailId`，展示 Branch 选择页面
- Continue 前必须选中 Branch
**US-04 登录成功且已有上次 Branch**

作为经常使用 POS 的用户，  
我希望系统自动复用我上次选择的 Branch，  
以便减少重复操作。

验收标准：

- 若本地已有 `retailId`，登录成功后自动进入目标页面
**US-05 带深链接进入受保护页面**

作为从其它入口访问 POS 业务页的用户，  
我希望登录成功后回到原始目标页，  
而不是总是回首页。

验收标准：

- 存在 `callbackUrl` 时，完成登录与 Branch 初始化后返回该页面
### 8.2 Branch 场景

**US-06 用户需要重新选择 Branch**

作为已登录用户，  
我希望在没有有效 Branch 时重新选择门店，  
以便继续使用 POS。

验收标准：

- 无 `retailId` 时不允许直接进入业务页
- 必须完成 Branch 选择后才能进入 POS
**US-07 用户未选择 Branch 点击 Continue**

作为用户，  
我希望在操作不完整时得到明确提示，  
以便我知道下一步该怎么做。

验收标准：

- 未选择 Branch 点击 Continue 时显示字段错误提示
### 8.3 异常场景

**US-08 UMS 回调失败**

作为用户，  
我希望在登录失败时看到可理解的错误说明，  
以便我知道应该重试或联系支持。

验收标准：

- callback 页面显示失败态
- 建议有返回登录页入口
**US-09 Branch 列表加载失败**

作为用户，  
我希望在 Branch 无法加载时看到可理解提示，  
以便我知道系统暂时不可用。

验收标准：

- Branch 列表失败时要有错误态设计
- 建议支持重试
**US-10 本地状态过期**

作为用户，  
我希望当本地登录态或 Branch 状态失效时，  
系统能把我引导回正确的初始化流程。

验收标准：

- 无登录态：回登录流程
- 有登录态但无 Branch：回 Branch 选择
### 8.4 登出场景

**US-11 用户退出登录**

作为用户，  
我希望退出后系统返回登录入口，  
以便我确认当前账号已退出。

验收标准：

- 登出后本地 OIDC 用户被清理
- POS 本地业务上下文也被清理
- 返回登录入口，并自动进入下一次登录流程

## 9. 关键说明

- `/` 不是登录页，而是市场选择页
- `/{locale}/login` 不是单纯“登录页”，而是“登录入口 + Branch 初始化页”
- callback 页面更多是技术中转页，设计可以相对轻量
- “是否已有 Branch”是关键分叉：

- 已有：自动跳转，用户可能根本看不到 Branch 表单
- 没有：必须停留在 `/{locale}/login` 完成 Branch 选择
- “是否已有 locale”也是关键分叉：

- 没有 locale：先进入 Region 选择
- 有 locale：再进入该市场下的登录/初始化流程
---

## 10. 待设计确认问题

- `/` 的 Region 选择页是否需要品牌化视觉，还是保持极简入口页
- `/{locale}/login` 在“自动跳转 UMS”阶段是否需要短暂过渡文案，还是直接空白 loading
- Branch 选择页是否需要带说明文案，例如 `Choose a branch to continue`
- `auth/callback` 是否需要单独视觉风格，还是统一 loading 模板即可  **产品留言使用空白页**
- `logout-callback` 是否需要成功反馈，还是仅展示极简“正在退出”状态 **产品留言使用空白页**
