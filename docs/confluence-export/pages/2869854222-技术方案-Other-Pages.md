## The key results after align with PM (first time)

### When migrating other pages, keep the URLs unchanged

#### Tech side

##### Pros:

It is beneficial for the current SEO data. Given the recent tickets related to reducing the number of redirects, at this stage, it may be preferable to keep the URLs unchanged to avoid increasing the number of 301 and 302 redirects.
有利于当前的 SEO 数据，鉴于最近出现的要减少 redirect 次数的相关 ticket，可能在当前阶段，更倾向于 keep url，让 301,302 数量不再增加。

##### Cons:

1. - The current framework being used is Next.js, and the solution to keep the URLs unchanged will result in not being able to use Next's folder structure for route management.
现在使用的 framework 是 Next.js，keep url 的 solution 会导致无法使用 Next 的 folder 管理 routes 的方式。
2. - Since most page types do not have fixed URLs, the majority of the pages will be handled by the `[...rest]` folder. However, **this is not the optimal solution**.
由于大部分 page 类型都没有固定的 url，所以大部分 page 会流入 [...rest] folder 中处理，但是这并不是最好的解决方案
##### Summary

Using folders to manage routes, after evaluation, it is concluded that this approach would make our routing structure clearer and the URLs more semantic in the future. However, since Google and other SEO engines have already crawled and defined our pages, changing the URLs would result in a large number of redirect status codes. It depends on whether we can accept the transitional period of having many redirects.
使用 folder 管理路由，评估之后得出，对于未来来说，会使我们的路由结构更加清晰，使 url 更加语义化。但是由于 google 等 SEO 引擎，已经对我们的页面爬取以及定义，如果更换 url，会导致出现大量的重定向状态码。取决于能否接受大量重定向的阵痛期。
However, most hard-coded pages don't actually need  SEO, so it's worth considering an evaluation of the impact caused by modifying the URLs.
然而，大多数硬编码页面实际上不需要 SEO，因此值得考虑评估修改 URL 所带来的影响。

## Routing Solution

For the "other pages," they can be divided into two parts: one part consists of pages that can be managed by folders, and the other part needs to be placed in the `[...rest]` folder. These pages will be routed to different page layouts for rendering based on path matching.
对于 other pages，可以将他们分为两部分，一部分是可以被 folder 管理的 page，一部分需要放在 […rest] 中，通过路径匹配，分发到不同的页面 layout 中去渲染。

#### Pages Managed by Folders

For hard-coded pages, we can directly create folders with corresponding names to place the respective `page.tsx`.
对于 hard code page，我们可以直接创建对应名称的 folder 来放置对应的 page.tsx。
For Storyblok blog pages, since they have a fixed prefix, we can also use folder routing to manage them.
对于 storyblok blog page，由于有固定的前缀，我们也可以使用 folder route 来管理。
Like：

```
└── [region]
    └── [locale]
        ├── reviews          
        │   └── page.tsx
        ├── contact-us           
        │   └── page.tsx
        ├── blog 
        │   ├── [slug]
        │       └── page.tsx
        │   └── page.tsx
        ......
```

#### Pages Managed by `[...rest]`

For the remaining pages, we consolidate them under `[...rest]/page.tsx` for routing. Since the URLs have non-fixed prefixes, they will eventually be caught by `[...rest]`.
翻译成英文：对于剩余的 page，我们收拢在 […rest] 下的 page.tsx 去分流，因为是具有非固定前缀的 url，最后都会被 […rest] catch 住。

> It is important to note that if new URLs are added in the future, such as those configured in Storyblok or other third-party systems, attention must be paid to whether they will be caught by the Next.js folder structure instead of being handled by `[...rest]`.
需要注意的是，以后新增了 url，比如在 storyblok 或者其他第三方配置的，需要注意是否会被 next folder 的结构在其他非 […rest] catch 住。

The routing rules for `page.tsx` are primarily based on the characteristics of each page. The code will first construct a page map before routing decisions are made.
page.tsx 的分流规则，主要是根据 page 的特性去判断，代码会在进入分流前先构建 page map。

##### page map

For the construction of the page map, the approach remains consistent with Onepiece. Since non-hard-coded pages all come from Storyblok, the page list will be fetched through the API provided by Storyblok. However, there are two TODOs that need to be addressed:
对于 page map 的构建，思路仍是和 onepiece 保持一致，因为非 hard code 的页面，page 均来自于 storyblok，所以会通过 storyblok 提供的接口拉取 page list。但是这里有两点 todo 待完善：

1. - It is uncertain whether there is a place to perform pre-filtering on the final page requests. Since our multiple markets are currently maintained under the same space, Storyblok’s API for fetching batch page data does not support specifying a particular folder. As a result, it retrieves pages for all markets, and we then have to differentiate them by folder prefixes. This leads to two issues:
不确定最后页面的请求有没有地方做前置过滤。因为我们的多 market 现在 都维护在同一个 space 下，但是 storyblok 调取批量页面的数据并不支持指定某个 folder。所以会把所有市场的 page 拉下来，再通过 folder 的前缀去区分判断。这就会导致两个问题：
- Since different machines manage different markets, each time the page list is retrieved, only a portion of it is valid.
因为是由不同机器管理不同 market，所以每次调取 page list 时，只有一部分是有效的
- Due to Storyblok's limitation of a maximum of six requests per second per token, there might be cases where requests are denied.
因为 storyblok 限制了单 token 一秒最多调取六次，所以可能会存在请求被拒的情况
1. - Caching issue: It has not yet been determined whether the caching will be handled internally by allocating space within Node, or if it will be managed by the CDN.
缓存问题。这一块尚未确定是由 node 内部自己开辟空间存储，还是由 CDN 代为存储。
The method for retrieving the page list is straightforward. Since the logic for differentiating markets has been encapsulated, it's just a simple call without needing to write any additional filters.
调取 page list 的方法很简单，由于将区分 market 的逻辑封装了，所以只是简单的调用，不需要再写其他 filter。

```
import { EcEnv } from "@castlery/config";
import { StoryblokPageItemType, StoryblokService } from "@castlery/modules-cms-services";

const cmsPageFilter = async () => {
  const storyblok = new StoryblokService({
    accessToken: EcEnv.NEXT_PUBLIC_STORYBLOK_TOKEN || ''
  });
  const pageList: StoryblokPageItemType[] = [];
  try {
    const pageListResult = await storyblok.fetchPageInStoryblok({
      contentType: 'page',
	});
	const { data } = pageListResult[0];
	pageList.push(...data);
  } catch (e) {
	console.error('🚀 ~ cmsPageFilter error ~', e);
  }
  return {
	pageList
  }
};

export default cmsPageFilter;
```

Then, in `page.tsx`, routing is handled as follows:
然后在 page.tsx 中，再通过分流：

```
const judgePageType = (str: string) => {

  if (pageListResult.status === 'fulfilled') {

      }
    });
  }

  if (blogPageListResult.status === 'fulfilled'){

        fullSlug = page.key;
      }
    });
  }

  return {
    pageKind,
    fullSlug,
  };
};
```

## Page Map Caching Solution

### Context

In Onepiece, the process involves calling the API to retrieve the page list within the pod, then filtering out the content from the page list, keeping only key information such as key, URL, and path. This is stored in a global object for constructing the sitemap or for other routing-related decisions. This content is then injected into the window object and included in the HTML via a script.
onepiece 中，是通过在 pod 中调用 获取 page list 的接口，然后将 page list 内的 content 过滤掉，只保留 key，url，path 等关键信息，放入 global 对象中，用于构建 sitemap，或者其他路由相关判断。然后会将这部分内容放入 window 中，跟随 script 带入到 html 上。

### Joyboy Solution

#### Using File-based Caching

Due to the Next.js structure, there is no need to place the page map in the window, which prevents adding extra weight to the page size. The idea is to avoid filtering out the content when retrieving the page list and instead merge the content and store it in memory. This way, when entering a page, there is no need to make a separate API call to fetch the page information, as rendering can be done directly from the cache.
由于 next 结构，不需要将 page map 再放到 window 中，不会占用页面体积。所以思考在获取 page list 时，不过滤掉 content，将 content 合并存入内存中。这样，在进入页面时，不需要再单独调用获取页面信息的接口，页面的渲染直接从缓存里面读取。

However, this approach presents the challenge of handling a large cache size, roughly estimated to be around 8 MB. If this cache is stored only in the additional space allocated by Node, it could impact the running memory of the pod. Therefore, the plan is to use file operations, dynamically generating files to maintain the page content provided by the CMS.
但是这样带来的问题是，缓存的内容会非常大，粗略估计达到 8 M 左右。如果只放在 node 开辟的额外空间去缓存，实际上会影响到 pod 的运行内存。所以打算采取文件操作，动态生成文件，维护 CMS 传递过来的页面内容。

##### Can use webhook？

Storyblok provides a webhook that notifies us when a story is modified. After receiving the notification in our service, we can perform the corresponding updates. This method appears to be better than periodically refreshing the cache. However, since our pods are independent and memory is not shared, when Storyblok calls the API, it will only hit a specific pod, triggering the update on that pod. Other pods cannot rely on the webhook for updates.
Storyblok 提供了 webhook，供我们在对 story 进行操作时发出通知，我们在服务中接收到通知后，再进行相应的更新，这个方法看到现在是比定时刷新缓存更好的。但是由于我们的 pod 之间是独立的，即内存是不共享的，当 Storyblok 调用接口后，实际上只会打到某个 pod 上，触发该 pod 的更新动作。其他 pod 没有办法依赖 webhook 进行更新。

Therefore, a scheduled update approach is adopted, where each pod independently updates the cache. Since Node reads files very quickly, this approach generally results in a performance gain. In terms of implementation, even when using segmented writing, a new file is created, and once it’s fully written, it replaces the outdated file.
所以采用了定时更新的方案。即每个 pod 独立更新缓存。而由于 node 在读文件的速度上是非常快的，所以这一块基本上对性能是正收益的。而且在实现方式上，即使是通过分段式写入，也是新建一个文件，待新文件写完成后，再覆盖过期文件。

##### Cons

Writing files in Node is relatively slower compared to reading them, so after a deployment, there may be a file-writing process. During this time, if no other action is taken, users might be directed to a "not found" page. However, in the middleware of `[...rest]`, it can be set up to check if the corresponding page map file exists. If not, it will make an individual API request, but this request will not involve writing the file.
node 的写文件相对于读文件来说比较慢，可能在发版后，会有一个写文件的过程，这段时间用户访问，如果不做其他操作，会进入到 not found 页面。 但是可以在 […rest] 的 middleware 中去判断，如果没有相应的 page map file，就会单独去请求接口，但是这里的请求就不涉及到写文件了。
