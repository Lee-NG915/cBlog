## 背景

在onepiece项目中，我们目前使用Storyblok的方式存在多个效率和管理上的问题。以下详细说明了这些问题，并提出了可能的改进方案。

1. - **组件复用性不足**：尽管理论上我们创建的每个component json应当具备高复用性，实际操作中却往往缺乏这种效率。例如，即使多个json组件都需使用到同一个button component，它们在实现时却需要分别定义如buttonText和buttonBgColor等属性，而这些属性在多个组件中重复定义。为改善这一点，我们需要定义更为通用的组件属性，通过全局样式或者主题管理来统一样式变量，从而减少冗余并提升组件的可复用性。
2. - **content json缺乏扩展性**：当前的content json设计不足以应对业务迭代带来的需求变化。例如，某个列表项被设计为适用多个场景，但随着新场景的加入，需要向现有列表中添加新字段，这导致原有场景中的列表项出现不必要的冗余字段。解决这一问题的方法是采用更灵活的数据结构，如引入可配置的字段集合或使用继承机制，以便根据不同的使用场景动态调整内容结构。
3. - **不支持灵活的A/B测试**：由于Storyblok的引入时间早于A/B测试的广泛应用，后期集成A/B测试时常常需要进行较大的代码改动，且这种改动多为页面级别的，不支持组件级别的微调。为此，可以考虑在Storyblok中实现更为灵活的内容变体管理功能，允许在组件级别进行A/B测试，减少因测试引入的冗余页面和资源浪费。
4. - **多市场支持不足**：原始设计中对不同市场的环境资源进行了隔离，这使得任何新增或修改配置都需要较大的人工介入，易出错且效率低下。改进措施包括简化市场间的内容共享和配置过程，利用自动化工具来支持内容的迁移和同步，确保测试和用户验收测试（UAT）能及时涵盖新的变更，降低错误率。
总之，通过优化组件设计、增强content json的灵活性、引入组件级的A/B测试支持以及提升对多市场的快速响应能力，可以显著提高Storyblok在onepiece项目中的应用效率和管理水平。



## 接入设计

### 回顾痛点

在原有的 onepiece 项目中，一个显著的问题是 Storyblok CMS 与应用的核心业务逻辑高度耦合。这种耦合导致从应用架构中完整剥离 Storyblok 变得极为困难。在代码库中，许多关键的功能和模块直接依赖于 Storyblok 的 API 和数据结构，这使得每一个与内容管理相关的功能都与 Storyblok 紧密相连。因此，一旦需要将 Storyblok 替换为另一个 CMS，就需要进行大量的代码重构。这不仅包括更新数据调用和管理方式，还可能涉及调整前端组件以适应新的内容结构，以及重写与内容交互的后端逻辑。此外，替换 CMS 可能还需要更新相关的测试用例和文档，确保新系统的稳定性和维护性，从而增加了项目的复杂性和转换成本。



## 架构设计原则

### 核心设计理念

CMS 接入遵循以下核心设计原则：

#### 关注点分离

核心理念：将系统划分为职责明确、相互独立的层次，每个层次只关注自己的核心职责。

##### 具体实现：

##### 数据获取层 （Service Layer）

职责：API 调用、数据获取、缓存管理、错误处理
不关心： UI 如何渲染、业务规则如何应用
示例：**SbService** 类只负责从 Storyblok 获取数据，不涉及数据如何展示

##### 业务逻辑层 （Domain Layer）

职责：业务实体定义、类型系统、状态管理
不关心：数据从哪里来、UI 如何展示
示例：**cmsSlice **管理 CMS 数据的状态，定义数据结构和业务规则

##### 展示层 （Components Layer）

职责：UI 组件实现、渲染逻辑、用户交互
不关心：数据如何获取、业务规则如何定义
示例：libs/modules/cms/components/src/lib/index.ts 导出的组件只接收数据并渲染，不关心数据来源

##### 设计优势

- 可维护性：修改某一层不影响其它层
- 可测试性：每层可以独立测试
- 可扩展性：新增功能只需修改对应层
- 团队协作：不同团队可以并行开发不同层

#### 服务端优先

核心理念：优先在服务端完成数据获取和处理，客户端仅负责交互和展示。

##### 具体实现：

##### Server Actions 数据获取

export default fetchCMSData;

```

##### Server Components 渲染

```
// 默认是 Server Component，数据在服务端获取
export default async function Page() {
  const data = await fetchCMSData.storyblok({ slugArray: ['home'] });
  return <CMSContent data={data} />;
}
```

##### 敏感信息保护

- API token 存储在环境变量中
- 客户端无法直接调用 Storyblok API
- 所有数据请求通过 Server Actions 代理
##### 设计优势

- 安全性：敏感信息（API Token、密钥）不暴露到客户端
- 性能：服务端计算能力强，可并行处理多个请求
- SEO 友好：内容在服务端渲染，搜索引擎可直接抓取
- 首屏性能：减少客户端 JS 体积，加快首屏加载
- 数据一致性：服务端统一数据源，避免客户端数据不一致
##### 对比传统客户端方案

| **方面** | **服务端优先** | **客户端优先** |
| API Token 安全 | ✅ 不暴露 | ❎ 可能暴露 |
| 首屏性能 | ✅ 更快 | ❎ 需要等待 JS 加载 |
| SEO | ✅ 完整内容 | ❎ 需要额外处理 |
| 数据一致性 | ✅ 统一 | ⚠️ 可能不一致 |
| 客户端负担 | ✅ 轻量 | ❎ 较重 |


#### 渐进增强

核心理念：基础功能在服务端完成，确保核心功能可用；交互功能在客户端增强，提升用户体验

##### Fallback 数据机制

```
// 编译时打包的 Fallback 数据
const FALLBACK_GLOBAL_NAV_DATA = {
  sg: {
    value: [...],  // 实际数据
    lastUpdated: '2024-01-01'
  }
};

// 运行时降级逻辑
private _loadFallbackGlobalNav(): any[] | null {
  const market = this.region.replace('/', '').toLowerCase();
  const fallbackData = FALLBACK_GLOBAL_NAV_DATA[market];
  if (fallbackData) {
    logger.warn('Using fallback data', { market });
    return fallbackData.value;
  }
  return null;
}
```

##### 错误处理策略

```
try {
  const data = await fetchFromStoryblok({ slugArray });
  return data;
} catch (error) {
  // 1. 记录错误
  logger.error('API request failed', { error });
  // 2. 上报监控
  Sentry.captureException(error);
  // 3. 尝试降级
  const fallback = loadFallbackData();
  if (fallback) {
    return fallback;
  }
  // 4. 返回空数据，不中断服务
  return null;
}
```

##### 降级场景处理

| **场景** | **检测方式** | **降级策略** | **用户体验影响** |
| API 返回 503 | HTTP 状态码 | 使用 Fallback 数据 | 内容可能不是最新的，但可以正常访问 |
| API 超时 | 请求超时 | 使用 Fallback 数据 | 延迟稍高，但可正常访问 |
| 数据格式错误 | 数据验证 | 使用 Fallback 数据 | 内容可能不是最新 |
| 网络错误 | 异常捕获 | 使用 Fallback 数据 | 可正常访问 |
| Fallback 不存在 | 数据检查 | 返回空数据 | 功能降级，页面仍可渲染 |


##### 设计优势

- 高可用性：即使外部服务故障，核心功能仍可用
- 用户体验：避免白屏或错误页面
- 业务连续性：关键业务不受外部依赖影响
- 可观测性：完整的错误日志和监控

### 多层缓存策略

### 缓存架构说明

> 缓存不是严格意义的串行层次，而是根据使用场景选择不同的缓存策略。不同的 api 方法使用不同的缓存组合。

#### 实际使用的缓存机制

##### CDN/浏览器缓存（HTTP 层）

- 通过 Cache-Control 响应头控制
- 在 Next.js 路由层面配置
- 所有请求都会经过此层
##### Next.js Cache (unstable_cache) (主要缓存)

- **所有 Storyblok api** 调用都使用此缓存
- 跨请求持久化缓存，支持 Redis
- 标签化失效，精确控制缓存
- 使用位置：

- fetchStoyblokApi： 获取单个 Story
- fetchFromStoryblokPage：批量获取 Story
- getStoryWithCache：带缓存的获取方法
- getStoriesWithCache：带缓存的列表获取
##### React Cache (reactCache)(请求去重)

- ⚠️ 仅用于特定方法
- 请求级缓存，同一请求生命周期内去重
- 主要用于配合 Next.js Cache 使用
- 使用位置：

- getStoryWithReactCache：仅 React Cache
- getStoryWithBothCache：React Cache + Next.js Cache
- getStoriesWithBothCache：React Cache + Next.js Cache
##### Memory Cache (memory-cache)(暂时未使用)

- ⚠️ 有可能造成内存泄露，或者内存无法释放
- 预留接口，未来可能需要时启用

// libs/modules/cms/services/src/lib/sbService.ts
private _loadFallbackGlobalNav(): any[] | null {
  try {
    const market = this.region.replace('/', '').toLowerCase() || 'sg';
    const fallbackData = FALLBACK_GLOBAL_NAV_DATA[market];

    if (!fallbackData || !fallbackData.value) {
      logger.error('Fallback global nav data not found', { market });
      return null;
    }

    logger.warn('Using fallback global nav data', {
      market,
      count: fallbackData.value.length,
      lastUpdated: fallbackData.lastUpdated,
    });

    return fallbackData.value;
  } catch (error) {
    logger.error('Failed to load fallback global nav data', { error });
    return null;
  }
}
```

##### fallback 数据特点

- 编译时打包进代码，无运行时文件读取
- 支持 Edge Runtime

- 包含最后更新时间，便于监控

#### 降级场景

| **场景** | **处理方式** | **影响** |
| api 返回 503 | 使用 fallback 数据 | 用户可正常访问，内容不是最新 |
| api 超时 | 使用 fallback 数据 | 用户可正常访问 |
| 数据格式错误 | 使用 fallback 数据 | 用户可正常访问 |
| fallback 数据不存在 | 返回空数据/默认值 | 功能降级，但不影响页面渲染 |

```
// apps/web/app/[deviceTheme]/[region]/[locale]/[...rest]/utils/content-processor.ts

// 工具函数：处理整个页面内容
export const processPageContent = (
  content: any,
  cookieContext: { cookies: any },
  dyCampaignsData: Record<string, any>
): ProcessItemsResult => {
  const dyCampaignPromises: Promise<void>[] = [];
  let hasFaqSchema = false;
  let jsonLd: WithContext<FAQPage> = {
    '@type': 'FAQPage',
    '@context': 'https://schema.org',
  };

  // 1. 处理 story.content.body
  const bodyResult = processItems(content?.body || [], cookieContext, dyCampaignsData);

  // 2. 合并结果
  dyCampaignPromises.push(...bodyResult.dyCampaignPromises);
  hasFaqSchema = bodyResult.hasFaqSchema;
  jsonLd = bodyResult.jsonLd;

  // 3. 处理包含 items 的组件（如 Hero 组件）
  for (const item of content?.body || []) {
    const componentResult = processComponentWithItems(item, cookieContext, dyCampaignsData);
    if (componentResult) {
      dyCampaignPromises.push(...componentResult.dyCampaignPromises);
      if (componentResult.hasFaqSchema) {
        hasFaqSchema = componentResult.hasFaqSchema;
        jsonLd = componentResult.jsonLd;
      }
    }
  }

  return {
    dyCampaignPromises,
    hasFaqSchema,
    jsonLd,
  };
};
```

## 总结

### 核心设计思想

CMS 接入设计遵循以下核心思想：

1. - 分层架构：清晰的职责划分，便于维护和扩展
2. - 服务端优先：数据获取在服务端，保证安全性和性能
3. - 多层缓存：减少 api 调用，提高响应速度
4. - 容错降级：确保服务可用性，提升用户体验
5. - 类型安全：完整的类型系统，减少运行时错误
6. - 性能优化：批量请求、数据预处理、代码分割
7. - 开发体验：实时预览、类型提示、调试支持
### 设计优势

1. - 高可用性：多层降级保护，确保服务不中断
2. - 高性能：多层缓存+批量请求+服务端渲染
3. - 高可维护性：清晰的分层架构+类型安全
4. - 高可扩展性：组件化设计+插件化架构
5. - 良好的开发体验：实时预览+类型提示+调试支持
