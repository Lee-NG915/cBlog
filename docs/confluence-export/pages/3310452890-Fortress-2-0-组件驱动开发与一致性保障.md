# 1. 引言

## 1.1 Fortress 2.0 组件库概述

Fortress 2.0 是我们品牌重构之后的全新版本，基于 MUI 的 Joy UI 构建，支持多主题配置，旨在提供一套统一、高效且易于使用的 UI 组件系统。

我们构建 Fortress 2.0 的核心目标，是为业务提供一套一致、易用、可维护的 UI 组件系统，同时也作为设计系统落地的一部分，把视觉规范以代码的形式沉淀下来。

**核心特性：**

- **设计 token 驱动**：继承 Joy UI 的 design token 系统，方便统一风格并支持多主题切换；
- **组件覆盖全面**：从基础组件（如 Button、Input）到复杂业务组件（如 Form、Stickey Header）均有提供；
- **强类型支持**：全量 TypeScript 类型定义，确保 IDE 智能提示与类型安全；
- **可访问性考虑**：遵循 WCAG 2.1，基础组件默认支持键盘和屏幕阅读器；
- **开发者友好文档**：结合 Storybook 提供可交互、可预览、可复制的组件使用示例。
Fortress 2.0 不只是一个 UI 库，更是我们推进组件驱动开发（CDD）理念在团队中落地的核心基础设施。

## 1.2 组件驱动开发 (CDD)

组件驱动开发 (Component-Driven Development, CDD) 是一种自下而上的 UI 开发方法论，它将复杂的用户界面分解为独立、可重用的组件，并在隔离环境中进行开发、测试和文档化。

在 Fortress 2.0 中，我们从一开始就全面采用 CDD 的方式开发组件。与传统页面驱动开发（Page-first Development）不同，CDD 更关注组件本身的独立性、组合性和一致性，这也更贴合现代前端工程体系的演进方向。

**CDD 的核心理念**

- **组件优先构建**：从最小的 UI 单元（如按钮、输入框）出发，逐步组合成业务模块；
- **隔离环境开发**：通过 Storybook 等工具独立开发和测试组件，减少外部依赖干扰；
- **状态可控**：组件对外暴露明确的 props，以状态驱动外观和行为；
- **视觉验证先行**：依赖视觉测试工具（如 Chromatic）进行 UI 回归检测，确保稳定性；
- **文档即规范**：组件文档同步更新，既是开发参考，也是设计对齐的桥梁。
**CDD 为我们带来的价值**

1. - **提高开发效率**

- 可以多人并行开发不同组件，无需等待页面结构；
- 在隔离环境中即时预览反馈，加快调试和迭代；
- 一处开发，多处复用，大量减少重复 UI 实现。
2. - **增强产品质量**

- 所有组件行为受控，外观风格统一；
- Storybook + 单测 + 视觉测试，确保每个组件都可测试；
- 模块化结构，让维护旧代码或扩展新功能变得更轻松。
3. - **提升团队协作体验**

- 设计与开发共享组件语言，不再“你说红色我理解粉色”；
- 不同产品团队共用同一套组件，降低沟通和对齐成本；
- Storybook 就是活文档，新人上手快、团队切换成本低。
4. - **加速业务创新**

- 新功能原型可以基于已有组件快速拼装；
- 设计师可以在不改产品代码的前提下试验新样式；
- 由于组件具备良好的封装性，底层技术迁移（如从 MUI 切到 Tailwind）也会更平滑。
Fortress 2.0 中，我们通过三个关键工具实现 CDD：

- **Design Token**：确保设计元素的一致性和可维护性
- **Storybook**：提供组件隔离开发环境和活文档
- **Chromatic**：自动化视觉测试，防止意外的 UI 回归
这套开发模式已经深入我们的日常流程，无论是开发者还是设计师都能从中受益。而 Fortress 2.0 本身就是这一方法论最佳实践的成果，解决我们团队中“组件复用难、风格不统一、缺少规范”等老问题。

# 2. Design System

## 2.1 Joy UI Design Token 体系

Design token 是设计系统的基础构建块，它们将设计决策编码为可重用的变量。在 Fortress 2.0 中，我们基于 Joy UI 的 token 体系构建了自己的设计语言，确保了视觉一致性和开发效率。

```
graph TD
    A[Design Token] --> B[颜色系统]
    A --> C[排版]
    A --> D[间距与布局]
    A --> E[阴影与层级]
    B --> F[品牌色板]
    B --> G[语义色]
    B --> H[组件特定色]
    C --> I[字体]
    C --> J[字号]
    C --> K[字重]
    D --> L[内边距]
    D --> M[外边距]
    D --> N[间隙]
    E --> O[阴影级别]
    E --> P[层级关系]
```

### 2.1.1 颜色系统

Fortress 2.0 的颜色系统采用语义化 token，而非直接使用硬编码色值。这种方式不仅提高了代码可维护性，也使主题切换变得简单。

#### 品牌色板

我们的品牌色板定义了一系列具有品牌特色的颜色，每种颜色都有从 10 到 900 的完整色阶：

| 色系 | 用途 | 示例色值 (500) |
| terracotta | 主色调 | #844025 |
| maroonVelvet | 中性色/文本 | #3C101E |
| leafGreen | 成功状态 | #414128 |
| burntOrange | 警告/强调 | #D25C1B |
| freshWaterBlue | 信息/链接 | #9CA3C1 |
| warmLinen | 背景/表面 | #F6F3E7 |
| rosewood | 危险/错误 | #65000B |

### 2.1.2 排版

Fortress 2.0 的排版系统基于清晰的层级结构，确保了文本在不同场景下的一致性和可读性。

#### 字体

定义了两种主要字体族，分别用于正文和标题：

```
export const fontFamilyV2: CssVarsThemeOptions['fontFamily'] = {
  // 正文字体
  body: `var(--font-sanoma-sans,"SanomatSans"),var(--fortress-fontFamily-fallback)`,
  // 标题字体
  display: `var(--font-aime, "Aime"),var(--fortress-fontFamily-fallback)`,
  // 备用字体
  fallback: ['Helvetica Neue', 'Arial', 'sans-serif'].join(','),
};
```

#### 字重

定义了四个字重级别，从轻量到粗体：

```
export const fontWeightV2: CssVarsThemeOptions['fontWeight'] = {
  sm: 300,  // 轻量
  md: 400,  // 常规
  lg: 600,  // 中粗
  xl: 700,  // 粗体
};
```

```
export const fontSizeV2: CssVarsThemeOptions['fontSize'] = {
```

```
  sm: 1.25, // 紧凑行高
```

### 2.1.3 间距与布局

Fortress 2.0 的间距系统基于 4px 进行设计，也就是说4px作为组件中像素变化的最小单位，通过 `spacing` 函数实现：

```
// 间距函数定义
spacing: (num: number) => num * 4,
```

- `spacing(1)` 等于 4px
- `spacing(2)` 等于 8px
- `spacing(3)` 等于 12px
- 以此类推
在组件中使用间距：

```
// 使用间距 token
<Box sx={{ p: 2 }}>  {/* p: 2 对应 8px 的内边距 */}
  <Stack spacing={3}>  {/* spacing: 3 对应 12px 的间隙 */}
    <Item />
    <Item />
  </Stack>
</Box>
```

### 2.1.4 阴影与层级 (Elevation)

阴影系统定义了元素的层级关系和空间感，增强用户对界面层级的理解。

#### 阴影级别

我们定义了基础阴影和两种特殊用途的阴影系列：

```
shadow: {
  /* offset-x | offset-y | blur-radius | spread-radius | color */
  sm: '0px 1px 2px 0px rgba(0, 0, 0, 0.15)',
  md: '0px 2px 6px 0px rgba(0, 0, 0, 0.25)',
  lg: '0px 4px 10px -1px rgba(0, 0, 0, 0.30)',
  // 抬升效果阴影
  raised: {
    sm: '0px 2px 4px 0px rgba(0, 0, 0, 0.15)',
    md: '0px 4px 8px 1px rgba(0, 0, 0, 0.15)',
    lg: '0px 6px 12px 2px rgba(0, 0, 0, 0.25)',
  },
  // 悬浮效果阴影
  floating: {
    sm: '0px 6px 20px -1px rgba(34, 34, 34, 0.20)',
    md: '0px 8px 30px -2px rgba(34, 34, 34, 0.30)',
    lg: '0px 12px 40px -3px rgba(34, 34, 34, 0.40)',
  },
},
```

使用示例：

```

<Card sx={{ boxShadow: 'md' }}>
  基础卡片
</Card>

<Card sx={{ boxShadow: 'raised.md' }}>
```

## 2.2 组件设计原则

### 2.2.1 一致性原则

一致性是 Fortress 2.0 最核心的设计原则，我们从三个维度确保一致性：

#### 视觉一致性

所有组件使用相同的 design token 体系，确保视觉语言的统一：

```
// ✅ 推荐：使用一致的 token
<Button color="primary" />
<Chip color="primary" />

// ❌ 不推荐：使用不一致的颜色
<Button sx={{ backgroundColor: '#844025' }} />
<Chip sx={{ backgroundColor: '#844025' }} />
```

#### 行为一致性

相似组件有相似的交互模式和反馈机制：

- 所有可点击元素有一致的悬停和点击反馈
- 所有表单控件有一致的聚焦和错误状态
- 所有加载状态有一致的指示器样式
例如，按钮组件的交互状态：

```
// 按钮组件的交互状态定义
...(ownerState.variant === 'solid' && {
  backgroundColor: theme.vars.palette.brand.burntOrange[500],
  ':hover': {
    backgroundColor: theme.vars.palette.brand.burntOrange[700],
  },
  ':focus': {
    backgroundColor: theme.vars.palette.brand.burntOrange[700],
  },
  ':active': {
    backgroundColor: theme.vars.palette.brand.burntOrange[800],
  },
}),
```

#### API 一致性

组件接口设计遵循一致的命名和结构模式：

```
// 一致的 API 设计
<Button variant="solid" color="primary" size="md" disabled={false} />
<IconButton variant="solid" color="primary" size="md" disabled={false} />
<Checkbox color="primary" size="md" disabled={false} />
```

### 2.2.2 可访问性原则

Fortress 2.0 将可访问性作为设计的基础要求，确保所有用户都能使用我们的组件。

#### 颜色对比度

所有文本与背景的对比度符合 WCAG AA 标准：

| 文本大小 | 最小对比度 | 我们的标准 |
| 小文本 (<18px) | 4.5:1 | 4.5:1 或更高 |
| 大文本 (≥18px) | 3:1 | 3.5:1 或更高 |


例如，主要文本颜色 `maroonVelvet[500]` (#3C101E) 与白色背景的对比度约为 16:1，远高于标准要求。

#### 键盘可访问性

所有交互组件支持键盘操作：

- 按钮、链接可通过 Enter 键激活
- 表单控件可通过 Tab 键聚焦
- 下拉菜单可通过方向键导航
- 模态框捕获并管理焦点
> 在使用时注意提供适当的ARIA 属性和语义化结构

```
// 正确使用 ARIA 属性
<Button
  aria-label="添加项目"
  aria-pressed={isPressed}
>
  <AddIcon />
</Button>
```

### 2.2.3 响应式设计原则

Fortress 2.0 的组件默认支持响应式设计，确保在各种设备上提供良好的用户体验。

#### 断点系统

我们定义了一致的断点系统：

| 断点名称 | 宽度范围 | 典型设备 |
| xs | <600px | 手机竖屏 |
| sm | ≥600px | 手机横屏、小平板 |
| md | ≥900px | 平板、小笔记本 |
| lg | ≥1200px | 笔记本、桌面 |
| xl | ≥1536px | 大屏桌面 |


这些断点在排版系统中广泛使用，确保文本在不同设备上都有最佳的可读性：

```
// 响应式排版示例
h1: {
  // 移动端
  '@media (min-width: 0px) and (max-width: 600px)': {
    fontSize: `var(--fortress-fontSize-xl2)`,
  },
  // 平板
  '@media (min-width: 601px) and (max-width: 900px)': {
    fontSize: `var(--fortress-fontSize-xl2)`,
  },
  // 桌面
  '@media (min-width: 901px)': {
    fontSize: `var(--fortress-fontSize-xl5)`,
  },
},
```

### 2.3.4 CSS 变量前缀

使用自定义前缀来避免与其他库的 CSS 变量冲突：

```
cssVarPrefix: 'fortress',
```

这使得所有生成的 CSS 变量都带有 `--fortress-` 前缀，例如 `--fortress-palette-primary-500`。

# 4. Chromatic 视觉测试与 CDD

## 4.1 视觉测试在 CDD 中的角色

### 4.1.1 防止视觉回归

在组件驱动开发（CDD）过程中，**保持视觉一致性** 是一项基础但关键的工作。我们使用 Chromatic 进行视觉测试，确保每个组件在不同开发阶段都能保持预期的 UI 表现，避免“没注意就变了”的问题。

Chromatic 的几个关键优势包括：

- **自动捕捉细微差异**
即使是像边距、颜色明暗、字体粗细这种微小的视觉调整，也能被准确识别，防止无意中的视觉回归。
- **跨浏览器一致性保障**
通过不同环境下的快照对比，帮助我们验证组件在主流浏览器中的一致性表现。
- **组件历史可追溯**
每次改动都会生成一个视觉快照记录，团队可以随时查看组件外观的变化历史，做到“谁改了什么，一目了然”。
```
flowchart TD
    A[组件修改] --> B[Storybook 构建]
    B --> C[Chromatic 捕获快照]
    C --> D{与基准比较}
    D -->|无变化| E[测试通过]
    D -->|有变化| F[需要审查]
    F --> G{变化是否预期?}
    G -->|是| H[接受变更]
    G -->|否| I[修复代码]
    I --> A
    H --> E
```

### 4.1.2 确保一致性

Fortress 2.0 是由多个开发者协作维护的组件库，在这样的多人协作环境中，保持组件的视觉一致性尤为重要。我们通过引入 Chromatic 的视觉测试机制，把一致性变成一种**可见、可控、可审查的流程**。

- **统一的视觉基准**
所有团队成员基于相同的组件快照进行开发和审查，避免因环境差异或习惯差异导致“看起来不一样”。
- **清晰的变更审核流程**
每一次视觉层面的变动，都会触发自动化对比，只有经过评审确认后才会被接受。这让“视觉修改”成为有据可查、明确把关的过程，而不是“改了但谁也不知道”。
- **促进设计与开发协作**
设计师和开发者可以在可视化界面中一起审阅每次 UI 变更，避免因认知偏差产生误解，也更容易达成一致决策。
通过这些机制，Fortress 在多人协作和持续迭代的背景下，仍能保持稳定、统一的用户界面表现。

## 4.2 本地开发视觉测试

### 4.2.1 Visual Tests 的使用方法

在 Fortress 2.0 的本地开发流程中，我们通过 Storybook 内集成的 **Visual Tests** 功能，方便地运行视觉测试，验证组件是否出现了意外的视觉变更。

本地使用流程如下：

1. - **进入 Storybook**
在本地启动 Storybook 后，点击顶部导航栏中的 **"Visual Tests"** 入口。
2. - **首次使用需登录**
第一次使用时，你需要登录你的 Chromatic 账号，并选择对应的项目。登录完成后，系统会自动在项目根目录下生成 `chromatic.config.json` 文件，用于记录配置。
3. - **触发视觉测试**
登录后，点击页面中的 **"Run Test"** 按钮，即可在本地触发一次视觉测试。这一操作会调用 `npm run chromatic` 命令，并将当前组件快照上传到 Chromatic 进行比对。
```
sequenceDiagram
    participant D as 开发者
    participant S as Storybook
    participant C as Chromatic
    
    D->>S: 点击 "Visual Tests" 按钮
    S->>D: 请求登录 Chromatic
    D->>C: 登录并选择项目
    C->>S: 生成配置文件
    D->>S: 点击 "Run Test"
    S->>C: 执行 chromatic 命令
    C->>C: 构建并上传快照
    C->>S: 返回测试结果
    S->>D: 显示测试结果
```

### 4.2.2 测试结果展示与解读

当你在本地触发 Visual Tests 后，Chromatic 会在 Storybook 中直接展示测试结果，反馈非常直观：

- ✅ **通过（绿色）**：组件外观与之前的基准一致，无需任何处理；
- ⚠️ **需要审查（黄色）**：检测到组件外观发生变化，需要人工确认是否为预期变更；
- ❌ **失败（红色）**：测试过程中出现错误，通常与配置或构建问题有关，需先修复。
![](attachment)



对于 **需要审查** 的变更，界面会展示非常详细的对比信息，包括：

- **视觉对比图**：组件变更前后的截图并排展示；
- **高亮区域**：变动区域用红框或阴影高亮标出，快速聚焦；
- **技术细节**：展示底层的 CSS 属性变更，方便判断是否为预期调整;
![](attachment)



### 4.2.3 如何处理视觉变更

当检测到组件发生视觉变化时，我们建议按以下流程进行处理，以确保变更是合理且可追踪的：

1. - **认真审查变更内容**
查看高亮区域和对比图，确认是否真的是预期内的改动。
2. - **判断是否为预期改动**

- 如果是有意修改（如改了样式、修复了 UI 问题）：可以接受变更；
- 如果是意外引入（如重构带来的样式抖动）：应回退或修复代码。
3. - **处理方式参考**

- **确认无误 → 接受变更**
接受时请务必添加注释或备注，说明变更目的，方便后续审查。
- **变更不合理 → 修复并重测**
修改组件代码后，重新运行测试，直到变更被接受或通过。
4. - **更新视觉基准**
一旦确认变更为“预期内的调整”，可以更新 Chromatic 的视觉基准，这样下一次测试将以当前状态为新参考点。
```
flowchart TD
    A[检测到视觉变更] --> B{是否有意为之?}
    B -->|是| C[接受变更]
    B -->|否| D[修复代码]
    C --> E[添加注释说明原因]
    E --> F[更新视觉基准]
    D --> G[重新运行测试]
    G --> H{测试通过?}
    H -->|是| I[完成]
    H -->|否| D
```

### 4.3.1 PR 自动触发测试

Fortress 2.0 设计系统通过 GitHub Actions 实现了 Chromatic 视觉测试的自动化集成。测试会在以下情况自动触发：

1. - **标签推送**：当推送符合 `fortress-v*.*.*` 模式的标签时
2. - **PR 创建/重新打开**：当针对 master 分支创建或重新打开 PR，且修改了 fortress 目录时
3. - **PR 同步**：当 PR 更新且检测到 fortress 目录有变更时
```
flowchart TD
    A[代码变更] --> B{变更类型?}
    B -->|标签推送| C[触发 Chromatic]
    B -->|PR 创建/重新打开| D{修改 fortress?}
    B -->|PR 同步| E{检测 fortress 变更}
    D -->|是| C
    D -->|否| F[不触发测试]
    E -->|有变更| C
    E -->|无变更| F
```

这种自动化配置确保了每次相关代码变更都会进行视觉测试，防止意外的视觉回归。

### 4.3.2 基准分支配置

Fortress 2.0 使用模块化的工作流结构实现 Chromatic 测试：

```
sequenceDiagram
    participant GH as GitHub
    participant CI as GitHub Actions
    participant C as Chromatic
    
    GH->>CI: 触发工作流
    CI->>CI: 检出代码
    CI->>CI: 安装依赖
    CI->>C: 发送组件快照
    C->>C: 比较与基准的差异
    C->>CI: 返回测试结果
    CI->>GH: 更新 PR 状态
    C->>GH: 添加结果评论
```

关键配置参数包括：

- **分支信息**：确保测试使用正确的分支作为比较基准
- **工作目录**：指定 `libs/fortress` 作为测试目录
- **变更范围**：默认测试所有组件，而非仅变更部分
这种配置确保了视觉变更的准确评估，同时支持并行开发流程。

## 4.4 视觉审查协作机制


视觉测试不仅仅是自动对比工具，更是 **设计与开发协作审查 **的桥梁。在 Fortress 2.0 中，我们将视觉回归的评估责任清晰划分，确保每一个变更既符合设计规范，也在技术上合理可控。

### 4.4.1 开发人员的职责

**开发者在视觉变更流程中，主要负责以下内容：**

- 技术实现检查：确认组件变更是否符合实现预期与代码规范；
- 功能完整性验证：确保组件在所有状态和交互下都正常工作；
- 性能评估：识别潜在的性能问题或 DOM/样式冗余；
- 初步视觉判断：及时发现明显的视觉异常（如错位、颜色错误等）。
**开发者视觉变更审查：**

- 是否符合设计规范（间距、色彩、层级等）？
- 所有组件变体、状态是否均表现正常？
- 是否可能影响其他组件或样式？
- 是否引入了潜在的性能隐患？
### 4.4.2 设计师的职责

**设计师在审查流程中提供最终的视觉把关：**

- 品牌一致性确认：确保变更不偏离品牌视觉语言；
- 细节审查：关注细微的 UI 表现，包括间距、对齐、排版等；
- 体验评估：从用户角度判断改动是否改善或影响使用体验；
- 最终视觉确认：决定该变更是否可以合并至主基准。
**设计师视觉变更审查：**

- 是否遵循 Figma 中的设计规范与品牌语言？
- 是否存在细节偏差（边距、色差、排版等）？
- 是否在不同分辨率或组件状态下保持一致性？
- 是否提升或损害了整体的用户体验？
```
sequenceDiagram
    participant D as 开发者
    participant C as Chromatic
    participant DS as 设计师
    
    D->>C: 提交代码变更
    C->>C: 运行视觉测试
    C->>D: 返回测试结果
    D->>D: 技术审查
    D->>DS: 请求设计审查
    DS->>C: 审查视觉变更
    DS->>D: 提供反馈
    alt 需要修改
        D->>D: 实施修改
        D->>C: 重新测试
    else 批准变更
        DS->>C: 批准变更
        C->>C: 更新基准
    end
```

## 4.5 视觉测试最佳实践

### 4.5.1 处理有意的视觉变更

对于开发过程中明确预期的视觉改动，我们建议遵循以下几项最佳实践，以提高审查效率并避免误解：

- **清楚说明变更背景**
在 PR 描述中明确指出视觉变更的原因（例如规范更新、设计稿调整等），并概述涉及的组件或页面范围。
- **关注点分离**
尽可能将视觉变更与功能逻辑变更拆分到不同的 PR 或 commit 中，方便审查者聚焦每类改动。
- **提供上下文信息**
在 PR 中附上设计参考图或 Figma 链接，让设计师和其他开发者快速理解变更意图。
- **迭代控制**
控制每次变更的影响范围，小而明确的视觉调整更容易回顾和验证，也降低了回滚或修改成本。
### 4.5.2 误报处理方法

视觉测试有时会产生误报，我们采用以下策略处理：

1. - **识别常见误报类型**：

- 字体渲染差异（特别是在不同操作系统间）
- 动画和过渡状态捕获
- 第三方组件渲染不一致
2. - **调整测试阈值**：

- 为特定组件设置自定义比较阈值
- 忽略已知会产生误报的微小差异
3. - **使用测试修饰符**：

- 为动画组件添加 `chromatic={{ disableSnapshot: true }}` 禁用快照
- 使用 `chromatic={{ delay: 300 }}` 等待组件完全渲染
4. - **维护测试基础设施**：

- 定期更新 Chromatic 和 Storybook 版本
- 确保测试环境的一致性
**例如：处理动画组件的测试配置**

```
// 在 Story 中禁用动画以避免误报
export const WithAnimation: Story = {
  parameters: {
    chromatic: { disableSnapshot: false, delay: 300 },
  },
  render: () => <AnimatedComponent />
};
```
