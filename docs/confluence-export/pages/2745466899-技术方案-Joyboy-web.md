## 2. React Server Component、Client Component

### 2.1 React Server Components

 

React 服务器组件允许编写可在服务器上渲染并可选缓存的 UI。
只要是在 `app` 目录内的页面，全部默认为服务端组件。

#### 2.1.1 服务端组件的优势

1. - **数据获取**：服务端组件具备完整的服务端能力，所以可以直接与数据库或其他数据源进行交互。这消除了需要从客户端到服务器的额外请求，从而加速了数据获取。
2. - **安全性**：因为渲染是在服务器上进行的，所以可以在服务器上使用敏感的 API 密钥或令牌，而不必担心它们被暴露给客户端用户。
3. - **缓存**：服务端渲染的结果可以被缓存。这意味着对于经常被访问的页面，你可以存储已经渲染的HTML，从而快速地为后续的请求提供响应，而不必每次都重新渲染。
4. - **包大小**：一些库或框架可能非常大，如果全部发送到客户端，会增加首次加载时间。通过在服务器上使用这些库，你可以避免增加客户端的包大小。
5. - **初始页面加载和首次内容绘制 (FCP)**：服务端渲染的页面可以立即为用户提供可见的内容，而不必等待客户端JavaScript加载和执行。
6. - **搜索引擎优化和社交网络分享**：服务端渲染的页面为搜索引擎提供了完整的HTML内容，这有助于提高SEO排名。此外，社交媒体平台也可以预览这些页面，从而提高分享的吸引力。
7. - **流式传输**：这是一个更高级的优化。你可以将页面渲染分成多个部分，并在它们准备好时发送到客户端。这允许用户更早地看到部分内容，而不必等待整个页面在服务器上渲染完成。
#### 2.2.2 组件渲染


在 Next.js 中，渲染工作进一步按路由段拆分，以实现流式渲染和部分渲染，并且有三种不同的服务器渲染策略：

1. - [静态渲染](https://nextjs.org/docs/app/building-your-application/rendering/server-components#static-rendering-default)（默认）
- 路由会在构建时进行渲染，或在数据重新验证后在后台运行
- 结果会被缓存，并可推送到内容分发网络（CDN）
这种渲染方式适合静态博客或产品介绍页面（包含未针对用户个性化且可以在构建时知道的数据）

1. - [动态渲染](https://nextjs.org/docs/app/building-your-application/rendering/server-components#dynamic-rendering)
当路线具有针对用户个性化的数据或具有只能在请求时知道的信息（例如 cookie 或 URL 的搜索参数）时，**通过动态渲染，可以在请求时**为每个用户渲染路线

在渲染过程中，如果发现[动态函数](https://nextjs.org/docs/app/building-your-application/rendering/server-components#dynamic-functions)或未[缓存的数据请求](https://nextjs.org/docs/app/building-your-application/data-fetching/fetching-caching-and-revalidating#opting-out-of-data-caching)，Next.js 将切换到动态渲染整个路由。下表总结了动态函数和数据缓存如何影响路由是静态渲染还是动态渲染：

| 是否有动态方法 | 数据是否缓存 | 路由渲染方式 |
| 否 | 是 | 静态渲染 |
| 是 | 是 | 动态渲染 |
| 否 | 否 | 动态渲染 |
| 是 | 否 | 动态渲染 |


1. - [流媒体](https://nextjs.org/docs/app/building-your-application/rendering/server-components#streaming)
![](attachment)

流式传输可从服务器逐步渲染 UI。工作被拆分成多个块，并在准备就绪时流式传输到客户端。这样，用户就可以在整个内容渲染完成之前立即看到页面的某些部分

![](attachment)

默认情况下，流式传输内置于 Next.js 应用路由器中。这有助于改善初始页面加载性能，以及依赖于较慢数据提取的 UI，因为较慢的数据提取会阻止渲染整个路由。例如，产品页面上的评论

在 NextJS 中，也可以使用 loading.js 对路由段进行流式处理，并使用 React Suspense 对 UI 组件进行流式输出处理



#### 2.2.3 总体介绍

| **React Server Components** |
| **渲染方式** | 在服务器端渲染，并将生成的 HTML 发送到客户端。 |
| **执行环境** | 运行在 Node.js 或 edge 环境中，可以直接访问服务器资源和后端API |
| **数据获取** | 可以直接在组件中进行数据获取，无需担心 CORS 问题，可以使用任何 Node.js 兼容的 API（fetch / 自定义请求方式） |
| **状态和交互** | 处理静态页面部分 |


呈现方式及原理（与后续的 cache 缓存模块有关！！！）

1. - 在服务器上，Next.js 使用 React 的 API 来协调渲染。渲染工作被分成几个部分：按单个路由段和[Suspense Boundaries](https://react.dev/reference/react/Suspense)。

每个块的渲染分为两个步骤：

1. - React 将服务器组件渲染为一种特殊的数据格式，称为 **React 服务器组件负载（RSC Payload）**
2. - Next.js 使用 RSC Payload 和客户端组件 JavaScript 指令在服务器上呈现 **HTML **
2. - 在客户端上：

1. - HTML 用于立即显示路线的快速非交互式预览 - 这仅适用于初始页面加载。
2. - React Server Components Payload 用于协调客户端和服务器组件树，并更新 DOM。
3. - JavaScript 指令用于[水合](https://react.dev/reference/react-dom/client/hydrateRoot)客户端组件并使应用程序具有交互性。
> **什么是 React 服务器组件有效负载 (RSC)？**

RSC Payload 是渲染的 React Server Components 树的紧凑二进制表示。React 在客户端使用它来更新浏览器的 DOM。RSC Payload 包含：

- 服务器组件的渲染结果
- 客户端组件应呈现的位置的占位符以及对其 JavaScript 文件的引用
- 从服务器组件传递到客户端组件的任何 props

### 2.2 Client Components

 

客户端组件允许编写[在服务器上预渲染的交互式 UI](https://github.com/reactwg/server-components/discussions/4) 并可使用客户端 JavaScript 在浏览器中运行
在app文件夹中，默认是服务端组件，如果要使用客户端组件，需要在文件顶部添加 `"use client"`

#### 2.2.1 客户端组件的优势

- 交互性：客户端组件可以使用状态、效果和事件监听器，这意味着它们可以向用户提供即时反馈并更新用户界面。
- 浏览器 API：客户端组件可以访问浏览器 API，如 window、localStorage，从而为特定用例构建用户界面
使用了`"use client"`等于声明了服务器和客户端组件模块之间的边界，如果文件顶部有该指令，则导入到其中的所有子组件和模块都被视为客户端包的一部分



![](attachment)

#### 2.2.2 客户端组件页面加载

为优化初始页面加载，Next.js 将使用 React 的 API 在服务器上为客户端和服务器组件呈现静态 HTML 预览。这意味着，当用户首次访问应用程序时将立即看到页面内容，而无需等待客户端下载、解析和执行客户端组件 JavaScript 包

**在服务器上：**

1. - React 将服务器组件渲染为一种称为 **React 服务器组件负载（RSC Payload）**的特殊数据格式，其中包含对客户端组件的引用
2. - Next.js 使用 RSC Payload 和客户端组件 JavaScript 指令在服务器上呈现路由的**HTML **
**在客户端上：**

1. - HTML 用于立即显示路线的快速非交互式初始预览
2. - React Server Components Payload 用于协调客户端和服务器组件树，并更新 DOM
3. - JavaScript 指令用于[水合](https://react.dev/reference/react-dom/client/hydrateRoot)客户端组件并使其 UI 具有交互性
#### 2.2.3 后续导航

- 在后续导航中，客户端组件完全在客户端上呈现，而无需服务器呈现的 HTML
- 这意味着客户端组件 JavaScript 包已下载并解析。一旦包准备就绪，React 将使用 RSC Payload 协调客户端和服务器组件树，并更新 DOM
#### 2.2.4 总体介绍

| **Client Components** |
| **渲染方式** | 在服务端也会预渲染，但状态、交互部分会在客户端通过水合完成 |
| **执行环境** | 运行在浏览器环境中，受限于浏览器的安全和性能限制 |
| **数据获取** | 数据获取通常使用 axios 进行，需注意 CORS 和跨域请求（项目中为 rtk query） |
| **状态和交互** | 适合处理交互性较强的部分，可以使用 React的状态和生命周期方法 |


## 3. 混合开发（RSC + CC）模式

 

### 3.1 两种组件的使用场景区别

![](attachment)

  

> 使用 RSC 时无法使用 RTK 的api 请求方式（没有状态），可使用 NextJS 优化后的 fetch 方法，自带 缓存能力，或者结合 unstable_cache 与自定义请求方式实现服务端数据请求

状态管理：

1. - 客户端应根据请求创建存储
2. - RSC 不能使用钩子或上下文。它们不具有状态。让 RSC 从全局存储读取或写入值违反了 Next.js 应用路由器的架构

### 3.2 React Server Components 最佳实践

#### **3.2.1 在组件之间共享数据**

当在服务器上获取数据时，可能需要在不同的组件之间共享数据。与其使用 React Context（在服务器上不可用）或传递数据作为 props，不如使用 `fetch` 或 React 的 `cache` 函数在需要的组件中获取相同的数据，React 将对重复请求进行合并

#### 3.2.2 **服务端代码隔离客户端组件**

由于 JavaScript 模块可以在服务器和客户端组件模块之间共享，因此原本只打算在服务器上运行的代码可能会潜入客户端

```
import 'server-only'

export async function getData() {
  const res = await fetch('https://external-service.com/data', {
    headers: {
      authorization: process.env.API_KEY,
    },
  })
 
  return res.json()
}
```

- 由于环境变量`API_KEY`没有前缀`NEXT_PUBLIC`，因此它是一个只能在服务器上访问的私有变量。为了防止环境变量泄露给客户端，Next.js 会将私有环境变量替换为空字符串
- 因此，即使`getData()`可以在客户端上导入和执行，它也不会按预期工作。虽然将变量公开会使函数在客户端上工作，但开发人员可能不想向客户端公开敏感信息
- 为了防止这种客户端意外使用服务器代码的情况，我们可以使用`server-only`包（仅限代码在服务端运行），防止其他开发意外将其中一个模块导入客户端组件，并在构建时输出错误信息
```
// 需要先下载该包
npm install server-only
```

`client-only` 则可用于标记包含仅限客户端代码的模块 - 例如，访问对象的代码`window`

#### 3.2.3 兼容第三方包

由于服务端组件是一个新的 React 功能，许多 npm 包可能还没有添加`use client`指令。如果一个第三方组件在服务端组件中不起作用，可以将其封装在客户端组件中来解决这个问题

例子：

```
// server component
import { Carousel } from 'acme-carousel'
 
export default function Page() {
  return (
    <div>
      <p>View pictures</p>
 
      {/* Error: `useState` can not be used within Server Components */}
      <Carousel />
    </div>
  )
}
```

解决方案 ：

```
'use client'
 
import { Carousel } from 'acme-carousel'
 
export default Carousel
```

```
import Carousel from './carousel'
 
export default function Page() {
  return (
    <div>
      <p>View pictures</p>
 
      {/*  Works, since Carousel is a Client Component */}
      <Carousel />
    </div>
  )
}
```

#### 3.2.4 Context Provider 设计

由于[React 上下文](https://react.dev/learn/passing-data-deeply-with-context)在服务器组件中不受支持，尝试在应用程序的根目录下创建上下文将导致错误

```
// app/layout.tsx
import { createContext } from 'react'
 
//  createContext is not supported in Server Components
export const ThemeContext = createContext({})
 
export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        <ThemeContext.Provider value="dark">{children}</ThemeContext.Provider>
      </body>
    </html>
  )
}
```

解决方案：

创建客户端组件包裹上下文

```
'use client'
 
import { createContext } from 'react'
 
export const ThemeContext = createContext({})
 
export default function ThemeProvider({
  children,
}: {
  children: React.ReactNode
}) {
  return <ThemeContext.Provider value="dark">{children}</ThemeContext.Provider> 
}
// ThemeContext 仅作为插槽确定子组件在页面的渲染位置，通过 children 的形式可以包裹使用服务端组件
```

layout 服务端组件中导入该客户端组件作为子组件

```
import ThemeProvider from './theme-provider'
 
export default function RootLayout({
  children,
}: {
  children: React.ReactNode
}) {
  return (
    <html>
      <body>
        <ThemeProvider>{children}</ThemeProvider>
      </body>
    </html>
  )
}
```

### 3.3 Client Components 最佳实践

#### **3.3.1 客户端组件沿树向下移动**

为减少客户端 JavaScript 包的大小，建议将客户端组件从组件树中向下移动

例如，有一个`<SearchBar />`的组件，这个组件需要交互，所以只能是客户端组件

如果要加快渲染速度，就可以把页面放在服务端渲染，而`<SearchBar />`作为客户端组件引入

```
// app/layout.tsx
// SearchBar is a Client Component
import SearchBar from './searchbar'
// Logo is a Server Component
import Logo from './logo'
 
// Layout is a Server Component by default
export default function Layout({ children }: { children: React.ReactNode }) {
  return (
    <>
      <nav>
        <Logo />
        <SearchBar />
      </nav>
      <main>{children}</main>
    </>
  )
}
```

联系上文，这样分离的方式，NextJS 14 会优先进行服务端组件渲染，而此时`<SearchBar />`的位置会是一个占位符，一旦开始混合渲染，占位符的位置会被真实的`<SearchBar />`填充

#### 3.3.2 **从服务器组件传递 props 到客户端组件（可序列化数据）**

考虑以下场景：服务器组件负责获取用户的信息，然后你希望将这些信息传递给一个客户端组件以展示一个交互式的用户资料卡

在这种情境中的操作：

- 在服务器组件中预获取用户数据。
- 将数据作为 `props` 传递给客户端组件。
- 为了使这个过程有效且无错误，从服务器组件传递给客户端组件的 `props` 必须是可序列化的
> **什么是“可序列化的”？**

基本的数据类型（如数字、字符串、数组和普通对象）通常都是可序列化的。但是，如函数、特定的对象实例（例如 `Date` 对象或自定义类的实例）或包含循环引用的对象可能不是

> **怎么处理不可序列化的数据？**

如果客户端组件依赖于不可序列化的数据，有两个选项：

- **客户端获取**：在客户端组件中直接获取数据，比如使用 AJAX 请求、RTK 请求
- **路由处理程序**：在 NextJS 中，可以使用 API 路由来在服务器上获取数据，并从客户端进行调用

> ### 3.4 服务端组件与客户端组件混合开发

当混合使用客户端和服务器组件时，应当将整体 UI 视为组件树。从根布局开始（它是一个服务器组件），你应该通过添加 "use client" 指令在客户端渲染某些子树。

> 虽然在这些客户端子树中仍然可以嵌套服务器组件或调用服务器操作，但仍有注意事项：

- 如果需要在客户端访问服务器上的数据或资源，客户端需要向服务器发出新的请求
- 当向服务器发出新请求时，首先渲染所有服务器组件，在客户端上，React 会使用 RSC Payload 将服务器组件和客户端组件调和成一棵树
- 由于客户端组件是在服务器组件之后呈现的，因此不能将服务器组件导入到客户端组件中。正确的方式应该是：将服务器组件作为属性传递给客户端组件（如：children），这种方式允许你在客户端组件中使用来自服务器组件的数据或内容，而不必再次请求服务器

#### 3.2.1 客户端组件引用服务端组件

### 6.1 request memorization

 React 扩展了`fetch`API（服务端组件中调用接口数据），自动**记忆**了具有相同 URL 和选项的请求。这意味着您可以在 React 组件树中的多个位置为同一数据调用 fetch 函数，只需执行一次真实请求

1. - 如果需要跨路由使用相同的数据（例如在布局、页面和多个组件中），则不必在树的顶部获取数据，也不必在组件之间转发 props。相反，可以在需要它的组件中获取数据，不必担心通过网络对同一数据进行多次请求对性能的影响
![](attachment)

#### 6.1.1 request memorization 工作原理

```
async function getItem() {
  // The `fetch` function is automatically memoized and the result
  // is cached
  const res = await fetch('https://.../item/1')
  return res.json()
}
 
// This function is called twice, but only executed the first time
const item = await getItem() // cache MISS
 
// The second call could be anywhere in your route
const item = await getItem() // cache HIT
```

![](attachment)

1. - 在渲染路线时，第一次调用特定请求时，其结果将不会在内存中，而是缓存中`MISS`。
2. - 因此，将执行该函数，从外部源获取数据，并将结果存储在内存中。
3. - 同一个渲染过程中，该请求的后续函数调用将会被缓存`HIT`，数据将从内存中返回，而无需执行该函数。
4. - 一旦路线被渲染并且渲染过程完成，内存就会“重置”并且所有请求记忆条目都会被清除。
> 1. - 请求记忆是React的特性，而不是Next.js的特性
2. - 记忆只适用于获取请求中的GET方法
3. - Memoization只适用于React Component树，这意味着:

1. - 它适用于generateMetadata、generateStaticParams、Layouts、Pages和其他服务器组件中的获取请求
2. - 它不适用于路由处理程序中的获取请求，因为它们不是React组件树的一部分
3. - 对于不适合fetch的情况(例如一些数据库客户端，CMS客户端或GraphQL客户端)，您可以使用React缓存功能来记忆函数

**6.1.2 request memorization duration:**

1. - 缓存持续服务器请求的整个生命周期，直到 React 组件树完成渲染
**6.1.3 revalidate:**

1. - 不需要重新验证，因为 memorization 不会在服务器请求之间共享，而仅在渲染期间适用
**6.1.4 opting out：**

1. - Memoization只应用于 GET 方法中的 fetch 请求，其他方法(如 POST 和 DELETE )不被 Memoization。这个默认行为是 React 优化的结果，不建议选择退出
2. - 为了管理单个请求，可以使用 AbortController 中的 signal 属性。然而，这不会选择请求退出记忆，而是中止正在执行的请求
```
const { signal } = new AbortController()
fetch(url, { signal })
```

### 6.2 data cache

Next.js 具有内置数据缓存，可**持久保存传入服务器请求**和**部署**中的数据提取结果。 这是因为 Next.js 扩展了本机`fetch`API，以允许服务器上的每个请求设置自己的持久缓存语义

默认情况下，使用fetch的数据请求将被缓存。可使用缓存和next。重新验证获取选项以配置缓存行为

  

#### 6.2.1 工作原理

![](attachment)

1. - 在渲染期间第一次`fetch`调用请求时，Next.js 会检查数据缓存中是否存在缓存的响应
2. - 如果发现缓存的响应，则立即返回并[记忆](https://nextjs.org/docs/app/building-your-application/caching#request-memoization)
3. - 如果未找到缓存的响应，则向数据源发出请求，将结果存储在数据缓存中并进行记忆
4. - 对于未缓存的数据（例如`{ cache: 'no-store' }`），结果总是从数据源获取并记忆
5. - 无论数据是否缓存，请求总是会被记住，以避免在 React 渲染过程中对相同数据进行重复请求
**duration**

除非重新验证或选择退出，否则数据缓存在传入的请求和部署中是持久的

**revalidate**

可以通过两种方式重新验证缓存数据：

- **基于时间的重新验证**：经过一段时间并发出新请求后重新验证数据。这对于不经常更改且新鲜度不那么重要的数据很有用

- ```
// Revalidate at most every hour
fetch('https://...', { next: { revalidate: 3600 } })
```

或者使用[路线段配置选项](https://nextjs.org/docs/app/building-your-application/caching#segment-config-options)来配置`fetch`某个段中的所有请求
- 工作原理：
- ![](attachment)

1. - 第一次调用获取请求时`revalidate`，数据将从外部数据源获取并存储在数据缓存中
2. - 在指定时间范围内（例如 60 秒）调用的任何请求都将返回缓存数据
3. - 在该时间范围之后，下一个请求仍将返回缓存的（现在已过时的）数据

1. - Next.js 将在后台触发数据的重新验证
2. - 一旦成功获取数据，Next.js 将使用新数据更新数据缓存
3. - 如果后台重新验证失败，之前的数据将保持不变
- **按需重新验证：**根据事件（例如表单提交）重新验证数据。按需重新验证可以使用基于标签或基于路径的方法一次性重新验证数据组。当想确保尽快显示最新数据时（例如，当无头 CMS 中的内容更新时）

- 可以根据需要通过路径 ( `revalidatePath`) 或缓存标签 ( `revalidateTag`) 重新验证数据
- 工作原理
- ![](attachment)

1. - 第一次`fetch`调用请求时，数据将从外部数据源获取并存储在数据缓存中
2. - 当触发按需重新验证时，相应的缓存条目将从缓存中清除

1. - 这与基于时间的重新验证不同，后者将陈旧数据保留在缓存中，直到获取新数据
3. - 下次请求的时候会`MISS`再次进行缓存，从外部数据源中取出数据并存放到Data Cache中

**revalidatePath：**



**opting out**

1. - 对于单独的数据获取，可以通过将缓存选项设置为 no-store 来选择退出缓存。这意味着无论何时调用fetch，都会获取数据

```
// Opt out of caching for an individual `fetch` request
fetch(`https://...`, { cache: 'no-store' })
```
2. - 也可以使用[路由段配置选项](https://nextjs.org/docs/app/building-your-application/caching#segment-config-options)来选择退出特定路由段的缓存。这将影响路由段中的所有数据请求，包括第三方库

```

export const dynamic = 'force-dynamic'
```

#### **6.2.2 data cache 和 request memorization 之间的差异**

- 虽然两种缓存机制都可以通过重新使用缓存数据来提高性能，但是数据缓存在传入的请求和部署中是持久的，而记忆仅持续请求的整个生命周期
- 通过记忆化，我们可以减少同一渲染过程中必须跨越网络边界从渲染服务器到数据缓存服务器（例如 CDN 或边缘网络）或数据源（例如数据库或 CMS）的**重复**请求数量。通过数据缓存，我们可以减少对原始数据源发出的请求数量

### 6.3  route cache

#### 6.3.1 full router cache

Next.js 会在构建时自动渲染和缓存路由。这是一项优化，允许您提供缓存的路由，而不是针对每个请求在服务器上进行渲染，从而加快页面加载速度

要了解完整路由缓存的工作原理，了解 React 如何处理渲染以及 Next.js 如何缓存结果会很有帮助：

##### 服务器上的 React 渲染

在服务器上，Next.js 使用 React 的 API 来协调渲染。渲染工作被分成多个部分：按单独的路由段和 Suspense 边界。

每个块的渲染分为两个步骤：

1. - React 将服务器组件渲染为一种针对流式传输进行了优化的特殊数据格式，称为**React 服务器组件有效负载 (RSP)**。
2. - Next.js 使用 React Server Component Payload 和 Client Component JavaScript 指令在服务器上呈现**HTML 。**
这意味着我们不必等待所有内容呈现完毕再缓存工作或发送响应。相反，我们可以在工作完成时流式传输响应。

##### Next.js 服务器缓存（完整路由缓存）

![](attachment)

Next.js 的默认行为是将路由的渲染结果（React Server Component Payload 和 HTML）缓存在服务器上。这适用于构建时或重新验证期间静态渲染的路由

##### 客户端上的 React Hydration 和 Reconciliation

在请求时，在客户端上：

2. - React Server Components Payload 用于协调客户端和渲染的服务器组件树，并更新 DOM

##### 后续导航

在后续导航或预取期间，Next.js 将检查 React Server Components Payload 是否存储在路由器缓存中。 如果是，它将跳过向服务器发送新请求

如果路由段不在缓存中，Next.js 将从服务器获取 React Server Components Payload，并填充客户端上的路由器缓存

##### 6.3.1.1 静态和动态渲染

路由在构建时是否缓存取决于它是静态还是动态渲染。

- 静态路由默认缓存
- 动态路由在请求时渲染，不缓存
![](attachment)

**期间**

默认情况下，完整路线缓存是持久性的。这意味着渲染输出会在用户请求之间缓存。

**无效化**

有两种方法可以使完整路线缓存无效：

- [**重新验证数据**](https://nextjs.org/docs/app/building-your-application/caching#revalidating)：重新验证[数据缓存](https://nextjs.org/docs/app/building-your-application/caching#data-cache)，反过来会通过重新渲染服务器上的组件并缓存新的渲染输出使路由器缓存无效。
- **重新部署**：与跨部署持久保留的数据缓存不同，完整路线缓存在新的部署中会被清除。
**选择退出（opting out）**

您可以选择退出完整路由缓存，或者换句话说，为每个传入请求动态呈现组件，方法如下：

- **使用**[**动态函数**](https://nextjs.org/docs/app/building-your-application/caching#dynamic-functions)：这将从完整路由缓存中选择路由，并在请求时动态呈现它。数据缓存仍可使用
- **使用**`dynamic = 'force-dynamic'`或`revalidate = 0`路由段配置选项：这将跳过完整路由缓存和数据缓存。这意味着组件将在每次传入服务器的请求时呈现并获取数据。路由器缓存仍将适用，因为它是客户端缓存（页面中配置对应字段参数）
- **选择退出**[**数据缓存**](https://nextjs.org/docs/app/building-your-application/caching#data-cache)：如果某个路由有`fetch`未缓存的请求，这将使该路由退出完整路由缓存。`fetch`每个传入请求都将获取特定请求的数据。其他`fetch`未选择退出缓存的请求仍将缓存在数据缓存中。这允许缓存数据和非缓存数据的混合
#### 6.3.2 router cache

Next.js 有一个内存中的客户端缓存，用于在用户会话期间存储按各个路由段拆分的 React 服务器组件有效负载。这称为路由器缓存

##### **6.3.2.1 路由器缓存的工作原理**

![](attachment)

当用户在路线之间导航时，Next.js 会缓存访问过的路线段并[预取](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching)用户可能导航到的路线（基于`<Link>`其视口中的组件

这会为用户带来更好的导航体验：

- 由于访问过的路线被缓存，因此可以立即实现向前/向后导航，并且由于预取和[部分渲染，](https://nextjs.org/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)可以快速导航到新路线。
- 导航之间无需重新加载整页，并且 React 状态和浏览器状态得以保留。
> **路由器缓存和完整路由缓存之间的区别**：

路由器缓存在用户会话期间将 React 服务器组件有效负载临时存储在浏览器中，而完整路由缓存会在多个用户请求期间将 React 服务器组件有效负载和 HTML 持久存储在服务器上。

完整路线缓存仅缓存静态渲染的路线，而路由器缓存则适用于静态和动态渲染的路线。

**期间**

缓存存储在浏览器的临时内存中。两个因素决定了路由器缓存的持续时间：

- **会话**：缓存在导航过程中持续存在。但是，它会在页面刷新时被清除。
- **自动失效期**：单个段的缓存会在特定时间后自动失效。持续时间取决于资源的[预取](https://nextjs.org/docs/app/api-reference/components/link#prefetch)方式：

- **默认预取**（`prefetch={null}`或未指定）：30 秒
- **完全预取**：（`prefetch={true}`或`router.prefetch`）：5 分钟
虽然页面刷新将清除**所有**缓存的段，但自动失效期仅影响从预取时起的单个段。

> **注意**：从 v14.2.0-canary.53 开始，对配置这些值提供了[实验性支持。](https://nextjs.org/docs/app/api-reference/next-config-js/staleTimes)

**无效**

有两种方法可以使路由器缓存无效：

- 在**服务器操作**中：

- 使用路径（`revalidatePath`）或使用缓存标签（`revalidateTag`）按需重新验证数据
- 使用`cookies.set`或`cookies.delete`使路由器缓存无效，以防止使用 cookie 的路由变得陈旧（例如身份验证）。
- 调用`router.refresh`将使路由器缓存无效，并向服务器发出当前路由的新请求。
**选择退出（opting out）**

无法退出路由器缓存。但是可以通过调用`router.refresh`、`revalidatePath`或 来使其无效`revalidateTag`（参见上文）。这将清除缓存并向服务器发出新请求，确保显示最新数据。

您还可以通过将`<Link>`组件的`prefetch` prop设置为`false`来选择退出**预取**。但是，这仍会暂时存储路线段 30 秒，以允许在嵌套段之间进行即时导航，例如标签栏或前后导航。访问过的路线仍将被缓存。

#### 6.3.3 缓存交互

配置不同的缓存机制时，了解它们如何相互作用非常重要：

##### 数据缓存（data cache）和全路由缓存（full router cache）

- 重新验证或退出数据缓存**将使**完整路线缓存无效，因为渲染输出取决于数据。
- 使完整路由缓存失效或选择退出**不会**影响数据缓存。您可以动态渲染包含缓存数据和未缓存数据的路由。当您的大多数页面都使用缓存数据，但有一些组件依赖于需要在请求时获取的数据时，这种方法非常有用。您可以动态渲染，而不必担心重新获取所有数据对性能的影响。
##### 数据缓存（data cache）和客户端路由器缓存（router cache）

- [重新验证路由处理程序](https://nextjs.org/docs/app/building-your-application/routing/route-handlers) 中的数据缓存**不会**立即使路由器缓存失效，因为路由处理程序不与特定路由绑定。这意味着路由器缓存将继续提供先前的有效负载，直到硬刷新或自动失效期已过。
- 要立即使数据缓存和路由器缓存无效，您可以在[服务器操作](https://nextjs.org/docs/app/building-your-application/data-fetching/server-actions-and-mutations)中使用`revalidatePath`或`revalidateTag`

## 9. Redux

### 9.1 redux 初始化及更新数据

1. - 非 NextJS 框架，如何实现服务端创建 store 并传递给客户端进行初始化
2. - NextJS 14 结合 Redux Toolkit 官方推荐（App router 集成方式）
> 注意：next-redux-wrapper 插件只支持 page router， web 项目中不选用

[https://toolkit.redux.js.cn/usage/nextjs](https://toolkit.redux.js.cn/usage/nextjs)

```
// app/ProductName.tsx
// example
'use client'
import { useRef } from 'react'
import { useAppSelector, useAppDispatch, useAppStore } from '../lib/hooks'
import {
  initializeProduct,
  setProductName,
  Product,
} from '../lib/features/product/productSlice'

export default function ProductName({ product }: { product: Product }) {
  // Initialize the store with the product information
  // 客户端组件会在服务端和客户端都执行初始化代码，不使用 useEffect 是确保水合正常
  // 可以实现服务端根据请求路由，获取相关数据并更新到 store 状态中，客户端会自动同步相关 store
  // 与 DDD 架构设计兼容
  // API endpoints 如果在服务端调用，相关 listener 等操作所更新的 state 也会同步到客户端 store
  const store = useAppStore()
  const initialized = useRef(false)
  if (!initialized.current) {
    store.dispatch(initializeProduct(product))
    initialized.current = true
  }
  const name = useAppSelector((state) => state.product.name)
  const dispatch = useAppDispatch()

  return (
    <input
      value={name}
      onChange={(e) => dispatch(setProductName(e.target.value))}
    />
  )
}
```

> 若在 ssr 模式下初次进入某客户端组件页面，当前页面若有 dispatch 相关操作，客户端组件则也会在服务器进行相关操作，该组件会在客户端再次执行，相关 redux 操作则会结合服务端 redux store 状态实现初始化，实现服务端、客户端 store 状态同步
