## 1. 背景与前提

### 1.1 业务背景

- 计划将 POS 登录从**现有账号密码 + 本地选门店**迁移至**自研 UMS**
- 接入后，用户在 POS 侧以 SSO 完成身份认证；**权限与可访问范围**拟通过登录成功后的 **UMS Info（或等价）接口**获取（如用户权限、可访问门店/市场等）。
### 1.2 多市场部署（补充）

- **多市场部署**：各市场（如 US / SG / AU / …）通常对应**独立构建与独立部署**（`EcEnv.NEXT_PUBLIC_COUNTRY`、`NEXT_PUBLIC_API_HOST` 等按环境注入）。
- **URL 形态**：路由上仍使用 `/{locale}/...`（如 `/us/login`），与 `libs/config` 中 `locales`、`defaultLocale` 一致。
- **重要行为**：`app/[locale]/layout.tsx` 在校验 `locale` 合法后，若 `locale !== EcEnv.NEXT_PUBLIC_COUNTRY.toLowerCase()`，会 **redirect 到当前构建对应市场路径**。即：**单次部署实例只服务一个市场**；用户若从根路径选错 region，会被纠正到该部署所属市场。
- **推论**：「市场」在**单实例**层面主要由**部署/域名/环境变量**决定，而非纯前端下拉框；多市场并行是通过**多套部署**实现的，而不是单进程内切换 API Host。

## 2. 现状方案（接入 UMS 前）

### 2.1 入口与 Region（Select Region）

| 环节 | 行为 |
| 根路径 `/` | 展示「Select Region」表单，提交后：已登录 → `/{region}/discover`，未登录 → `/{region}/login` |
| 默认值 | `EcEnv.NEXT_PUBLIC_COUNTRY.toLowerCase()` |
| 与部署关系 | 进入 `[locale]` 后 layout **强制** URL locale 与构建市场一致，选错仅产生重定向 |


**实质**：Region 选择主要影响**首次进入的 URL 段**；在单市场部署实例上，**最终有效市场 = 该实例构建变量**，与 SSO 前「登录页能否区分市场」的问题在多实例模型下**部分等价**——用户应落在**正确市场的域名/部署**上，而不是在同一部署上靠登录页区分市场。

### 2.2 登录页与 Branch（门店）

| 环节 | 行为 |
| UI | 「Branch」对应表单字段 `retailId`，选项来自 `GET retails`（当前部署 API） |
| 认证 | `adminLogin(username, password)` 换取 token，**请求体不包含 retailId** |
| 登录后 | `salesRepLogin`：`adminLogin` →（SG/AU 且 `enableBranch`）`getRetailById` → `getCurrentAdmin` → **持久化 **`retailId`、`posSalesId` |
| 后续 API | `sharedPrepareHeaders` 为 POS 请求附加 `X-Retail-Store-ID`（来自持久化 `retailId`）、`X-Sales-ID` 等 |


**区域差异**：`enableBranch` 仅在 **SG、AU** 为 true，登录流程会校验 `getRetailById`；**US** 等默认表单 `retailId` 可为固定值（如 2），且不经过 `getRetailById`。

### 2.3 防护与重定向

- **middleware**：未登录访问非白名单路径 → 重定向到 `/{defaultLocale}/login`，并带 `callbackUrl`。
- **白名单**：含 `/`、`/{locale}/login` 等。

## 3. 目标方案（接入 UMS ）

### 3.1 预期流程（逻辑）

1. - 用户打开**对应市场的 POS 部署**（域名/环境已决定市场）。
2. - 跳转 UMS 完成认证。
3. - 回调 POS，换取或写入 **UMS/业务侧 token**（具体与后端契约为准）。
4. - 调用 **Info 接口**，获取权限、可访问门店列表、角色等。
5. - **Region 初始化**：在多实例部署下，可简化为「当前实例即市场」，或与 Info 中的市场/租户字段做**一致性校验**（防止错域登录）。
6. - **Branch 初始化**：用户从 **Info 允许的门店集合** 中选择，或采用默认规则（唯一门店自动选中、上次门店等），写入 `retailId`（及必要的 `posSalesId` 等），再进入 discover/checkout 等。
### 3.2 与「登录页无法区分市场」的关系

- **多市场多部署**：用户应在**正确市场的 URL** 发起 SSO；回调也应回到该市场前端；Info 返回的权限应与该市场后端一致。
- 若未来存在**单入口多租户**（一个前端多套 API），才必须在 Info + 路由层显式绑定「租户/市场 → API Host」，超出当前「每市场一部署」模型的范围，需单独架构设计。

## 4. 新旧方案对比

| 维度 | 旧方案（当前） | 新方案（UMS + SSO + Info） |
| **身份认证** | 邮箱 + 密码，`adminLogin` / OAuth token | 微软 SSO → UMS 签发/交换业务 token（实现以最终契约为准） |
| **市场（Region）** | 根路径选 region；实例级仍被 layout 约束到 `NEXT_PUBLIC_COUNTRY` | 以**部署/域名为准**；可选：用 Info 校验用户是否允许访问该市场 |
| **门店（Branch）** | 登录前在表单选 `retailId`，登录成功后写入持久化并带 `X-Retail-Store-ID` | **登录成功后再选/再确认**；选项应来自 **Info 授权门店 , GET retails（可选二次校验）** |
| **权限来源** | 后端依赖 token + 请求头（如门店）隐式约束 | **显式 Info**：权限列表、门店范围、角色等，前端可做门禁与 UI 裁剪（后端仍需强校验） |
| **SG/AU 门店校验** | `enableBranch` 时登录流内 `getRetailById` | 在选定 `retailId` 后保留同等校验或后端统一校验 |
| **未选门店即进业务** | 当前与登录表单强绑定，较少出现「已登录无 retailId」 | 风险上升：需 **路由/中间件门禁**，未完成门店上下文前禁止业务页 |
| **购物车/会话** | `beforeLogin` 清理 `xPosCartToken` 等 | SSO 登录成功后、切换门店时，需沿用或扩展清理策略，避免串单 |

## 5. 风险与缓解

| 风险 | 说明 | 缓解建议 |
| **错市场部署** | 用户在某市场 URL 完成 SSO，但账号无权访问该市场 | Info 返回市场/租户声明；前端跳转提示；**后端拒绝**错误市场 token |
| **未设置 **`X-Retail-Store-ID` | 业务 API 上下文不明 | 强制「选门店」步骤；middleware 或 layout 检查持久化 `retailId` |
| **门店越权** | 用户手工改 localStorage | Info 白名单；**后端按 token + 门店校验** |
| **US 等无 **`getRetailById` 路径 | 历史上依赖默认值 | UMS 后统一「允许列表 + 可选校验」策略，避免无效 id |
| **callbackUrl 与安全** | SSO 回调与深链接 | 校验 `callbackUrl` 范围，防开放重定向 |

## 6. 待澄清项（对接 UMS/后端时建议闭环）

1. - Token 形态：与现有 `accessToken` / `refreshToken` / cookie 是否兼容，还是双轨迁移期。
2. - Info 接口：字段定义（权限模型、门店 ID 与现有 `retails` id 是否一致、多市场字段名）。
3. - `X-Sales-ID`（`posSalesId`）与 UMS 用户主键的映射关系。
4. - 刷新令牌、登出是否与 UMS 会话联动（单点登出 SLO 需求）。
5. - 多市场下 SSO 的 **redirect_uri** 是否每市场独立注册。

## 7. 结论（接入前）

- **Region**：在多市场**多部署**前提下，市场主要由**实例配置**决定；旧版根路径 Select Region 与 layout 行为并存，**与「SSO 后无法在市场登录页选市场」并不矛盾**，前提是用户从**正确市场 URL** 进入。
- **Branch**：本质是 **POS 会话级门店上下文**（`X-Retail-Store-ID`），旧方案已在 **token 之后** 写入；新方案将选择后置到 Info 之后**可行**，但必须增加 **门禁与授权门店列表**，避免无门店上下文进入业务。
- **新旧核心差异**在 **认证渠道与权限显式化**；门店绑定时机可从「同屏密码登录前选择」调整为「SSO 成功后选择」，需补齐 **路由与状态** 设计。
