# 背景

## **多区域电商网站概述**

**Castlery**服务Singapore, United States, Australia市场，以及即将开启的Canada, United Kingdom等区域的市场。

在多市场运营中，时区差异会带来一系列技术和运营挑战，尤其是在跨国电商平台中。时区差异可能影响订单处理、促销活动时间、库存更新等多个方面。若时区管理不当，可能导致客户体验受损，例如促销信息提前或延迟显示、活动时间错乱等问题，从而影响用户满意度。

因此，数据的一致性与同步至关重要，而时区管理也是确保这些目标实现的关键环节。

## **时区概念阐述**

时区是指地球上根据经度划分的标准时间区域。由于地球自转和不同地区日照时间的差异，全球被划分为多个时区，每个时区通常代表一个小时（1h）的时间差。时区的划分旨在解决全球范围内不同地区时间的不一致性，以便协调国际交流和活动。

| **专业术语** | **说明** |
| **标准时区(GMT UTC+0)** | - 世界被划分为24个标准时区，每个时区相差1小时。时区的起始点是格林威治标准时间（GMT），即零时区（UTC+0）。 |
| **时区偏移** | - 每个时区通常与格林威治标准时间（GMT）或协调世界时（UTC）有固定的时间偏移量，例如UTC+8（如中国、新加坡）、UTC-9:30（Pacific/Marquesas）。
> 偏移量：时区偏移量并非都是1h的整倍数，也有时区是存在45分钟或者30分钟的偏移。 |
| **夏令时（DST）** | - 夏令时是为了充分利用太阳光照，某些地区在夏季将时钟拨快1小时，延长晚上的光照时间，减少夜间黑暗时间。这通常是政府为提升能源效率和改善生活质量而采取的措施（*约定俗成，根据当地政策调整*）。这一调整会影响到时间计算。
> 什么是夏令时：

例如，A地区规定早上9:00 AM 上班，晚上6:00 PM 下班，但6:00 PM 时太阳已落下，夜间光照时间短。通过夏令时调整，9:00 AM上班（实际时间8:00 AM），6点下班（实际时间5:00 PM），从而增加了夜间的光照时间，减少了能源消耗并促进了夜间经济。 |

## 时间管理的核心原则

1. - **统一标准时间**：网站服务器和数据库操作需采用 UTC 时间。
2. - **本地化展示**：在用户界面上，时间应根据用户所在时区或地区偏好进行本地化显示。
3. - **时间戳存储**：所有存储的时间戳采用 UTC 格式，避免时区转换引发的问题。
#### 1. 统一使用UTC存储

> - 使用UTC时间戳（毫秒ms）存储时间   1737537365104
- 使用ISO 8601格式存储时间    2025-01-22T09:17:10.322Z

```
// 后端使用UTC时间戳存储时间 （GO）
utcTimestamp := time.Now().UTC().UnixMilli(); //1737537365104
```

1. - **UI渲染：做时间本地化显示**
```
// 前端本地化显示时间
DateTime.fromMillis(1737537365104).setZone("Asia/Shanghai").toLocal(); //2025-01-23T23:36:05.104+08:00
```

# 设计思路

在多区域电商网站架构中，主要分为:

- **Business Side**（Business Management Systems: Knight, etc.）
- **Client Side**（Website, POS, etc.）
- **Server Side**
- **Third Party Systems  **(DY,  Storyblok,  etc.)
在多区域电商网站架构中，**Business Side、Client Side 和 Server Side** 需要协同工作来处理时间相关的规则。以下是设计这种协同时间处理的一些基本思路和推荐的优秀案例。

**1. 处理时间规则的设计思路（内部系统）**

| #### **Business Side（Business Management Systems: Knight, etc.）** |
| **时间存储** | UTC时间戳

`1737537365104` | - 在B端管理系统中，所有时间（如库存更新、订单处理、促销活动开始/结束时间等）统一使用UTC时间进行时间记录和存储。 |
| **本地化显示** | 时间本地化后显示

Asia/Shanghai: `2025/1/22 17:16:05` | - B端管理系统可以根据不同市场需求，提供本地化的时间显示和设置，例如可以按用户所在区域的时区自动调整时间 |
| *Business Side的本地化支持，需要结合管理人员的操作习惯设计。* |


| #### **Client Side（Website, POS, etc. ）** |
| - **动态时区转换**：C端根据用户的时区动态展示本地时间。例如，用户访问网站时，前端会通过浏览器自动获取本地时区，并将后台传递的UTC时间转换为本地时间，显示给用户。
- **跨时区的活动显示**：针对跨时区的活动（如促销时间、订单状态），前端需将展示的时间自动调整为用户本地时间，确保用户获得准确的时间信息。
- **时间的实时更新**：前端需处理实时时区变化（如夏令时变更），并在页面中及时更新展示的时间信息 |


| #### **Server Side** |
| - **统一时区处理**：服务器端应该处理所有与时间相关的操作（如促销开始/结束、库存变动、订单处理时间等），并使用标准化的时区（如UTC）进行存储和传输。
- **时区转换**：在存储时间信息时，使用标准时区（如UTC），在获取和展示数据时，根据用户的时区进行转换。服务器也需要处理来自不同地区的请求，并根据请求的时区进行时间调整。
- **定时任务与计划任务**：服务器端还需要处理跨时区的定时任务和活动调度，确保在不同地区的活动按预定时间正确启动和结束。 |

# 技术方案

## 前端时间处理

| ** 问题分类** | **描述** |
| - **时区检测** | - CSR，通过浏览器的 `Intl.DateTimeFormat().resolvedOptions().timeZone` 或 IP 定位，自动获取用户所在时区。
- SSR，可以借助IP检测时区，或者cookie传递时区信息。
- 处理未知时区的回退机制。 |
| - **时间显示** | - 利用 JavaScript 日期库（如 `date-fns`、`moment-timezone`）进行时区转换和格式化，显示用户所在地的时间。
- 处理如夏令时（DST）等边界情况。 |
| - **用户设置优先** | - 允许用户手动选择偏好的时区，并存储在用户配置文件中。
- 基于用户位置/设置自动调整页面时间内容。 |


## **后端时间处理**

| **问题分类** | **描述** |
| - **统一存储** | - 后端以统一的时区（如 UTC）存储时间。建议存储为timestamp。 |
| - **接口设计** | - 后端 API 提供 UTC 时间，前端根据用户时区进行转换。 |
| - **日志记录** | - 所有日志时间使用 UTC，便于跨区域排查问题。 |


## 数据库时间存储

| **问题分类** | **描述** |
| - **字段类型** | - 数据库中时间字段使用 `DATETIME` 或 `TIMESTAMP` 类型，默认存储为 UTC。 |
| - **索引优化** | - 对于时间相关查询，确保索引支持 UTC 时间。 |

## **前端时间处理方案技术选型**

### **时区库的选择与使用**

常用的时区库有：`moment-timezone`,` date-fns-tz`, `luxon`, `Intl(node)`

| **Technical Library** | **Advantages and disadvantages comparison** |
| `moment-timezone` | - 优点**：**前端最常用的时间处理库，功能非常强大，兼容新好，且时区功能完善。
- 缺点**：**其团队宣布`moment`相关库不再更新，进入维护模式，后续`node`升级可能会导致兼容问题。 |
| `Intl` | - 优点：JS的原生API，轻量级，性能好。
- 缺点：不能直接支持时间计算，时区支持有一定局限（仅支持和UTC的转换）。 |
| `date-fns-tz` | - 优点：强大的时区功能和丰富的日期操作。
- 缺点：`date-fns-tz`需要额外安装`date-fns`来支持，有一定的API理解成本，在npm使用趋势上来看，不够有优势。 |
| `luxon` | - 优点：由moment核心开发者开发，兼容最新的Web技术，功能强大切API灵活，且是基于现代 JavaScript API（例如 `Intl` 和 `Date`）构建的。
- 缺点：对于历史时区数据（过去的年份，比如以往年份的夏令时数据）支持不是很好，API丰富度不如moment-timzone。 |

#### **时区缓存**

在前端，我们可以将用户的时区信息缓存到 LocalStorage、SessionStorage 或 Cookie 等存储区域。

然而，由于在 Next.js 框架下的 SSR 和 ISR 模式中，服务器端需要获取时区信息，而无法直接访问前端的本地存储，因此更适合将时区信息存储在 Cookie 中。这种方式不仅可以在服务端渲染时使用，还能确保时区数据在客户端和服务端之间保持同步。

如果项目中的 SSR 页面对时区不敏感，建议将时区信息缓存到 LocalStorage 或 SessionStorage 等本地存储中，以避免因使用 Cookie 而导致其体积过大等问题。这种方式既能满足前端的使用需求，又能减少对网络传输和存储性能的影响。

如果页面对时区敏感并需要高精度，推荐结合客户端和服务器端的方法（例如优先读取 Cookie 或客户端传递的数据，再退回到基于 IP 的推断）。

| **Cookie** | **Local storage** | **Session storage** |
| 支持SSR和ISR等服务端渲染场景中获取 timezone信息
