## Payment：长事务状态机

一句话：Web Payment = 状态流，不是一次表单提交。

骨架：

`Ready → OrderPrepared → PaymentInitiated → UserActionPending → Capturing → Resolved`

要点：

- 第三方 SDK 成功 ≠ 订单成功；**后端状态是真相**，回跳只是线索。  
- `initiate → SDK / redirect / popup → capture`。  
- Server Actions：`initiatePaymentAction` / `capturePaymentAction`。  
- 返回动作类型：`SDK_CONFIRM | REDIRECT | SDK_POPUP | SUCCESS`。  
- POS 更短闭环；Web 跨第三方边界更难。

### 幂等与重试（现行）

- `idempotencyKey = params.idempotencyKey ?? traceId`（initiate/capture）。  
- 同一次尝试复用同一 key；新尝试必须新 key。  
- `captureWithRetry`：单次 race **5s**；最多 **3 次重试（共 4 次）**；backoff **1s→2s→4s（cap 8s）**，无 jitter。  
- **仅 timeout 重试**；明确成功/失败立即返回；全超时 → `processing`。  
- 后端靠 idempotencyKey 去重 → 同 params 重试安全。

### 支付 → 埋点短链

`createOrder → initiate → SDK/Redirect → captureWithRetry → webPaymentCapturedEvent → 完成页 purchasedSucceededEvent → 多渠道`

## Transaction Observability（交易观测）

门面：`trackTransactionStart/Success/Failure/Timeout`、`captureTransactionError`、`reportTransactionMessage`。

挂点：

| 位置 | step 例 |
| --- | --- |
| Client `use-payment-execution` | `create_order`、后续 initiate/capture |
| Server initiate/capture | `payment_initiate` / `payment_capture` |
| `captureWithRetry` | retry/timeout message（常 skipSentry） |

关键标签/字段：

- `txn_trace_id`（伞状业务 ID，关联 Sentry ↔ Loki）  
- `step` / `result` / `errorCategory` / provider / orderId / paymentId…  
- `business_rule_error` / `user_abort` → 默认不上 Sentry  

与站点观测分工：

- Sentry：根因栈、分类 issue  
- Loki：结构化交易日志按伞状 ID 查  
- Grafana：目标看漏斗成功率 / SLO（**OTel metrics 仍预留**）  

面试句：

> 交易告警设计走 Grafana SLI，而不是再给 payment 加一条 Sentry per-domain 规则；现在能稳定讲的是 Sentry+Loki 字段闭环，Grafana 是下一阶段。

更多标签与缺口 → [03](./03-可观测性体系.md)。

## 鉴权：Token 刷新与分层

链路：

`凭证存储 → prepareHeaders → 401 silent refresh（Mutex）→ Middleware 兜底 → 错误分层`

要点：

- Web：cookie 存 access/refresh；另有 cart/checkout 业务 token。  
- `prepareHeaders` 统一注入 channel/token，业务不手写。  
- 并发 401：`async-mutex` 单飞 `POST /oauth/token`（refresh_token）→ 重放；失败清 token。  
- POS：legacy refresh；UMS 路径 `forceRenew` bearer，**403 不按未登录清会话**。  
- Middleware：进页前看 exp，临期 refresh；Server Component **只读 cookie，不自己刷新**。  
- 错误分层：401/429/5xx/网络在基建；券失效/库存不足在业务；**200 业务错误不能只靠 HTTP 拦截器**。

### Session / Cookie / JWT / OAuth（常识）

- Session：服务端会话 + HttpOnly Cookie id。  
- JWT：自包含；注意过期/撤销/存储。  
- OAuth2：授权拿 access/refresh。  
- Cookie：`HttpOnly / Secure / SameSite` ↔ CSRF。

## Server Actions / Middleware（全栈水位）

### Server Actions 安全边界

- 校验身份与输入，不信任客户端。  
- 不暴露密钥；副作用配合 revalidate。  
- throw → `withServerActionInstrumentation`；优雅降级 return 对象 → catch 里结构化捕获。  
- 适合应用内 mutation；重逻辑可下沉服务。

### Middleware

- 登录门禁、locale/region、token 临期 refresh、重定向。  
- 保持轻量；**不做 Sentry capture**（AGENTS 硬规则）。

## 缓存 / REST vs GraphQL / DB / Webhook（补水位）

- 缓存：ISR Redis + 搜索 responsesCache（见 04）；缓存≠DB。  
- REST vs GraphQL：资源语义 vs 按需字段；看客户端与治理成本。  
- DB：索引、唯一约束支撑幂等 Upsert；事务/隔离能点名即可。  
- Webhook：验签 → 快回 2xx → 异步 → id 去重 → 乱序用版本 → TTL/对账。  
  你的迁移点：支付回调恢复、CAPI 回传、CMS revalidate。
