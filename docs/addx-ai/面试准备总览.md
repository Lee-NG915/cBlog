## 4. Shopify 必修知识

### 4.1 先建立 Shopify 全景图

Shopify 开发至少分成三条路线：

| 路线 | 核心技术 | 适合场景 | 主要代价 |
| --- | --- | --- | --- |
| Theme | Liquid、JSON Template、Sections、Blocks、Theme App Extension | 快速上线、运营自主搭页面、依赖 Shopify 托管能力 | 自由度和复杂交互能力有限 |
| Headless | Storefront API、React、Hydrogen 或自建 Next.js | 高度定制、复杂体验、多系统组合 | 缓存、SEO、部署和预览都由团队承担 |
| Shopify App | Node / React Router、Admin API、OAuth、Webhook、Extensions | 扩展商家后台、订单流程、营销和运营能力 | 需要独立后端、数据存储和运维 |

面试最常见的问题不是“哪个最好”，而是“你会怎么选”。

建议答案：

> 如果业务主要是品牌展示、商品详情和营销落地，我优先用 Online Store 2.0 Theme，利用 JSON Template、Section 和 Block 让运营可配置。如果体验高度定制、需要组合多个后端或复杂个性化，再考虑 Headless。Headless 不是默认更先进，它会把缓存、SEO、预览、发布和稳定性责任重新交给研发。

### 4.2 Theme / Liquid

至少掌握：

- Liquid 的 output、tag、filter、object
- `layout`、`template`、`section`、`block`、`snippet`、`asset`
- JSON Template 与 Section Group
- Section schema、settings、presets
- App Block 和 Theme App Extension
- Metafield / Metaobject 如何成为动态数据源
- Theme Editor 中运营配置与代码默认值的关系
- Liquid 服务端渲染与 JavaScript 增强的边界

与你的经验连接：

- Section / Block 很像 CMS 可配置组件，不完全等同 React Component
- Fortress 组件 API 设计可迁移到 Theme Section schema 设计
- Storyblok / CMS 经验可用于解释“运营可配置但仍需 Schema 约束”

### 4.3 核心电商数据模型

必须能说清：

- Product：产品主体
- Product Variant：SKU、价格、库存、选项组合
- Collection：商品集合和导航组织
- Cart：用户购买意图，可修改但不是订单
- Checkout：Shopify 承载的结算过程
- Order：已创建的交易记录
- Customer：客户身份与地址
- Discount：折扣码或自动折扣
- Inventory Item / Inventory Level / Location：库存对象、地点和数量
- Fulfillment：订单履约和物流状态
- Metafield：给现有资源增加类型化字段
- Metaobject：定义新的结构化实体

你应重点理解 Product 和 Variant 的区别，以及订单、支付状态、履约状态为什么不能合并成一个 status。

### 4.4 Storefront API 与 Admin API

Storefront API：

- 面向买家体验
- 查询商品、集合、本地化信息
- 创建和更新 Cart
- 可用于 Theme 之外的 Headless Storefront
- 公共 Token 可以在客户端使用，私有 Token 只能放服务端

Admin API：

- 面向商家和后台管理
- 管理商品、订单、客户、库存、折扣和 Fulfillment
- 新 Public App 应以 GraphQL Admin API 为主
- 需要权限 Scope 和安全保存 Access Token
- GraphQL 使用 calculated query cost，必须处理 throttle

高频追问：

> 为什么不能把 Admin Token 放在浏览器？

因为它具有商家数据和写操作权限，泄露后可能直接读取订单或修改商品。客户端只能使用明确允许公开的 Storefront Token，Admin API 必须由受信任服务端代理。

### 4.5 OAuth 与 Session

需要理解：

- App 安装是在给某个 Store 授权 Scope
- Embedded App 的用户请求使用 Session Token 验证身份
- 服务端再通过 Token Exchange 获取 Shopify API Access Token
- Standalone App 通常使用 Authorization Code Grant
- Token、Shop domain、Scope 和安装状态需要持久化
- `app/uninstalled` Webhook 后要标记店铺失效并清理相关任务

与你的 Token Refresh 经验连接：

- 两者都涉及短期身份凭证和服务端权限 Token
- 但 Shopify Session Token 不是普通站点的 Refresh Token
- 不要把 Joyboy 的 401 refresh 流程原样套到 Shopify OAuth

### 4.6 Webhook、幂等与最终一致性

这是最可能考你的全栈部分。

必须掌握：

1. 使用原始 Request Body 校验 `X-Shopify-Hmac-Sha256`
2. 快速返回 2xx，把耗时任务放入队列
3. 使用 `X-Shopify-Webhook-Id` 做重复投递去重
4. 处理乱序，比较 `updated_at` 或触发时间
5. 数据库写入使用唯一约束或 Upsert，不能只依赖内存 Set
6. 失败需要重试、指数退避和死信记录
7. 定时 reconciliation，不能假设 Webhook 永不丢失

面试回答：

> Webhook 是至少一次投递语义，我不会把“收到一次”当成前提。入口先用原始 body 校验 HMAC，再以 webhookId 做持久化去重，快速返回成功；真正处理进入队列，并用资源 ID 加版本时间做幂等 Upsert。因为 Shopify 不保证事件顺序，也不能保证我们永远在线，所以还需要定时 reconciliation，从 Admin API 补齐漏掉或处理失败的数据。

这与你在 Payment 里使用同一 `idempotencyKey` 重试、由后端去重的思路是相通的。

### 4.7 API 限流

需要记住原则，不需要死背所有数字：

- GraphQL Admin API 按查询成本限流，不只是按请求次数
- 响应 `extensions.cost.throttleStatus` 会告诉你剩余容量和恢复速度
- 大数据同步应使用 Bulk Operations
- 处理 `429 / THROTTLED`，结合退避、队列和缓存
- 查询只拿必要字段，合理分页，避免 N+1
- Storefront API 没有普通固定请求数限制，但仍有安全和 Checkout 级保护

### 4.8 Checkout 扩展

需要知道：

- 现代 Shopify Checkout 定制主要使用 Checkout UI Extensions
- 业务规则更适合 Shopify Functions
- 埋点使用 Web Pixel Extension
- 信息、配送和支付步骤上的 UI Extension 通常要求 Shopify Plus
- Extension 运行在受限沙箱，不能把它当普通 React 页面
- 旧的 `checkout.liquid` 不是新项目的默认方案

你的 Payment 经验最适合迁移到：

- 支付状态和订单状态分离
- 回跳参数不是最终真相
- 服务端确认与幂等
- 不在客户端存储敏感支付信息

### 4.9 Shopify Markets 与多语言

必须能连接到你的多市场经验：

- Markets 管理国家、语言、货币、价格和市场上下文
- Storefront API 使用 `@inContext(country:, language:)`
- URL 路径式 locale 对 SEO 更友好
- 翻译不等于区域化，价格、税、配送、支付和政策也不同
- 缓存 Key 必须包含 market / country / language / currency
- `hreflang`、canonical、sitemap 和结构化数据需要按市场处理

## 8. 高概率面试题与回答方向

### 8.1 你没有 Shopify 经验，为什么认为可以胜任？

> 我没有生产 Shopify 经验，这一点我会明确说明。但 Shopify 岗位的核心不只是 Liquid 语法，而是全球电商、组件模块化、性能、第三方集成和交易一致性。我的当前项目正好覆盖多市场、i18n、Checkout / Payment、Redis 缓存、Core Web Vitals 和设计系统。与此同时，我已经在补 Shopify 的 Theme、Storefront / Admin GraphQL、Webhook 和 OAuth。我的优势是平台语法需要学习，但复杂电商问题和工程治理不是从零开始。

### 8.2 你有全栈经验吗？

> 我更准确的定位是高级前端加 BFF 经验，还不是成熟后端。项目里我做过 Server Actions、API Route、Redis、Token refresh、支付策略、幂等重试和错误治理，所以理解前后端边界以及服务端可靠性问题。数据库建模、队列和独立 Node 服务不是我过去的主要职责，这是我需要补齐的部分。我会如实区分做过、参与过和正在学习的内容。

### 8.3 Shopify Theme 和 Headless 怎么选？

> 先看业务是否真的需要脱离 Shopify Theme。品牌官网和标准商品交易优先 Theme，因为运营可配置、托管能力成熟、交付成本低。只有在体验高度定制、多后端聚合或个性化要求明显时才选 Headless。Headless 会带来更高自由度，但团队要自己承担缓存、SEO、部署、预览和稳定性。

### 8.4 如何设计一个可复用 Shopify Section？

> 我会先定义业务边界和运营可配置项，再设计 schema。稳定结构放在 Section，重复内容用 Block；颜色、间距、排版尽量引用统一 Token；settings 保持有限且有默认值，避免把每个 CSS 属性都暴露给运营。最后验证空数据、不同 Block 数量、移动端、可访问性和性能。

### 8.5 Webhook 重复或乱序怎么办？

> 校验 HMAC 后快速返回，使用 `X-Shopify-Webhook-Id` 做持久化去重，处理逻辑进入队列。更新资源时比较版本或更新时间并做幂等 Upsert。因为 Webhook 可能丢失，还要用定时 reconciliation 补数据。

### 8.6 如何优化 Shopify 商店性能？

> 我会先看真实用户数据，再拆 LCP、INP、CLS。Theme 场景重点检查首屏 Section、Hero 图片、字体、第三方 App Script、同步 JavaScript 和未使用 CSS；Headless 还要检查 Storefront API 瀑布、缓存 Key 和服务端渲染。优化后用相同页面类型和流量分组对比，而不是只展示一次 Lighthouse 分数。

### 8.7 如何设计全球化缓存 Key？

> Key 至少要包含资源 ID、market、country、language、currency 和影响结果的筛选条件。用户身份、会员价格或地址会影响结果时，不能错误共享公共缓存。失效上优先用商品或集合维度 Tag，再用 TTL 兜底。

### 8.8 如果 Shopify API 限流怎么办？

> GraphQL Admin API 要看 query cost 和 `throttleStatus`，减少字段、合理分页并缓存；大批量同步用 Bulk Operations。写操作通过队列控制并发，遇到 throttle 使用退避重试，同时保证 mutation 本身幂等。

### 8.9 如何解释你从业务交付走向平台架构？

> 我不是脱离业务单独做架构，而是在重复出现的交付问题中抽平台能力。比如 UI 漂移推动 Design Token 和 Chromatic，跨 Pod 缓存不一致推动 Redis Cache Handler，模块错误引用推动 Nx Tag，支付重试风险推动 idempotency 和统一动作模型。决策都来自真实业务约束，也有明确代价。

## 9. 简历高风险追问

### “负责组件库从 0 到 1”

准备数据：

- 第一批组件范围
- Token 层次
- Story 数量或覆盖方式
- Chromatic PR 流程
- 谁使用、如何迁移、失败案例

不要只讲技术名词，要说明你亲自负责了哪些决策。

### “核心模块技术负责人”

准备边界：

- Account / Promotion / Checkout / Payment 中哪些由你主导
- 哪些是参与方案或维护
- 一次冲突决策和一次事故处理
- 与后端、产品和 QA 的协作方式

### “ISR + Redis”

必须承认当前实现的真实限制：

- Redis 初次连接会阻塞启动且代码注释也指出没有超时
- Redis 不可用时降级到本地 LRU，跨 Pod 一致性会下降
- Prefix 包含环境和 buildId，可以隔离发布版本但会增加旧 Key 清理问题

讲出限制比只讲收益更可信。

### “搜索 200ms -> <5ms，减少 30% 请求”

准备口径：

- 200ms 是网络请求还是端到端交互时间
- <5ms 是缓存读取耗时还是页面完成渲染
- 30% 是哪段时间、哪些事件、哪些市场
- 缓存命中率和用户后退场景占比

如果没有完整监控报表，使用“在目标场景的测试和日志采样中”而不是“全站稳定降低”。

### “Joy UI -> Tailwind”

当前代码仍使用 Joy UI。推荐口径：

> 我主导的是迁移 ADR 和取舍评估，结论包含 SSR / RSC 兼容、包体、组件 API 和渐进迁移成本。它不是已经全量完成的 Tailwind 重写。

### “DDD 设计”

推荐口径：

> 我们借鉴 DDD 的业务域边界，但不是严格后端 DDD。前端内部仍使用 Redux Slice、Service 和 Component，更准确是 domain-oriented modular architecture。

### “AI 自动修复”

推荐口径：

> AI 自动完成问题聚类、上下文收集、候选修复和 PR 创建，生产发布仍经过测试、Review 和 CI，不允许模型直接上线。
