# 02 · 组件库与设计系统

一句话：Token 分层 + Storybook CDD + Chromatic 门禁，统一 Web/POS/CMS 视觉语言，并兼顾 RSC、包体积与多主题。

---

## Design Token 分层（global / alias / component）？多主题怎么切？

三层（同义：Primitive / Semantic / Component）：

1. **Global / Primitive**：纯值，无业务语义（`color-blue-500`、`spacing-8`）。
2. **Alias / Semantic**：用途语义（`text-primary`、`bg-surface`、`danger`）→ 换肤改映射。
3. **Component**：组件私有（`button-primary-bg`、`input-border-focus`）→ 组件可独立演进。

原则：

- 业务代码优先用语义层，少直接碰 primitive。
- 多品牌时可在 semantic 与 primitive 间加 **theme/brand mapping**。

### Token / Theme / 组件样式关系

- Token = 设计变量  
- Theme = 映射规则（浅/深、品牌）  
- 组件样式 = 消费 theme，不写死色值  

### 多主题切换实现

- CSS 变量 / Theme Provider 注入语义变量（运行时切换常见）。
- 重构期可两套 theme 配置，Provider 切换；组件只绑语义 token。
- 编译时主题：构建多份 CSS；运行时：换 class/data-theme 或 context。项目以 **运行时 Provider + 语义变量** 为主叙事。

### 为什么不只定义一堆 CSS variables？

- 无分层会变成「变量垃圾场」：换肤仍要改组件、命名无语义、难约束用法。
- 分层 = 治理：谁能引用谁、主题改哪里、组件 API 如何稳定。

## Joy UI → Tailwind 迁移 ADR：为什么迁？怎么权衡？

**口径：主导 ADR/评估，非全量重写完成；master 仍大量 Joy UI。**

为何评估迁：

- SSR/RSC 下部分 UI 库 client 边界重、包体与样式运行时成本。
- 设计系统希望更可控的 utility/token 落地方式。
- 长期维护：自定义主题与 tree-shaking 预期。

权衡三角：

| 维度 | 要点 |
| --- | --- |
| SSR/RSC | client 边界、hydration、第三方 server 兼容 |
| 包体积 | Joy 运行时 vs Tailwind 生成 CSS；实测 analyzer，不空谈 |
| 迁移成本 | 组件 API 兼容、渐进替换、双轨共存、回归（Chromatic） |

决策表达：

- 渐进兼容，不 big-bang。  
- 用 ADR 记「为何现在不迁/部分迁」同样是工程结论。

## Chromatic 视觉回归：接入与误报

接入流程要点：

1. Storybook 沉淀状态（default/loading/empty/error/disabled…）。  
2. PR CI 跑 Chromatic，出视觉 Diff。  
3. Review 接受/拒绝变更 → 视觉门禁，不只靠人眼扫页面。

误报处理：

- 动画/字体/时间戳/随机内容 → 固定 fixture、禁动画、稳定 viewport。  
- 非意图变更：修组件或回滚。  
- 意图变更：明确 approve 并更新基线。  
- 区分「真回归」与「环境抖动」；必要时隔离 flaky story。

## RSC 下组件库兼容

- `'use client'` 边界：交互、浏览器 API、多数第三方 UI 放 client；展示型可留 server。
- 库入口：区分 **server entry** 与 **client components**，避免 barrel 一把拖成 client。
- 第三方：确认是否依赖 `window`/`useEffect`；不兼容则 wrapper + client 边界。
- API 设计：少强制 Context 包整树；props 可组合，避免业务需求堆成巨型 props。
- Tree shaking：避免副作用 import；用 analyzer 验证；副作用写清 `sideEffects`。

### 如何避免组件 API 变成业务需求集合？

- 稳定语义 + 可组合原语；业务逻辑外置到业务层。  
- Story 定义契约状态，先于业务页面。  
- Breaking change 走版本与迁移说明。

---

## Storybook / CDD 一句话

- Storybook = 开发、联调、验收、回归中间层，不是「好看的目录」。  
- CDD：先组件状态与边界，再拼业务；适合多端/多团队；一次性活动页勿过度组件化。

---

## 回答骨架

1. 问题：多端 UI 漂移、交付重复、回归靠人。  
2. 方案：Token 治理 + CDD + Chromatic 门禁。  
3. 约束：RSC、包体、团队学习、渐进迁移。  
4. 验证：adoption、Diff 门禁、体积与性能。  
5. 坑：只堆组件数量、无版本/兼容策略。
