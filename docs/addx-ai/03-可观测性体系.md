# 03 · 可观测性体系（Sentry + Transaction + Grafana）

> 依据 Joyboy `master`（含 `docs/ai-specs/observability`、`transaction-observability`、核心代码）。  
> AI 修复闭环详见 [09-AI实践-Skill-Harness.md](./09-AI实践-Skill-Harness.md)。

一句话：`@castlery/observability` 统一「日志 + 结构化上报」；Sentry 管根因与 11 条互斥告警；交易漏斗目标走 Grafana（Loki 字段已落地，OTel metrics 仍预留）。

---

## 架构口述（先背）

```
业务代码
  ├─ setGlobalSentryContext / SentryContextProvider
  │     → tags: domain / page_type / priority
  ├─ captureStructuredError（站点通用）
  └─ trackTransaction* / captureTransactionError（交易）
         ↓
  beforeSend：脱敏 → classifyErrorBucket → error_bucket
         ↓
  ├─ Sentry：Issue + Ownership + 11 Alert Rules
  ├─ pino/stdout JSON → Loki（txn_trace_id / order_id…）
  ├─ [目标] OTel → Tempo + Prometheus → Grafana SLO
  └─ Datadog RUM/APM：现状仍在；交易方案规划替换后下线
```

分工：

| 平台 | 角色 | 现状 |
| --- | --- | --- |
| Sentry | 为什么失败、哪行、哪个 release | 已落地 |
| Loki | 按 `txn_trace_id` / order / payment 回溯日志 | 交易 helper 已写字段 |
| Grafana + Prom | 成功率、漏斗、SLO/burn rate | **规格为主**；依赖 OTel metrics |
| Tempo | 分布式 trace | **未集成 OTel** |
| Datadog | 现有 RUM/APM | 仍在；交易侧不再作为目标依赖 |

面试诚实句：

> 支付主链路的 Sentry + 结构化 Loki 字段已落地；Grafana Dashboard/SLO 协议写好了，OTel metrics 出口仍是预留，不能说「SLO 告警已全线上跑通」。

## captureStructuredError：标签体系

统一入口，禁止裸 `Sentry.captureException`（ESLint `observability/no-bare-sentry-capture`）。

| Tag | 谁写 | 要点 |
| --- | --- | --- |
| `domain` | 捕获时 / Provider | user/order/payment/cart/product/search/checkout/cms/promotion… |
| `page_type` | page context / Provider | pdp/plp/cart/checkout/account/home/cms/blog/other；**无 SEARCH**；`/search`=PLP+domain:search |
| `priority` | 由 domain 推断 | high/medium/low |
| `severity`→level | 同上 | CRITICAL→fatal，HIGH→error，MEDIUM→warning，LOW→log |
| `error_bucket` | **仅 beforeSend** | 业务不分桶 |

流程：normalize → enrich → log → PII filter → scope tags → Sentry。

### error_bucket 分桶

- Client 顺序：`hydration → third_party → network → api_5xx → api_timeout → app_error → js_fatal → unclassified`
- Server：`upstream_5xx → upstream_timeout → js_fatal → app_error → unclassified`
- `app_error`：显式 `captureStructuredError`（handled + generic）
- 关键三方（永不进 third_party）：DY / Stripe / PayPal / Algolia / Casa — 单一数据源 `error-bucket.ts`
- 低基数：bucket 是有限枚举；勿把 userId、全 URL、动态 message 当 tag

### 11 条互斥告警

- 分层：Sev-1 `page_type`(PDP/PLP/Cart) → Sev-2 domain(user/order) → Sev-2 bucket(API/Upstream) → Sev-3 fatal/error → Sev-4 warning/unclassified/network
- **每个 issue 只进一条规则**；禁止再堆 per-domain Sentry alert
- 口径：规范已是 **11** 条（勿再说死 10）

### Ownership

- Dashboard 规则：按 `domain` / path / `error_bucket` 自动 assignee
- Runbook：含义、影响、触发、排查、止血、owner、升级、验证

## Transaction Observability（交易专用）

与站点观测拆开讲：

| | 站点 | 交易 |
| --- | --- | --- |
| 入口 | `captureStructuredError` | `trackTransaction*` / `captureTransactionError` |
| 关键维度 | domain + page_type + error_bucket | step + result + errorCategory + `txn_trace_id` |
| 告警 | Sentry 11 规则 | 目标：Grafana SLI/SLO（非 per-domain Sentry） |
| 降噪 | bucket/噪音过滤 | `business_rule_error` / `user_abort` / `skipSentry` 默认不上 Sentry |

### API 门面

- `trackTransactionStart / Success / Failure / Timeout`
- `captureTransactionError` / `reportTransactionMessage` / `addTransactionBreadcrumb`
- 四出口：日志(Loki) + breadcrumb + Sentry；**metrics 预留**

### 关键字段

- 伞状业务 ID：`txnTraceId` → tag/日志 **`txn_trace_id`**（勿与 OTel `trace_id` 混淆）
- step 例：`create_order`、`payment_initiate`、`payment_sdk_*`、`payment_capture`、`payment_callback_receive`…
- result：`started|success|failure|timeout|canceled|processing|retrying`
- errorCategory：`system_error|provider_error|network_error|validation_error|business_rule_error|timeout_error|user_abort|unknown_error`
- 交易 CRITICAL 在 Sentry 常降为 error，避免 fatal 刷屏

### 与支付代码挂钩

- Server Action：`initiatePaymentAction` / `capturePaymentAction` 打 initiate/capture
- Client：`use-payment-execution` 打 `create_order` 等
- `captureWithRetry` 超时用 `reportTransactionMessage`（常 `skipSentry`）

### Grafana 怎么讲

- Dashboard/SLO 规格：成功率、漏斗、burn rate、release 窗口加严
- 关联：告警 message 挂 Sentry issue search（按 step 等）
- **高基数字段禁作 metric label**：`txn_trace_id`、`order_id`、`payment_id`…
- 现状：协议与 Loki 字段先行；OTel→Prom 未完成 → 面试区分「设计」与「已上线」

## AGENTS 硬规则（可背）

- Middleware/Edge：**禁止** Sentry capture（logger 可）→ 已知 gap
- 每个 `page.tsx`：`setGlobalSentryContext`；layout 用 Provider
- Server Action：throw → `withServerActionInstrumentation`；降级 return → `captureStructuredError`
- Checkout/Payment **page 级** instrumentation 仍 blocked；**交易 API 半边已允许**
- Sentry auto-resolve：`fixes ISSUE-ID` 只写 PR title/body，不写 commit

---

## 已知限制（主动说更可信）

1. OTel metrics/Tempo 未落地 → Grafana SLO 非全量运行事实  
2. `x-txn-trace-id` 请求头透传仍弱/未齐  
3. Datadog 仍在，交易目标替换后下线  
4. Edge 错误无 bucket、不进告警  
5. 部分 step（如 `payment_result_render`）枚举有、代码未 fire  

---

## 高概率追问 → 要点

- 为何统一入口？→ 字段一致、可分桶、可路由、可度量  
- bucket 为何在 beforeSend？→ 所有入口一条管线，业务只供 context  
- 11 条如何互斥？→ 高层 page_type/domain 优先，下层用否定条件排除  
- 交易 vs 站点？→ 漏斗 step + 伞状 ID；业务规则失败不上 Sentry  
- Grafana 跑起来了吗？→ 字段与 Sentry 链路已有；metrics/SLO 仍规格阶段  
- AI 修 bug？→ 见 09；只到 PR，人审 merge  

---

## 回答骨架

1. 噪音大、归属不清 → 结构化采集  
2. beforeSend 分桶 + 11 互斥告警 + ownership  
3. 交易另建 step/`txn_trace_id` 漏斗，Grafana 看影响、Sentry 看根因  
4. AI harness 有界修复，出口 PR  
5. 承认 OTel/Edge/Datadog 过渡态
