## ISR + Redis 共享 Full Route Cache

### 为什么要共享？

- Next 默认缓存偏单实例内存。  
- 多 Pod：A 已缓存，打到 B/C 仍回源 → 命中率低、重复生成。  
- 生产：`cacheHandler` + `cacheMaxMemorySize: 0`（关掉默认内存双写）。

### 现行实现要点（`apps/web/cache-handler.js`）

- `@neshca/cache-handler`；仅 `NODE_ENV===production` 启用。  
- Redis `keyPrefix`：`${APPLICATION_ENV}-${buildId}:` → 环境 + 发版隔离。  
- **`timeoutMs: 1000`** → 快速失败。  
- Redis 不可用 → **进程内 LRU 降级**（跨 Pod 不一致可接受）。  
- `estimateExpireAge` 固定约 **3600s（1h）**（以代码为准）。  
- 已知限制：`client.connect()` **无超时，可能阻塞启动**——主动承认。

### Key 业务维度

- path + **market/region/locale/currency** + 影响结果的筛选。  
- 用户态 / 会员价 **不进** 公共页缓存。

### 失效

- ISR TTL + `revalidateTag` / `revalidatePath` / CMS Webhook。  
- 事件 + TTL 双保险；重复乱序靠幂等与版本；丢失靠 TTL/对账。

### 穿透 / 击穿 / 雪崩（结合讲）

- 穿透：空结果短 TTL/占位。  
- 击穿：热点单飞（锁/Promise 复用）。  
- 雪崩：TTL 抖动 + 降级；避免同时大面积过期。

### 观测

- TTFB p50/p75/p95、命中率、回源量、错误率。  
- 降级不是无影响：可用性优先，性能/一致性短期变差。

## 搜索双层缓存（现行）

权威代码：`libs/modules/search/.../search-client.ts`（**不是**旧文档里的 `search-view-client.tsx`）。

机制：Algolia transporter **`responsesCache`** + hydrate 契约，消掉 InstantSearch 强制二次 search / routing 双发。

| 项 | 现行值 |
| --- | --- |
| TTL | **2 分钟**（`RESPONSES_CACHE_TTL_MS`） |
| 容量 | **100** entries（FIFO） |
| 请求超时 | 10s |
| in-flight | 缓存 Promise → 同 key 去重 |
| 失败 | 驱逐，不长期缓存 artificial/失败结果 |

两层语义仍成立：

1. **结果缓存**：重复同条件少打请求。  
2. **进行中去重**：并发同 key 复用 Promise。

与 React Query/SWR：同类「stale + in-flight」思想；实现挂在 search client。

### 口径怎么说（重要）

- 旧文档「5min/50、200ms→&lt;5ms、−30%」**勿当现行 SLA**。  
- 面试：说明问题（冗余 search / 返回闪烁）、方案（responsesCache 2min/100）、验证看请求数与交互耗时。  
- 若简历仍写旧数字：改口为「目标场景下命中缓存后交互显著下降、重复请求减少」，或更新简历对齐代码。

## Core Web Vitals

方法论：RUM 发现 → 归因 → 改 → 同页面类型/流量分组对比。

### LCP

- 定位：Hero/字体/TTFB/资源优先级。  
- 优化：ISR/Redis 降 TTFB；图片尺寸/格式/priority；减阻塞脚本。

### INP

- 三段：input delay / processing / presentation。  
- 项目有 **INP attribution 上报**：tags `inpPhase` / `inpBottleneck` / `page_type`；fingerprint 含 phase/bottleneck。  
- 常见因：长任务、大 chunk 顶层求值、重渲染、GTM/第三方、`useBreakpoints` 等（历史 case 见 INP 文档）。  
- 优化：拆任务、减 hydrate、dynamic import、延后非关键 SDK。

### CLS

- 无尺寸图/广告/字体 swap/动态插入。  
- 预留空间、字体策略、骨架。

### Bundle

- `pnpm web:analyze` + `@next/bundle-analyzer`。  
- Tree shaking：ESM + 无副作用；barrel/CJS 易失效。  
- `optimizePackageImports`（含仍在的 `@mui/joy`）。  
- dynamic import 延后重模块；字体子集化。  
- 区分：本地编译慢 ≠ 线上首屏大。  
- Joy UI：**仍在依赖**；勿说已彻底去掉。

## 高概率追问 → 要点

- 自带缓存不够？→ 单机，多 Pod 重复生成。  
- Redis 挂了？→ 1s 超时 + LRU；承认跨 Pod 不一致与 connect 无超时。  
- 搜索 key？→ query+筛选+市场等；失败不缓存。  
- 旧文档数字？→ 以 `search-client.ts` 为准。  
- INP 怎么定位？→ attribution tags + 长任务/第三方/重渲染。
