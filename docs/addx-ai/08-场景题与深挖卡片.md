## 卡片 A：架构重构（首选）

**一句话**：legacy 按页面堆叠 → 域 + 三层 + Nx tag 可执行边界；Flag 控灰度。

**可被追问的事实**

- Web/POS 共用领域能力，展示层不同。  
- 域：Product、Cart、Checkout、Payment、Promotion、Order、Search…  
- tag + lint/CI，不是只靠文档。  
- 灰度与新旧共存。

**高概率追问 → 要点**

- 为何按域不按 hooks/api？→ 改动按业务归属。  
- 三层职责？→ 状态 / 编排 / UI。  
- 跨域下单？→ 上层编排，domain 不互调。  
- tag 防啥不防啥？→ 防依赖方向，不防烂抽象。  
- 收益怎么证？→ affected 范围、违规数、交付/回归，承认学习成本。  
- 重做调啥？→ 域粒度、composite 边界、payment 额外层。

**别踩坑**：只背名词无边界冲突；声称零耦合；只讲结构不讲迁移回滚。

## 卡片 B：ISR + Redis / 搜索缓存（首选）

**一句话**：多 Pod Redis 共享 Full Route Cache（1s 超时→LRU）；搜索用 `responsesCache` 2min/100 去重。

**可被追问的事实**

- `cacheHandler` + `cacheMaxMemorySize:0`；prefix=`env-buildId`。  
- `timeoutMs:1000`；connect 无超时是已知限制。  
- 搜索权威代码：`search-client.ts`（旧 search-result-cache 文档过时）。  
- INP attribution tags：`inpPhase` / `inpBottleneck`。

**高概率追问 → 要点**

- 自带缓存不够？→ 单机，多 Pod 重复生成。  
- key？→ env/buildId/market/locale/currency；慎用户态。  
- Redis 挂了？→ 快速失败 + LRU；承认跨 Pod 不一致。  
- 搜索两层？→ 结果 TTL + in-flight Promise 去重。  
- 旧简历数字？→ 勿背 5min/30%；讲现行参数与问题动机。  

**别踩坑**：Redis 当 DB；串市场；把过时文档当 SLA。

## 卡片 C：可观测性 + Transaction + AI Harness（差异化）

**一句话**：结构化错误 + 11 互斥告警；交易用 step/`txn_trace_id`；AI 走 Harness 有界修复到 PR。

**可被追问的事实**

- `captureStructuredError` + beforeSend `error_bucket`。  
- 交易：`trackTransaction*`；Loki 字段已有；Grafana/OTel metrics **预留**。  
- Datadog 仍在，交易目标替换后下线。  
- alert-harness 默认 analyze-only；biz/baseline 周 routine。  
- 11 条互斥告警（现行规范）。

**高概率追问 → 要点**

- bucket 为何 beforeSend？→ 全入口一条管线。  
- 交易 vs 站点？→ step 漏斗 + 伞状 ID；业务规则失败不上 Sentry。  
- Grafana 跑通了吗？→ 协议与 Loki/Sentry 先行；metrics 未全落地。  
- AI 如何防误修？→ Spec/ESLint、max 3、禁区、人审、禁 auto-merge。  
- Harness 8 组件？→ Spec→Static→…→Evolution，Observability 是样板。  

**别踩坑**：吹全自动上线；吹 Grafana SLO 已全跑；混淆 Datadog「现状」与「目标」。

## 通用场景题（非 Shopify）

### 没有某平台经验为何能胜任？

- 承认缺口。  
- 迁移的是：全球电商、模块化、性能、交易一致性、可观测。  
- 语法可学，复杂治理非从零。

### 你有全栈经验吗？

- 高级前端 + BFF；列举 Server Actions/Redis/鉴权/支付幂等。  
- 数据库/独立服务非主责；区分做过/参与/在学。

### 如何设计全球化缓存 Key？

- 资源 + market/country/language/currency + 筛选。  
- 用户态不进公共缓存。  
- Tag 失效 + TTL。

### 从业务交付到平台架构？

- 问题驱动：UI 漂移→Token/Chromatic；多 Pod→Redis；乱依赖→Nx tag；支付重试→幂等。  
- 有代价：学习成本、一致性取舍。

### 灰度发布怎么做？

- Flag 三层；监控核心指标；瞬关回滚；再删旧代码。

### 线上支付失败怎么排查？

- Datadog 看成功率/市场/版本 → Sentry 看栈与 bucket → traceId/idempotencyKey 对单笔 → 区分第三方成功与订单状态 → 降级/Flag。

### 性能优化从哪开始？

- RUM CWV → 归因 LCP/INP/CLS 或接口/Bundle → 改 → 同维度对比验证。

## 比心直播经历（各 1 分钟版）

按你真实经历填细节；无则改成「活动大促/电商峰值」同类叙事，勿编造。

### 高并发活动前端保障

- 问题：峰值流量、接口打满、首屏与脚本争主线程。  
- 做：静态/ISR 扛展示；限流与降级开关；非关键 SDK 延后；核心路径监控与回滚 Flag。  
- 结果：可用性优先，接受非核心功能降级。

### WebSocket

- 场景：实时状态（库存/直播互动/消息）。  
- 点：重连退避、心跳、断线队列、与 HTTP 降级、鉴权与房间隔离。  
- 坑：扇出压力、背压、乱序。

### Canvas

- 场景：可视化/动效/导出。  
- 点：分层绘制、离屏、节流 rAF、内存与清晰度（DPI）、销毁监听防泄漏。  
- 与 DOM：高频变化用 Canvas，复杂可访问性仍要 DOM 兜底。
