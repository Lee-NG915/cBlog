**理解版**
先说结论，`Joyboy` 这套 `Monorepo + DDD`，更准确讲是“`Nx Monorepo + 按业务域拆分 + 三层分层 + tag 约束依赖`”，不是严格教科书式 DDD，但它已经把业务边界和技术边界都落到工程里了。

**1. 原 legacy 问题是什么**
从仓库文档看，重构前最核心的问题有这几类：

- 代码按页面和技术类型自然生长，业务相关代码容易散在很多地方，改一个 `checkout` 或 `payment` 需求要来回找。
- 页面、接口调用、状态管理、副作用、埋点容易混在一起，模块边界越来越模糊。
- Web、POS、多市场有大量共享能力，但共享逻辑和定制逻辑没有被很好隔离，容易重复实现。
- 复杂系统长期演进后，如果只靠约定不靠约束，依赖关系会越来越乱，最后很难治理。

这部分在 [docs/architecture-review.md](/Users/lidonghai/castlery/joyboy/docs/architecture-review.md:19) 里能看到整体描述，里面明确提到 monorepo、模块边界、分层和依赖复杂度这些问题。

**2. 为什么按业务域拆**
因为传统 `components/pages/services` 是按技术类型收目录，代码“长得像什么”就放哪里；但复杂业务更关心“这是谁的业务”。

所以项目先按业务域横向拆：

- `product`
- `cart`
- `checkout`
- `payment`
- `order`
- `promotion`
- `user`
- `cms`

这样做的好处是：

- 同一业务的代码尽量收敛在一起
- 业务边界更清楚
- 团队协作更自然
- 后续拆 ownership、做治理、做迭代都更容易

仓库里这层结构很清楚，在 [docs/architecture-review.md](/Users/lidonghai/castlery/joyboy/docs/architecture-review.md:33) 和 `libs/modules/*` 目录里都能看到。

**3. 三层分层具体是哪三层**
每个业务域内部再做纵向分层，主流是三层：

- `domain`
  放领域模型、types、entities、Redux slices、业务状态
- `services`
  放接口调用、业务编排、跨对象协作逻辑
- `components`
  放 React 组件、页面交互、UI 组合

文档里直接写了这一点，在 [docs/architecture-review.md](/Users/lidonghai/castlery/joyboy/docs/architecture-review.md:68)。

如果拿 `product` 举例，三个包分别是：

- [libs/modules/product/domain/project.json](/Users/lidonghai/castlery/joyboy/libs/modules/product/domain/project.json:1)
- [libs/modules/product/services/project.json](/Users/lidonghai/castlery/joyboy/libs/modules/product/services/project.json:1)
- [libs/modules/product/components/project.json](/Users/lidonghai/castlery/joyboy/libs/modules/product/components/project.json:1)

**4. domain / services / components 的依赖规则**
这是这套设计最关键的地方，不是“分了目录”就结束，而是用 `Nx + ESLint` 真正限制依赖方向。

核心规则在 [.eslintrc.base.json](/Users/lidonghai/castlery/joyboy/.eslintrc.base.json:1)：

- `components` 可以依赖：
  - `services`
  - `domain`
  - `shared-*`
  - `util`
- `services` 可以依赖：
  - `domain`
  - `shared-services`
  - `shared-domain`
  - `util`
  - `config`
- `domain` 只能依赖：
  - `shared-domain`
  - `util`
  - `config`

也就是说依赖方向基本是：

`components -> services -> domain -> util/config`

不能反过来依赖，比如：

- `domain` 不能依赖 `components`
- `services` 不能依赖 `components`

这就是“上层依赖下层，不能反向污染”。

**5. Nx tag 怎么配置**
这里主要用了两类 tag：

- `scope:*`
  表示它属于哪个业务域，比如 `scope:product`、`scope:checkout`、`scope:payment`
- `type:*`
  表示它属于哪一层，比如 `type:domain`、`type:services`、`type:components`

例如：

- [libs/modules/product/components/project.json](/Users/lidonghai/castlery/joyboy/libs/modules/product/components/project.json:1) 里是 `["type:components", "scope:product"]`
- [libs/modules/product/services/project.json](/Users/lidonghai/castlery/joyboy/libs/modules/product/services/project.json:1) 里是 `["type:services", "scope:product"]`
- [libs/modules/product/domain/project.json](/Users/lidonghai/castlery/joyboy/libs/modules/product/domain/project.json:1) 里是 `["type:domain", "scope:product"]`

然后在 [.eslintrc.base.json](/Users/lidonghai/castlery/joyboy/.eslintrc.base.json:8) 里用 `@nx/enforce-module-boundaries` 配 `depConstraints`，把“哪些 tag 可以依赖哪些 tag”写死。

除了三层规则，还有几条很重要：

- `scope:composite` 可以聚合多个业务域
- `scope:cms` 和 `scope:search` 更特殊，依赖更宽松
- `shared-components` 不能依赖其他模块的 `components`
- `base-components` 基本不依赖业务层，尽量纯设计系统

**6. 带来的收益**
收益很明确：

- 边界更清楚，知道一段代码属于哪个业务域、哪一层
- 改动影响范围更可控
- 多人协作时不容易随手乱引用
- 共享能力和业务能力更容易沉淀
- 对 CI/CD、affected build、依赖图分析更友好
- 长期演进时系统不容易继续失控

这些点在 [docs/architecture-review.md](/Users/lidonghai/castlery/joyboy/docs/architecture-review.md:47) 也有总结。

**7. 带来的代价**
这套设计也有成本，不是没有代价：

- 学习成本高，新人需要先理解业务域、分层和 tag
- 简单需求有时也要跨三层，开发路径更长
- 模块一多，project 和 alias 管理本身会变复杂
- 如果拆分粒度不稳，有的模块会变很大，有的模块会比较碎
- 某些跨域场景，如果边界设计不好，会出现“到底放 shared 还是 composite”的判断成本

文档里也提到过度分层和复杂度问题，在 [docs/architecture-review.md](/Users/lidonghai/castlery/joyboy/docs/architecture-review.md:88)。

**8. 有没有实际 case 证明边界变清晰了**
有，最典型的例子就是 `shared-components -> composite-components` 的迁移。

文档在 [docs/SHARED_TO_COMPOSITE_MIGRATION.md](/Users/lidonghai/castlery/joyboy/docs/SHARED_TO_COMPOSITE_MIGRATION.md:1)。

这个 case 说明了什么：

- `shared-components` 按规则只能依赖别人的 `services` / `domain`
- 不能依赖别人的 `components`
- 但像 `fortress-video`、`shipping-location-modal`、`pinch-zoom` 这些组件实际上依赖了别的模块的 `components`
- 所以它们就不该继续放在 `shared-components`
- 最终统一迁移到 `composite-components`

这个例子特别适合面试讲，因为它不是抽象原则，而是已经在项目里发生过一次真实治理动作。它说明边界不是只写在文档里，而是真的能指导重构和纠偏。

**面试回答版**
如果面试官问这一整块，你可以这样答：

“我们这套其实是 `Nx Monorepo + DDD 思想驱动的业务域拆分`，不是严格教科书式 DDD。  
原来 legacy 最大的问题是代码按页面和技术类型自然生长，业务相关代码分散，页面、状态、接口、副作用容易混在一起，多市场和多端下复用成本也越来越高。

所以我们先按业务域横向拆，比如 `product`、`checkout`、`payment`、`promotion`，然后每个域里再按三层纵向拆成 `domain`、`services`、`components`。`domain` 放领域状态和模型，`services` 放接口和业务编排，`components` 放 UI 和交互。

为了避免这个分层只停留在文档里，我们用 `Nx tag` 做依赖约束。比如每个库都会打 `scope` 和 `type` 两类 tag，像 `scope:product + type:services`，然后在 ESLint 里用 `@nx/enforce-module-boundaries` 限制依赖方向，基本就是 `components -> services -> domain`，不能反向依赖。

收益是边界更清楚、协作和治理更容易，代价就是学习成本和分层成本会更高。我们项目里有个比较直接的 case，就是以前一些组件放在 `shared-components`，但实际上依赖了别的模块的 `components`，后来按规则迁到 `composite-components`，这个就说明边界规则真的能指导我们做架构纠偏。”
