## 业务交付到平台架构是如何拓展延伸的，具体体现在你的哪些决策

我觉得这个延伸，核心不是说我不做业务了，而是我在做业务的时候，开始更多去思考“这个能力怎么沉淀成平台能力，怎么让后续需求的成本更低、风险更可控”。

具体体现我觉得主要有几类决策。

第一类是模块边界和架构分层上的决策。  
在 `Joyboy` 重构里，我们不是继续按页面或功能堆代码，而是按业务域去拆，比如 `product`、`checkout`、`payment`、`promotion`，同时在每个域内部再拆成 `domain`、`services`、`components` 这些层。这个决策的价值在于，后面无论是业务迭代还是多人协作，都能明确什么逻辑该放哪一层，避免页面组件不断膨胀。我自己在参与 `Account`、`Promotion`、`Checkout`、`Payment` 这些模块时，已经不只是实现需求，而是会主动去判断这块逻辑应该沉到领域层、服务层还是平台共享层。

第二类是渐进式迁移策略的决策。  
我们面对的是一个运行多年的 legacy 系统，所以不可能推倒重来。我参与比较多的是通过 `Feature Flag`、多市场配置和分阶段替换的方式，把迁移拆成可灰度、可回滚的过程。这个其实已经不是单纯业务开发视角了，因为你要考虑上线风险、市场差异、兼容成本，以及新旧系统怎么并存一段时间。也就是说，我做决策时会优先考虑“如何安全演进”，而不只是“如何把当前需求做完”。

第三类是设计系统和组件库建设。  
这部分也很能说明从业务交付到平台建设的转变。因为如果只看单个需求，最快的方式一定是局部写组件、局部调样式；但我后面开始主导 `Design Token`、`Storybook CDD`、`Chromatic` 视觉回归、多主题这些事情，本质上是在解决跨 Web、POS、CMS 的一致性和可复用性问题。这个决策意味着我会主动为团队后续几十个需求铺路，而不是只优化当前页面。

第五类是可观测性和工程治理上的决策。  
比如 `Sentry` 的结构化错误上报、`error_bucket` 分桶、ownership 路由、Server Action 包装、埋点事件契约这些，其实都不是某个业务功能强制要求你做的，但它们会直接影响后续团队排障、协作和迭代效率。我开始投入这类工作，说明我关注的已经不只是“功能是否上线”，而是“上线后出了问题，团队能不能快速定位，系统能不能长期稳定演进”。

所谓“从业务交付延伸到平台架构”，并不是角色突然变了，而是我在做同样的业务时，决策视角变了。以前更多是交付一个功能，现在会更主动去思考边界、复用、演进、安全和治理，这些决策累积起来，就会体现出平台架构层面的责任感。

## “为什么要做 Monorepo？之前的问题是什么？”

我们当时做 Monorepo，不是因为它“听起来更先进”，而是因为原来的开发方式已经开始明显制约业务和团队效率了。

第一，业务能力分散，复用成本高。  
在电商场景里，很多能力其实天然是共享的，比如商品、购物车、结算、支付、埋点、账户、设计系统这些。但如果代码是按应用、按页面或者按历史项目各自生长，最后就会出现同一套业务规则在不同端各写一份，或者看起来能复用，实际上复制修改了一份。短期看交付很快，长期就会带来逻辑漂移，改一个地方，另外几个地方不一定同步。

第二，边界不清，系统容易越做越耦合。  
之前 legacy 系统里，一个常见问题是页面、业务逻辑、接口调用、状态管理、埋点、副作用会混在一起。尤其在多人长期迭代之后，很多模块其实没有明确边界，新增需求时大家更容易“就近改动”。这种模式在需求少的时候还能接受，但一旦涉及多市场、多团队协作，系统复杂度会上升得很快，后面改动的风险也会越来越大。

第三，多端和多市场场景下，协作效率会持续下降。  
Castlery 不是单一站点，它有 Web、POS，还有不同国家和地区的市场差异。如果没有一个统一的代码组织方式，大家做相似能力时就会重复建设，而且每个市场的差异逻辑也容易散落在各处。最后会变成：公共能力不够公共，定制逻辑又没有被很好隔离，维护成本会很高。

第四，工程治理和发布治理会比较困难。  
如果代码仓分散，或者模块关系不清晰，像依赖分析、受影响构建、统一 lint 规则、测试策略、组件库联动、设计系统发布、可观测性规范落地，这些事情都会变难。你会发现不是不能做，而是做起来成本非常高，很难形成平台级约束。

本质上是因为原有模式下已经出现了共享能力分散、模块边界模糊、多端多市场协作成本高、工程治理困难这些问题，而 Monorepo 是我们做平台化重构的基础设施，它让系统能够在统一规则下持续演进。

## Nx 和 Turborepo / pnpm workspace 相比，你为什么选 Nx？

“我们当时选 Nx，主要不是因为它比 Turborepo 更流行，而是因为它更适合我们那种大型业务 monorepo 的治理场景。
pnpm workspace 更偏包管理，本身不解决架构治理问题；Turborepo 在任务编排和缓存上很好，但我们当时更需要的是模块边界约束、依赖图和 affected build 这种能力。因为我们的项目不是几个简单 package，而是 Web、POS、组件库、共享模块、多个业务域一起协作，如果没有强约束，后面很容易失控。
所以最后选 Nx，本质上是因为它不只是帮我们提速，更重要的是能帮我们把 monorepo 管起来。”

## 你们 Monorepo 怎么分包？按业务域还是按技术层？以及你们NX tag是如何约束依赖的

我们是两层一起做的，不是只按业务域，也不是只按技术层。

第一层是按业务域拆，比如 `product`、`checkout`、`payment`、`promotion`、`user`、`cms` 这些，这样做的好处是业务边界清楚，团队协作也更自然。

第二层是在每个业务域里面再按技术层拆，一般会分成 `domain`、`services`、`components`。
`domain` 放状态、实体、类型这些核心模型，`services` 放接口调用和业务编排，`components` 放 UI 和交互层。这样可以避免所有代码都堆到一个包里，后面越来越难维护。

`Nx tag` 这块，本质上就是把架构规则工程化。我们主要用了两类 tag。

一类是 `scope`，比如 `scope:product`、`scope:checkout`、`scope:shared`，表示它属于哪个业务域。  
另一类是 `type`，比如 `type:domain`、`type:services`、`type:components`，表示它属于哪一层。

约束上最核心的就是两点：

第一，限制分层依赖方向。  
比如 `components` 可以依赖 `services` 和 `domain`，但 `domain` 不能反过来依赖 `components`。这样能保证底层模型不会被 UI 污染。

第二，限制跨域依赖。  
比如普通业务域不能随便互相乱引用，共享能力要放到 `shared`，跨多个域的组合场景再放到 `composite`。这样做是为了防止时间长了之后，所有模块互相连成一团。

所以可以简单理解成：  
我们分包方式是“业务域横向拆分 + 技术层纵向分层”，`Nx tag` 负责把这个设计真正落到代码依赖约束上，而不是只停留在文档里。

## 这个是DDD领域驱动设计吗？和传统的按照技术分层 components/pages/services 有什么区别？

可以说我们是借鉴了 DDD 的思路，但不是很教科书式、很完整的那种 DDD。

更准确一点说，我们是在前端里用了“按业务域组织代码”的方式。  
传统的 `components / pages / services` 更像是按技术类型分目录，比如所有页面放一起、所有组件放一起、所有服务放一起。这样前期简单，但项目一大之后，和某个业务相关的代码会散得很开，改一个 `checkout` 需求，可能要到很多地方找代码，边界也容易变模糊。

DDD 这种按领域拆的方式，核心区别是它先问“这是谁的业务”，再问“它属于哪一层”。  
比如 `payment`、`checkout`、`product` 都是独立业务域，每个域下面再有自己的 `domain`、`services`、`components`。这样相关代码会收敛在一起，团队对边界的理解也更清楚。

我觉得它和传统模式最大的区别有两点：

第一，传统分法更偏“技术视角”，DDD 分法更偏“业务视角”。  
第二，传统分法到后面容易形成一个大共享池，DDD 这种方式更强调边界和职责，方便长期演进。

所以如果面试里回答，我一般会说：  
“我们不是严格照搬 DDD，而是吸收了它按业务域建模、强调边界的思想。相比传统 `components/pages/services` 这种纯技术分层方式，我们更适合复杂业务和多人协作场景。”

## 为什么当前的项目不是严格的DDD

因为我们虽然借鉴了 DDD 的思想，但本质上还是一个前端工程项目，不是从零按 DDD 理论完整建模出来的系统。

最主要有三个原因。

第一，前端天然更贴近交互和集成，不完全是业务规则中心。  
很多代码其实是在处理页面展示、状态同步、接口编排、埋点、缓存、第三方 SDK 这些事情，而不是纯领域规则。所以它很难像后端那样，把大量精力都放在 `Entity`、`Aggregate`、`Repository` 这种严格建模上。

第二，我们是基于 legacy 做渐进式重构，不是从零设计。  
如果严格做 DDD，前提 usually 是先把领域边界、上下文、模型都设计得很清楚。但我们当时更现实的目标是，一边承接业务，一边把系统慢慢拆开，所以很多地方是“先把边界收敛、先把依赖理顺”，而不是一步到位做完整 DDD 建模。

第三，严格 DDD 成本比较高，前端不一定划算。  
比如真的去做完整的聚合、仓储、领域事件这些，概念会很重，团队理解和维护成本也会上升。对前端来说，很多时候借鉴它最有价值的部分就够了，比如按业务域拆分、强调边界、分清领域逻辑和表现层，而不是把整套理论全部搬过来。

所以我一般会说，我们做的是“DDD 思想驱动的前端架构”，而不是严格意义上的 DDD。核心目标还是让复杂业务更清晰、更可维护，而不是追求概念上的完整。

## Product / Cart / Checkout / Search 这些 domain 之间怎么通信？

我们不是让 Product / Cart / Checkout / Search 彼此直接互调，而是先用 Nx tag 把边界卡住，然后通过三种方式通信：读共享 store 的 selector，写 command/listener 的副作用链，页面级组合放到 composite。这样做的好处是，跨域协作是允许的，但耦合点是显式的，可审计，也更容易做渐进式重构。

如果只看 `Product / Cart / Checkout / Search` 这几个 domain，关系可以这么理解：

`Product -> Cart`
在 PDP 里，Product 负责商品、变体、库存、配置选择；用户点加购之后，不是 Product 自己持有购物车，而是把当前选中的 `product / variant / quantity / bundle` 这些信息交给 Cart，由 Cart 去执行加购、刷新购物车数据。

`Cart -> Checkout`
Cart 是 Checkout 的上游。进入结算时，Cart 会把当前 `line items / gifts / coupon 后的购物车状态` 组装出来，调用 Checkout 初始化 session，所以 Checkout 不是自己“找商品”，而是吃 Cart 的结果。

`Checkout -> Order / Payment / Promotion / User`
虽然你这次只问四个 domain，但 Checkout 实际上是个中枢。它会继续去拿：

- `Order` 的订单号和订单信息
- `Payment` 的支付方式和支付结果
- `Promotion` 的 coupon / credits 变化
- `User` 的地址和用户信息

所以 Checkout 更像“交易编排层”。

`Search -> Product`
Search 主要负责搜索结果、筛选、PLP/Quickship 这类列表场景；点击商品后再进入 Product。也就是说 Search 负责“找到什么”，Product 负责“这个商品怎么配、怎么买”。

`Search -> Cart`
这条关系比 Product 弱一些，但有。比如搜索/Quickship 场景里的 zipcode、shipping 配置会影响后续加购和购物车可售判断，所以 Search 的一些结果会间接影响 Cart。

`Product -> Search`
也有反向的轻联系，比如 Product 里会复用搜索能力做商品搜索、联想搜索之类，但它不是主链路。

如果再压缩成一句面试话术，我会这么说：

“我们项目里几个业务域不是彼此平级乱调，而是有比较清晰的链路：Search 负责发现商品，Product 负责商品配置，Cart 负责交易前聚合，Checkout 负责最终结算编排。跨域调用主要发生在 service/composite 这一层，不会把业务逻辑直接糊在每个 domain 里。”

## 如果 Checkout 依赖 Promotion，又不能直接跨域依赖，你怎么设计？

**不是完全不能跨域依赖，而是不能在错误的层级跨域依赖。**

我不会让 Checkout 直接吃 Promotion 的业务规则，而是把 Promotion 当成一个上游能力。Promotion 负责改优惠状态，Checkout 只消费最终结算结果。跨域协作用 service、listener 或 composite 做编排，这样边界清楚，后面 promotion 规则变了，也不容易把 checkout 一起拖坏。

如果 `Checkout` 需要用到 `Promotion`，我的设计通常是这样：

`Checkout domain` 不直接依赖 `Promotion domain`
它只关心自己的结算态，比如订单摘要、支付前可否提交、优惠后的金额结果，不去理解 coupon 规则本身。

跨域编排放到 `service` 或 `composite`
也就是让上层去同时依赖 `Checkout` 和 `Promotion`，做一次 orchestration，而不是让 `Checkout domain` 自己去调用 `Promotion domain`。

具体有两种常见方式：

1. 拉模式
   用户在 checkout 页用 coupon，实际先走 `Promotion` 的 apply/remove 能力；
   成功后由 `Checkout service/listener` 刷新 `checkoutInfo`。
   这样 Checkout 拿到的是“优惠后的结算结果”，而不是自己去算优惠规则。

2. 推模式
   `Promotion` 成功后发一个事件，比如“coupon applied / credits redeemed”；
   `Checkout listener` 订阅这个事件，再触发 `getCheckoutInfo` 或重新拉 order summary。
   这就是“Promotion 负责发信号，Checkout 负责更新自己”。

## Monorepo 带来了哪些新问题？构建速度、依赖版本、边界维护怎么解决？

Monorepo 它把性能、依赖升级和架构边界都放大了。

我的思路是：用 Nx 解决 affected build，用 workspace 和分阶段升级控制版本风险，用 tag 和 lint 规则把跨域依赖收紧。这样 Monorepo 才会成为提效工具，而不是把复杂度集中到一个大仓库里。

我自己的体感，主要会带来三类新问题。
第一类是构建和开发速度。
项目一大，最容易出现的就是本地启动慢、CI 跑太多、一个小改动带着半个仓库重建。我们这边的处理方式比较直接，就是用 Nx 做 project graph 和 affected 机制，只让真正受影响的模块参与构建、测试和 lint，而不是每次全量跑。再往下会配合更细的 library 拆分，避免一个公共包过于肥大，改一点点就把很多下游一起打脏。
第二类是边界维护。
这个其实是最难的。因为大家都在一个仓库里，跨模块 import 太容易了，时间久了就会变成隐式耦合。我们项目里比较重视这个，所以不是只靠约定，而是用 Nx tag + ESLint module boundaries 去硬性限制，比如 domain、services、components 各自能依赖谁，composite 才允许做跨域聚合。这样至少能保证复杂度增长的时候，边界不会完全失控。

## Feature Flag 三层模型具体是哪三层？数据结构怎么设计？

开关层、规则层、配置层

**1. 开关层：这个功能整体开不开**
对应代码里的 `status`、`featureName`、`description`。

这层解决的是：这个功能是否被全局关闭。比如线上出问题，可以直接把 `status` 改成 `false`。

**2. 规则层：对谁生效**
同一个功能可以只对 `WEB` 开、不对 `POS` 开；只对 `SG/AU/CA/UK` 开、不对 `US` 开；只在某个环境或时间段开。

**3. 配置层：开启后具体怎么表现**

比如 `LEAD_TIME_FILTER` 不只是开关，它还带了不同市场的筛选配置：

```ts
payload: {
  enabled: true,
  leadTimeGroups: [
    { label: 'Ready To Ship', start: 0, end: 15 },
    ...
  ]
}
```

业务使用时不会自己判断所有规则，而是这样拿：

```ts
featureManager.isFeatureEnabled(FeatureName.LEAD_TIME_FILTER);
featureManager.getFeatureFlagPayload(FeatureName.LEAD_TIME_FILTER);
```

所以完整链路是：

```txt
Feature 配置
  ↓
FeatureManager 根据 channel / region / env / time 判断是否生效
  ↓
业务模块读取 boolean 或 payload
  ↓
组件/API 根据结果走新旧逻辑或展示不同 UI
```

面试里可以这样说：

“我理解的 Feature Flag 不应该只是一个 boolean，而是三层：
第一层是全局开关，比如 status；
第二层是生效规则，比如 market、channel、environment、时间窗口；
第三层是 payload，用来控制具体配置或变体。

我们项目里 `FeatureManager` 会统一做规则判断，业务模块只调用 `isFeatureEnabled` 或 `getFeatureFlagPayload`，避免每个组件都自己写一堆 region 判断。比如搜索页的 lead time filter，就是通过 flag 判断当前市场是否展示，并从 payload 里拿对应市场的筛选项配置。”

## 灰度迁移过程中，如果新旧逻辑同时存在，如何保证数据一致性？

你说的这种情况，我理解是比如写入走了新链路，但读取还在走旧链路；或者订单、支付、优惠这几个状态分别来自不同系统；再或者回滚之后新流量回到了旧系统，但已经落到新系统的在途数据还没有被正确收口。真正要控制的是同一笔数据的归属权和 source of truth，而不是单纯避免接口互调。”

但是我们在发布过程中一是会选择当地时间的凌晨，用户访问比较少，二是web 会进入维护模式，pos 不需要因为此时线下店下班，所以这种情况我们基本上不需要考虑

## 重构过程中如何避免业务停摆？

1. 新旧逻辑共存
   先实现旧逻辑，再开发重构后的组件和逻辑，两方的组件、service、hook都互相隔离，这样可以通过 feature flag、路由或市场维度把影响面收窄

2. 分模块局部发布
   按照业务模块，先让容易重构的模块发布上线，再解决复杂模块，上线一部分模块之后，通过新项目承接局部的流量，旧链路还可以兜底，使用AWS切换比较方便，避免全量切换的业务风险

3. 分市场发布

4. 发布、监控、回滚全链路设计
   发布前模拟发布流程，同时根据不同的模块制定不同的发布策略，有的模块需要在发布当天进行打包上线，有的模块提前上线但是不打feature开关，直到发布的时候才打开开关

重构期间真正的目标不是尽快删掉旧逻辑，而是保证业务持续可交付、线上持续可控。
