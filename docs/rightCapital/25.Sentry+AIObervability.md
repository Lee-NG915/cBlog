`Sentry + AI Observability` **在我们项目里，本质上不是“让 AI 自动修 bug”，而是先把错误变成结构化、可路由、可分诊的问题，再让 AI 参与分析和生成修复建议，最后通过 PR 流程进入人工审核，而不是直接上线。**

---

# 1. 这套设计到底想解决什么问题

如果没有这套体系，线上 bug 一般会有几个老问题：

- 错误很多，但噪音太大，不知道先看哪个
- 同一个问题可能被不同人反复看
- 支付、checkout、product、search 的错误混在一起
- 知道“报错了”，但不知道影响多大
- 知道“影响很大”，但不知道根因在哪
- 就算 AI 能分析错误，也不能直接让它改线上代码

所以这套方案其实是在做 4 层收敛：

1. **先统一采集**
   所有错误都走统一入口 `captureStructuredError`
2. **再统一分类**
   按 `domain`、`priority`、`severity`、`error_bucket`
3. **再统一路由**
   按 ownership rules 自动分配给对的人
4. **最后才是 AI 协助修复**
   AI 给分析和修复建议，但出口一定是 PR，不是直接上线

# 2. Sentry 和 Datadog 的分工

这块项目文档写得很清楚，在  
[docs/transaction-related/transaction-observability/README.md](/Users/lidonghai/castlery/joyboy/docs/transaction-related/transaction-observability/README.md:107)

你可以直接记一句：

- **Sentry 负责“为什么失败”**
- **Datadog 负责“影响有多大”**

### Sentry 负责什么

Sentry 更偏 root cause：

- 代码异常
- stack trace
- breadcrumb
- error grouping
- 哪一层抛错
- 哪个 release 开始有问题

适合看：

- checkout 为什么失败
- payment capture 报错在哪里
- 哪个 provider 的 adapter 抛异常
- 某个页面是不是 hydration error

### Datadog 负责什么

Datadog 更偏趋势和影响面：

- RUM 行为
- success rate / failure rate
- p95 时延
- 按 region / provider / version 下钻
- dashboard / SLO / alert

适合看：

- 支付成功率是不是掉了
- 某个市场是不是大面积受影响
- 某个版本上线后失败率是否上升
- 某个 provider timeout 是不是异常升高

### 为什么要分开

因为这两个问题不是一个平台最擅长同时解决的：

- Sentry 能告诉你“哪行代码炸了”
- Datadog 能告诉你“炸得有多广、多严重”

所以面试时不要说“两个都监控错误”，而要说：

**Sentry 看根因，Datadog 看趋势和影响面。**

# 3. `captureStructuredError` 的字段设计

这个函数是整个设计的核心入口。源码在：

[libs/shared/observability/src/lib/sentry/errors/structured-error.ts](/Users/lidonghai/castlery/joyboy/libs/shared/observability/src/lib/sentry/errors/structured-error.ts:1)

它做的事情不是简单 `Sentry.captureException(error)`，而是多做了几层统一处理：

### 第一层：标准化错误

先把各种错误格式变成标准 `Error`：

- 原生 Error
- RTK Query error
- API error
- string error
- 乱七八糟对象

这在  
[libs/shared/observability/src/lib/sentry/errors/error-utils.ts](/Users/lidonghai/castlery/joyboy/libs/shared/observability/src/lib/sentry/errors/error-utils.ts:101)  
里的 `normalizeError` 里做了。

### 第二层：自动补上下文

`captureStructuredError` 会调用 `enrichContext(context)`。

也就是说你传的不是一堆散乱字段，而是统一的 Sentry context。

字段里最关键的是：

- `domain`
- `pageType`
- `priority`
- `severity`
- `tags`
- `extra`
- `skipSentry`
- `fingerprint`

### 第三层：自动推断优先级

优先级不是每次手填，而是根据 `domain` 自动推断。

这块规则在：

[docs/observability/sentry-issue-routing-guide.md](/Users/lidonghai/castlery/joyboy/docs/observability/sentry-issue-routing-guide.md:37)

比如：

- `payment / checkout / user` → `priority: high`
- `order` → `priority: high`
- `cart / product / search / promotion` → `priority: medium`
- `wishlist / cms` → `priority: low`

这点很重要，因为它让错误上报不再只是“记录一下”，而是天然带有业务严重程度。

### 第四层：日志和上报合一

这个函数会：

- 先统一打日志
- 再决定要不要上报 Sentry

所以它不是只做 monitoring，也兼顾 debugging。

### 第五层：自动过滤 PII

真正发 Sentry 前，会自动做 `filterPII`。

这说明这个函数设计出来就不是裸上报，而是“安全版上报入口”。

# 4. `error_bucket` 怎么分

这一块有两层材料：

- 文档说明
- 真实代码

最重要的真实代码在：

[libs/shared/observability/src/lib/sentry/errors/error-bucket.ts](/Users/lidonghai/castlery/joyboy/libs/shared/observability/src/lib/sentry/errors/error-bucket.ts:1)

## 实际 bucket 有 6 类

- `js_fatal`
- `api_5xx`
- `api_timeout`
- `third_party`
- `hydration`
- `unclassified`

## 真正的分类顺序

代码里现在的分类顺序是：

1. `hydration`
2. `third_party`
3. `api_5xx`
4. `api_timeout`
5. `js_fatal`
6. `unclassified`

这个顺序很重要。

### 为什么这个顺序重要

因为一个错误可能同时满足多个特征。

比如：

- 某个第三方脚本请求 503
- 如果你先判 `api_5xx`，它就会被错归到 infra
- 但其实它更像第三方故障

所以项目里把 `third_party` 放在 `api_5xx` 之前，避免误路由。

### bucket 还有 confidence

除了 bucket 本身，还有 `bucket_confidence`：

- `high`
- `medium`
- `low`

这让我们后面做 dashboard 和告警时，可以区分“这是高置信分类”还是“兜底猜测”。

### bucket 是在哪里打上的

不是业务代码手填，而是在客户端 Sentry 初始化的 `beforeSend` 里统一打：

[apps/web/instrumentation-client.ts](/Users/lidonghai/castlery/joyboy/apps/web/instrumentation-client.ts:201)

这意味着：

- 业务代码只管报错
- bucket 逻辑由平台统一维护
- 策略升级不用业务侧到处改

# 5. ownership 怎么路由

这块特别适合面试讲，因为它体现的是“错误治理”，不是单点上报。

文档在：

- [docs/observability/sentry-issue-routing-guide.md](/Users/lidonghai/castlery/joyboy/docs/observability/sentry-issue-routing-guide.md:1)
- [docs/observability/sentry-ownership-rules-setup.md](/Users/lidonghai/castlery/joyboy/docs/observability/sentry-ownership-rules-setup.md:1)

### 核心原则

**自动分配不是写在代码里，而是代码负责打标签，Sentry Dashboard 负责路由。**

也就是：

- 代码产出 `domain / priority / error_bucket`
- Sentry Ownership Rules 根据 tag 规则分配给 owner

### 路由依据有哪些

主要有三类：

#### 1. `domain`

比如：

- `payment` → 支付 owner
- `checkout` → checkout owner
- `product` → product owner
- `cms` → CMS owner

#### 2. `error_bucket`

比如：

- `api_5xx` → infra
- `api_timeout` → infra
- `third_party` → platform

#### 3. `path / url`

作为兜底规则。

比如：

- `path:libs/modules/product/*`
- `url:*/checkout*`

### 为什么这样设计

因为只靠 path 不够稳，只靠 domain 也可能有漏网之鱼。  
所以它做成了多层兜底：

- 最优：domain
- 次优：pageType / context
- 兜底：path / url

这样分配给错人的概率会低很多。

# 6. Runbook 怎么写

我们这套 Runbook 不是普通“排障说明书”，更像给 AI 执行的 **结构化 SOP**。

核心目标是：AI 可以自动做低风险修复，但不能自由发挥。

**1. 输入层：AI 先拿什么信息**
Runbook 会规定先从 Sentry 取 issue，比如 title、stack trace、影响用户数、tags、error_bucket、transaction、release。不是让 AI 直接猜，而是先拿证据。

**2. 分类层：先判断能不能自动修**
AI 必须先分类：

```txt
baseline：监控配置问题，比如缺 domain/page_type/error_bucket
business-error：真实业务错误
boundary-caught：被边界捕获但体验需要优化
third_party / network：可能是外部噪声
```

然后再判断是否适合自动修。比如 checkout/payment、observability core、middleware/edge 这些高风险区域，Runbook 会限制不能随便改。

**3. 风险评分层：决定自动修还是交给人**
项目里有 estimation rubric，按四个维度打分：

```txt
文件范围
根因清晰度
测试覆盖
跨 domain 耦合程度
```

分数低、根因明确、影响范围小，AI 才能继续；如果分数高或 confidence 低于阈值，就输出“needs human review”。

**4. 修复规则层：告诉 AI 该改哪里，不该改哪里**
Runbook 不只是说“修一下”，而是把常见问题映射到固定修复点。

比如：

```txt
error_bucket 分错 → 看 error-bucket.ts
页面缺 domain/page_type → page.tsx 加 setGlobalSentryContext
layout 缺上下文 → layout.tsx 加 SentryContextProvider
第三方域名误判 → 更新 THIRD_PARTY_PATTERNS
```

这样 AI 不会看到一个 Sentry issue 后到处乱改。

**5. 验证层：修完必须证明没破坏**
修复后必须跑：

```txt
pnpm lint:observability
必要时跑 Sentry E2E
有相关单测就跑单测
```

而且是 bounded retry，最多几轮，失败就 escalate，不允许无限修。

**6. 闭环层：PR、ClickUp、Sentry 自动关联**
如果确实修复了 Sentry issue，PR title 和 description 要写：

```txt
fixes JOYBOY-WEB-XXX
```

但 commit message 里不能写。这样 Sentry 可以在 PR merge 后自动 resolve，同时 ClickUp 任务也能追踪。

为什么这么设计：

第一，**降低 AI 幻觉风险**。AI 不能凭感觉修 bug，必须先拿 Sentry 证据、stack、tags、影响面，再按规则判断。

第二，**控制自动化边界**。低风险问题可以自动修，高风险交易链路、支付、核心 observability 代码必须人工 review。

第三，**保证可追踪**。每个修复都能追到 Sentry issue、ClickUp task、PR、验证结果，不会变成“AI 改了一堆但没人知道为什么”。

第四，**保证可验证**。Runbook 强制 lint、E2E、测试证据，避免只生成代码不证明结果。

第五，**适合团队协作**。它把排查方式、owner、验证方式、PR 格式都标准化，AI 和人都按同一套流程工作。

一句话总结：

“我们的 AI Observability Runbook 本质是一套可执行的自动修复协议：先取证、再分类、再评分、再按固定规则修复、最后强制验证和 PR/ClickUp/Sentry 闭环。这样 AI 可以处理低风险、规则明确的问题，但不会越权修改高风险业务链路。”

# 7. AI 修复为什么进入 PR，而不是自动上线

### AI 在这里的角色不是 deployer，而是 assistant

AI 可以做的事情：

- 读错误上下文
- 看 stack trace / breadcrumbs / tags
- 生成 root cause hypothesis
- 给出修复建议
- 在本地改代码
- 生成 PR

但 AI **不负责最终上线**。

### 为什么一定要停在 PR

因为生产 bug 修复本身是高风险动作，尤其是：

- 交易链路
- 支付链路
- 用户态逻辑
- 多市场兼容逻辑

如果 AI 直接改线上，会有两个巨大风险：

#### 1. 误修风险

AI 可能修掉 symptom，但没修 root cause。  
甚至可能引入回归。

#### 2. 安全和合规风险

错误上下文里可能包含：

- token
- orderId
- user context
- payment context
- 第三方响应信息

这些不能让 AI 无限制写回线上逻辑。

所以最合理的出口就是：

**AI 生成 patch / PR → 人审核 → 测试 / CI → 合并发布**

这和项目里现有 PR 工作流、ClickUp 任务流也是一致的。

你面试里可以强调一句：

**我们让 AI 进入“分析和提案”环节，而不是进入“最终发布决策”环节。**

# 8. 如何处理隐私和误修风险

这是你面试时很加分的一块，因为很多人只会讲“AI 提效”，不会讲“AI 风险控制”。

## 8.1 隐私风险怎么控

### 第一层：代码层自动脱敏

`filterPII` 在源码里已经做了敏感字段过滤：

[libs/shared/observability/src/lib/sentry/errors/error-utils.ts](/Users/lidonghai/castlery/joyboy/libs/shared/observability/src/lib/sentry/errors/error-utils.ts:1)

过滤字段包括：

- password
- token
- accessToken
- refreshToken
- secret
- clientSecret
- cardNumber
- cvv
- email
- phone

而且联系方式会被打成 `[PII Filtered]`。

### 第二层：Server Action 显式 `sensitiveFields`

像 `withServerActionInstrumentation` 支持传：

- `sensitiveFields`

文档里也有例子，比如 `password`。

这说明服务端入口不是盲采集，而是默认考虑敏感字段。

### 第三层：Sentry 侧过滤噪音和第三方异常

`beforeSend` 里有很多过滤规则，会直接丢掉：

- 第三方脚本噪音
- WebView unload 噪音
- offline network error
- AbortError
- 一些 browser/privacy extension 造成的误报

这能减少无意义的数据进入 AI 分析链路。

# 10. 面试里怎么回答

你可以直接用这版，长度比较适中：

“我们项目里的 `Sentry + AI Observability`，核心不是单纯接一个错误监控，而是把线上问题治理做成一条结构化闭环。  
Sentry 和 Datadog 是分工明确的，Sentry 主要看 root cause，比如 exception、stack trace、breadcrumb 和 release 影响；Datadog 主要看影响面，比如 RUM、成功率、失败率、时延和按 provider/region/version 的趋势。

在代码层，我们统一通过 `captureStructuredError` 上报错误。这个入口会先把错误标准化，再补齐 `domain`、`priority`、`severity` 等上下文，并自动做 PII 过滤。然后在 Sentry 的 `beforeSend` 里统一做 `error_bucket` 分类，比如 `hydration`、`api_5xx`、`api_timeout`、`third_party`、`js_fatal`。这样错误不是散乱的，而是天然带有业务域和故障类型。

路由上，我们不是在代码里写死分配人，而是代码负责打标签，Sentry Dashboard 里的 Ownership Rules 根据 `domain`、`error_bucket`、path、url 自动把 issue 分给对应 owner。再往后，runbook 规定了告警来了之后先看 Datadog 影响面，再回到 Sentry 看根因，并按 traceId 把 Logs、RUM、Exception 串起来。

AI 修复这块，我们的原则是 AI 只能进入分析和修复建议阶段，最终出口一定是 PR，而不是自动上线。原因是交易链路对误修和隐私风险都很敏感，所以必须保留人工审核、CI 和 owner 决策这层安全阀。”
