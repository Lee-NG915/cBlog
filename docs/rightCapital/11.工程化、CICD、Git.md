## pnpm 和 npm/yarn 的区别？

`pnpm` 用内容寻址存储和硬链接，依赖不会在每个项目里重复复制，安装快、省磁盘；同时 `node_modules` 更严格，包不能随便访问未声明依赖。

`npm/yarn` 更偏传统扁平安装，幽灵依赖问题更容易出现。

## pnpm workspace 怎么组织包？

用 `pnpm-workspace.yaml` 声明 workspace 范围，比如 `apps/*`、`libs/*`、`packages/*`。包之间通过 package name 引用，pnpm 会在本地建立 workspace link，适合 monorepo 多包开发。

## Vite 和 Webpack 的核心差异？

Vite 开发环境基于原生 ESM，启动时不需要先把整个项目打包，按需加载模块，所以冷启动快；Webpack 是先构建依赖图再打包，配置能力强但 dev 启动和 rebuild 通常更重。生产环境 Vite 默认用 Rollup 打包。

## Webpack Tree Shaking 依赖什么条件？

主要依赖 ES Module 静态结构、生产模式压缩、`sideEffects` 标记、代码没有不可分析的动态导出或副作用。CommonJS、barrel export、副作用代码、错误的 `sideEffects` 配置都会影响 Tree Shaking。

## 你写过 ESLint 自定义规则吗？怎么实现？

写过。核心是基于 AST 做静态检查：定义 rule meta、create visitor，监听 `ImportDeclaration`、`CallExpression`、`JSXAttribute` 等节点，发现不符合规范的代码后 `context.report`，必要时提供 fixer。

我们这类项目可以用它限制模块边界、禁止裸 Sentry 调用、强制 page context。

## ADR 是什么？什么时候需要写 ADR？

ADR 是 Architecture Decision Record，记录重要技术决策、背景、选项、取舍和后果。

不是每个小改都写，通常是影响架构边界、长期维护、团队协作、不可轻易回滚的决策才写，比如缓存策略、组件库选型、监控规范、模块拆分方式。

## 代码规范如何落地，而不是只停留在文档？

靠自动化约束：ESLint、Prettier、TypeScript、单测、CI、commit hook、PR template、code review checklist。文档只解释原因，真正拦截要放在工具链和 CI 里。

## pre-commit、CI、review 分别拦什么？

`pre-commit` 拦低成本问题，比如格式化、lint staged、简单类型检查。`CI` 拦可重复验证的问题，比如完整 lint、test、build、E2E、bundle 检查。`review` 拦工具看不出来的问题，比如业务逻辑、架构边界、可维护性、风险判断。

## Git rebase 和 merge 的区别？

`merge` 保留分支历史，会产生 merge commit；`rebase` 把当前分支提交重新放到目标分支之后，历史更线性。团队协作分支通常避免随便 rebase 已共享提交，个人 feature 分支整理历史可以用 rebase。

## rebase 冲突怎么处理？

按提交顺序逐个解决冲突，先看当前冲突发生在哪个 commit，理解两边意图，修改后 `git add`，再 `git rebase --continue`。如果发现方向错了可以 `git rebase --abort` 回到开始前状态。

## CI/CD webhook 有什么安全风险？

风险包括伪造请求、重放攻击、泄露 secret、被恶意触发部署、payload 注入、权限过大。要做签名校验、时间戳/nonce、防重放、最小权限 token、IP/来源限制、敏感信息脱敏、环境隔离。

## 单元测试、组件测试、E2E 分别测什么？

单元测试测函数、hook、selector、service 这类小逻辑是否正确。组件测试测 UI 在不同 props、状态、用户交互下是否正确渲染和响应。E2E 测真实用户链路，比如搜索、加购、checkout、登录，验证多个系统组合后是否可用。
