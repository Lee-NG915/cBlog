## 你从 0 到 1 做组件库，第一步做了什么？为什么选择JoyUI?

第一步是和设计师沟通，确认出我们web整体的设计语言，包括颜色家族 字体家族 按钮 spacing 等等这些最基础最原子的规范，“统一设计语言和组件边界”，

不是着急写代码；选 Joy UI 是因为它在开发效率、主题能力、可扩展性和工程稳定性之间，性价比比较高。

## Design Token 怎么分层？

Design Token 分层核心就是“值、语义、组件”三层解耦，底层保证一致性，中层保证可换肤，上层保证组件可维护。

- **Primitive / Base Token 原子变量层**
  - 最底层原子值，比如 `color-blue-500`、`spacing-8`、`radius-12`、`font-size-14`
  - 这一层只存“值”，不带业务语义

- **Semantic Token 语义层**
  - 把原子值映射成语义，比如 `text-primary`、`bg-surface`、`border-subtle`、`danger-default`
  - 这一层解决的是“这个值是干什么用的”，方便换肤和主题切换

- **Component Token 组件层**
  - 再往上是组件级别，比如 `button-primary-bg`、`input-border-focus`、`card-padding`
  - 这一层让组件可以独立演进，不需要页面直接碰底层 token

如果是多品牌、多主题或者暗黑模式，我还会在 semantic 和 primitive 之间再加一层 **theme/brand mapping**，也就是：

- 同一套 semantic token
- 在不同 theme 下映射到不同 primitive value

我通常会强调一个原则：业务代码尽量不要直接用变量层，而是优先用语义层token；组件内部再根据需要消费 component token。

## Token、主题、组件样式之间是什么关系

这三层我一般理解成一个自上而下的映射关系：

- **Token** 是最底层设计变量
  - 比如颜色、间距、圆角、字号、阴影这些基础值，或者语义值如 `text-primary`、`bg-surface`
- **主题（Theme）** 是 token 的组织和映射规则
  - 它决定在某个品牌、某个模式下，这些 token 最终取什么值，比如浅色、深色、不同品牌皮肤
- **组件样式** 是最终消费 theme 的地方
  - 组件不会直接写死颜色值，而是去取 theme 里的 token，生成 button、input、card 这些具体样式

所以关系可以概括成：

**Token 提供基础设计语言，Theme 负责按场景做映射，组件样式负责真正落地渲染。**

如果再说工程上的意义，我会补一句：

**这样分层之后，改品牌或换主题时，尽量只动 token 和 theme，不需要大面积改组件实现。**

## 你们如何支持多主题和新旧主题无感切换？

这里首先我们在组件开发的时候，要注意使用语义层的变量覆盖组件的样式，每一套主题他们都有固定的语义层样式变量，比如同样的surfaceBg，在主题1可能为红色，在主题2就为绿色，本质上是通过注入不同的主题系统变量来实现主题切换，从技术实现上来说，在重构过程中我们有两套theme配置，基于 theme provider 切换系统注入的theme来实现整体主题样式切换

## Storybook 在你们流程里承担什么角色？

Storybook 在我们的流程里不是单纯“看组件”的工具，它更像是 **组件开发、联调、验收和回归的中间层**。

- **独立开发环境**
  - 组件可以脱离业务页面单独开发，不用每次跑整站找入口
- **组件文档和示例中心**
  - 把 props、状态、交互场景沉淀下来，方便研发和设计对齐
- **视觉回归基线**
  - 组件样式改动后，可以配合 Chromatic 这类能力做 diff，减少样式回归
- **CDD 的承载工具**
  - 我们做组件驱动开发时，会先把组件状态和边界场景在 Storybook 里跑通，再接业务页面
- **跨角色协作载体**
  - 设计、前端、测试都可以基于同一个 story 讨论 loading、empty、error、disabled 这些状态，而不是只看截图

最大的价值是把组件从业务页面里解耦出来，让开发、验收和回归都有统一载体。

## CDD 和普通业务开发有什么区别？

CDD 的出发点不一样，普通业务开发通常是：

- 先从页面或需求流程出发
- 组件更多是为了把当前页面拼出来
- 关注点偏业务交付速度

CDD 则是：

- 先从组件能力和状态边界出发
- 先把按钮、表单项、弹层、卡片这类基础单元做稳定
- 再用这些组件去组装页面和业务流程

所以两者差异我一般会总结成三点：

- **设计对象不同**
  - 普通业务开发以页面和功能为中心
  - CDD 以组件和可复用能力为中心
- **产出沉淀不同**
  - 普通业务开发容易交付完就停在页面里
  - CDD 会沉淀成组件、Story、文档、规范和测试资产
- **长期收益不同**
  - 普通业务开发短期更快
  - CDD 前期投入更高，但后期一致性、复用性、维护成本会更好

“是不是所有项目都适合 CDD”：

- 不是，业务验证期或者一次性页面很多的时候，没必要过度组件化
- 但在中后台、电商、多团队协作、设计统一要求高的场景里，CDD 的价值会非常明显。

## Chromatic 视觉回归能解决什么问题？有什么误报？

Chromatic 这类视觉回归工具主要解决的是 **样式类变更不容易被代码 review 发现** 的问题。

它能解决的核心问题有三类：

- **防止无意的 UI 回归**
  - 比如按钮高度、字体、间距、对齐、颜色被改了，但代码层面很难一眼看出来
- **覆盖组件状态**
  - 像 hover、disabled、loading、empty、error 这些状态，在 Storybook 里可以稳定沉淀成回归基线
- **降低多人协作时的样式风险**
  - 组件库、主题、token、公共样式一改，受影响面很大，视觉 diff 能比人工点页面更快发现问题

它不是用来证明“功能一定没问题”，而是更适合兜底 **视觉一致性**。

误报也很常见，常见来源有：

- **动态内容**
  - 时间、随机数、接口返回、轮播图、头像、推荐位这种内容一变就会 diff
- **动画和过渡**
  - 截图时机不稳定，opacity、transition、skeleton 可能导致前后截图不同
- **字体和渲染环境差异**
  - 字体加载时序、浏览器渲染细微差异，有时会造成轻微像素变化
- **异步状态未稳定**
  - 组件还在 loading、图片还没加载完、effect 还没跑完就截图了
- **主题或全局样式污染**
  - 某个 story 没隔离好 context、provider、全局 CSS，也会导致误报

如果面试官问“怎么减少误报”，我会补一句：

- 核心就是让 story **可控、静态、可复现**
- 把动态数据 mock 掉，关掉动画，等异步稳定后再截图，确保字体和运行环境一致

一句话总结：

**Chromatic 擅长发现视觉回归，但前提是 story 足够稳定；它最大的挑战不是能力不够，而是误报治理。**

## 为什么从 Joy UI 迁移到 Tailwind？收益和成本是什么？

ADR = Architecture Decision Record（架构决策记录）

目前由于项目优先级问题，组件库迁移被搁置了，我对现有 Fortress 体系更熟

Q：ADR 最终决策是什么？

「草案倾向 渐进式共存，而不是一次性替换 Joy。全量迁移风险在 theme/token 和 200+ 业务组件耦合。正式 ADR 还在等 design + 平台组 sign-off。」

Q：为什么选 Tailwind，不继续 Joy？

「Joy 通过 Fortress 已经满足大部分需求；考虑 Tailwind 主要是 bundle/DX/招聘市场 等因素。调研结论是：收益存在，但 迁移 ROI 取决于新 UI 改版节奏，所以还没拍板启动。」

Q：迁移第一步会做什么？

「如果启动，第一步不是改组件，而是 design token 映射 + 试点范围（例如一个新 landing 或 isolated widget），并定义 Fortress 组件的 deprecation 策略。这部分我在 ADR 草案里已经写了，但还没进 sprint。」

## 如何设计一个高度可定制的 DatePicker / Calendar / Modal / Form？

总原则：高可定制不是 props 越多越好，而是把稳定能力、扩展点、状态控制边界设计清楚。

- **DatePicker / Calendar**
  - 核心是日期状态、面板状态、选择模式分离
  - 选择模式要支持 single、range、multiple
  - 面板部分要暴露 `renderDay`、`renderHeader`、快捷选项、禁用日期、最小最大日期这些扩展点
  - 不把业务规则写死在组件里，比如节假日、库存日历、价格日历都应该通过外部配置注入

- **Modal**
  - 核心状态就是 `open`、`onClose`
  - 但真正的可定制点在于 `header / body / footer` 插槽、关闭策略、遮罩点击、ESC、focus trap、portal、z-index 管理
  - 也就是说，Modal 不应该只是一块壳子，而要把可访问性和弹层行为先做好，再给业务定内容结构

- **Form**
  - 我不会让 Form 自己管太多字段细节，而是把它设计成编排层
  - 字段值、校验、错误展示、布局、提交状态分开
  - 对外暴露 `Field`、`FormItem`、`FormSection` 这类组合能力
  - 和 `react-hook-form`、schema 校验这类生态兼容，比自己发明一套闭环更重要

**我设计高可定制组件时，重点不是多加 props，而是让状态可控、结构可插拔、样式可覆盖、业务规则可注入，这样组件才能既通用又不失控。**

## 组件库如何兼容 RSC？哪些组件必须是 client component？

- **拆入口**
  - `server-safe` 组件单独导出，比如纯展示型 `Box`、`Typography`、`Sheet`
  - 交互组件走 client entry
- **组件内部最小化 client 边界**
  - 不要为了一个小交互把整个大组件树都标成 client
- **样式体系走 SSR/RSC 友好方案**
  - 比如 CSS Variables、theme token，避免组件渲染必须依赖浏览器阶段
- **把数据获取留在 RSC，交互下沉到 client island**
  - 页面、列表、详情结构在服务端输出，筛选、弹窗、表单交互只包局部 client 组件

哪些组件必须是 client，我一般按这个标准分：

- **有 state / effect / ref 交互的**
- **依赖浏览器 API 的**
- **依赖事件、Portal、表单控制、动画、响应式监听的**

纯展示组件适合服务端渲染，比如布局容器、排版、静态卡片外壳、token 驱动的基础样式层，这些更适合保持 server-safe。

## 你对组件库做了哪些性能优化？有什么效果？

- 复杂的组件中合理使用`useMemo`、`useCallback`、`React.memo`提升加载速度
- 对首屏没有用到的theme style进行剪纸，减少首屏包的体积
- 使用joyUI emotion cache + useServerInsertedHTML 避免服务端和客户端样式重复注入、减少 hydration 阶段闪动和样式错位
- 字体库

- **减少无效渲染**
  - 对一些高频复用、纯展示或者局部交互组件用了 `useMemo`、`useCallback`、`React.memo`
  - 比如 `Button`、`Box`、`Card`、`Grid`、`Typography` 这类组件都会把 `sx` 标准化结果做 memo，避免每次 render 都重复算
  - 一些子节点比较重的场景，比如 DatePicker 的 header/day content、Modal holder、Icon wrapper，也做了 memo 化

- **降低样式计算和注入成本**
  - 我们没有让业务层大量写散乱的动态样式，而是尽量收口到 theme token 和 css variables
  - 这样组件切主题、切 variant 时，更多是变量切换，不是每次都生成一大堆新的样式对象

- **优化 SSR / App Router 下的样式输出**
  - 主题层专门做了 emotion cache + `useServerInsertedHTML` 的处理
  - 这个优化重点不是“更快一点点”，而是避免服务端和客户端样式重复注入、减少 hydration 阶段闪动和样式错位

- **控制组件边界，避免过重封装**
  - 通过 `slotProps`、`forwardRef` 这类方式做扩展，而不是再包很多无意义中间层
  - 这样能减少组件树层级和 props 透传损耗

如果面试官问“效果怎么样”，我会比较实话实说：

- **效果更多是工程层面的稳定收益**
- 在组件大量复用、页面节点很多的时候，无效 rerender 会少一些
- SSR 页面首屏样式更稳定，FOUC 和 hydration 样式错位问题会少很多
- 主题切换和样式扩展成本也更低，不容易因为样式系统失控拖慢页面

## Storybook 和 Chromatic 在流程里解决什么问题？

Storybook 解决的是**组件开发和组件验收**问题。

Chromatic 解决的是**视觉回归**问题。

简单说：

```txt
Storybook：把组件状态沉淀下来，方便开发、调试、评审、文档化。
Chromatic：基于 Storybook 做截图对比，防止 UI 被无意改坏。
```

在流程里通常是：

```txt
开发组件
  ↓
补 stories 覆盖关键状态
  ↓
PR 触发 Chromatic
  ↓
有视觉差异就 review：是预期变更就 accept，不是预期就修
```
