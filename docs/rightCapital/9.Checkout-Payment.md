## Checkout 链路最容易出什么问题？

其实这个过程包括payment关键的校验处理都在后端，前端看到的 total 只是展示结果，真正下单前后端还要重新校验价格、库存、优惠和配送条件。

前端侧主要负责防重复提交、展示中间状态、处理失败恢复；后端侧要保证幂等和最终一致性。

## Payment 前端如何处理重复点击？

- 可以在UI层处理，点击后禁用按钮
- 回调添加标记，只能等上一个请求处理完成之后才能继续

都有风险，因为state 时异步的，UI更新也可能有延迟，真正避免重复扣款一定要靠后端幂等，比如 checkoutSessionId、paymentIntentId 或 idempotency key。”

## 服务端幂等时如何防止重复处理的？

本质上就是将此次交易请求加一个key，后续根据key判断是不是同一个订单

服务端幂等主要靠稳定的 idempotency key 加唯一约束。请求进来后先原子写一条 PROCESSING 记录，只有写成功的请求能继续执行下单或支付逻辑；并发进来的重复请求会命中同一个 key，只能复用已有结果或返回处理中。这样可以避免重复创建订单、重复调用支付渠道。对于支付超时这种未知状态，我们会优先查询第三方支付结果，而不是重新发起扣款

## 支付状态如何和后端保持一致？

支付状态最终展示以后端订单状态为准。

前端发起支付后只进入 pending，然后用 orderId 或 paymentIntentId 去查我们后端的订单状态。

后端会结合支付渠道同步返回和 webhook 更新最终状态。

前端跳回成功页时也不会只相信 redirect 参数，而是再次查询后端。

如果 webhook 有延迟，就展示 processing 并轮询一小段时间。这样即使页面刷新、网络中断或者第三方回调慢，最终展示还是以后端订单状态为准。

## 前端如何处理接口超时但实际支付成功的情况？

如果支付接口超时，我不会在前端直接展示支付失败，也不会让用户立刻重新支付。因为这个时候请求可能已经到达后端，甚至第三方支付已经成功了，只是前端没有拿到响应。

我的处理方式是把状态标记为 processing 或 unknown，提示用户‘支付结果确认中’，然后重新去查询后端支付状态

## Stripe 集成时前端要注意哪些安全问题？

前端只能使用 publishableKey 和 Stripe Elements，不保存、不传输、不打印银行卡号、CVV、payment token 这类敏感信息。secretKey、创建 PaymentIntent、最终金额计算都必须在后端完成。

前端负责展示 Stripe UI 和触发确认支付，但不能持有密钥、不能决定金额、不能保存卡信息、不能判定最终支付成功。最终金额、幂等、订单状态和 webhook 确认都必须在后端闭环。

## 关键交易链路如何做监控和告警？

先把关键阶段拆出来，比如 cart validation、shipping、promotion、create payment intent、confirm payment、create order、order confirmation。

每个阶段都要有统一的 traceId / checkoutSessionId / orderId 关联起来，这样出问题时能知道用户卡在哪一步。
