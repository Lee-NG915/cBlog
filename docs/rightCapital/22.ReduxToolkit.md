这部分我尽量讲得非常白话一点。

先给你一句总括：

我们项目里的 `Redux Toolkit` 不是只拿来“存状态”的，它实际上做了三件事：

- 用 `slice` 管本地业务状态
- 用 `RTK Query` 管接口请求、缓存和自动生成 hooks
- 用 `listener middleware` 管“接口成功后要触发的一连串副作用”

你如果先把这三件事分开，后面就不容易乱。

---

**1. 先回答你最困惑的问题：项目里到底用了 React Toolkit 的什么特性？**

其实不是 React Toolkit，准确说是 `Redux Toolkit`。

在项目里主要用了三块：

- `createSlice`
  用来管理本地状态，比如订单状态、checkout 状态、用户状态
- `createApi / RTK Query`
  用来定义接口、缓存数据、自动生成 `useXXXQuery / useXXXMutation`
- `createListenerMiddleware`
  用来监听 action 或接口结果，然后做副作用

所以它不是单一功能，而是一整套状态和副作用方案。

你可以把它理解成：

- `slice` = “我本地要存什么状态”
- `query` = “我怎么请求后端、怎么缓存”
- `listener` = “某个动作发生后，我还要顺便干什么”

---

**2. 为什么项目里要用 Redux Toolkit？**

因为这个项目是复杂电商系统，不是简单页面。

它有几个特点：

- 全局共享状态很多，比如用户、订单、checkout、cart、payment
- 接口很多，而且很多页面会复用同一份接口数据
- 一个动作经常会带出很多后续动作，比如更新地址后要刷新 checkout、刷新 shipment options、处理埋点、处理错误
- Web 和 POS 还有多市场场景，状态逻辑很难只靠组件局部 state 解决

所以这里选 `Redux Toolkit`，核心不是因为它“新”，而是它把三个问题统一了：

- 状态怎么存
- 接口怎么管
- 副作用怎么管

相比手写 Redux，它的好处是：

- 模板代码少
- 类型支持更好
- 官方提供 `RTK Query`
- 官方提供 `listener middleware`
- 整个项目风格会统一

---

**3. RTK Query 在项目里到底是怎么用的？**

先看基础 API 定义，在  
[libs/shared/redux/services/src/api.ts](/Users/lidonghai/castlery/joyboy/libs/shared/redux/services/src/api.ts:1)

这里先建了一个基础 `api`：

- 统一 `baseQuery`
- 统一重试策略
- 统一认证 header 处理
- 统一 `tagTypes`
- 先不写具体 endpoints

也就是说，这里像一个“公共接口底座”。

然后具体业务模块再往这个底座里注入自己的接口。

比如 checkout 的 shipping 接口在  
[libs/modules/checkout/domain/src/api/shipping.api.ts](/Users/lidonghai/castlery/joyboy/libs/modules/checkout/domain/src/api/shipping.api.ts:1)

这里就是：

```ts
export const shippingApi = api.injectEndpoints({
  endpoints: (builder) => ({
    ...
  }),
});
```

它会定义：

- `query`
- `mutation`
- `invalidatesTags`
- 自动生成 hooks

比如：

- `useGetAvailableShipmentServicesQuery`
- `useAddServiceProductMutation`

所以业务组件用起来会比较直接。

---

**4. 为什么要用 `.injectEndpoints()`，不把所有接口都写在一个大文件里？**

这个问题很关键。

原因很简单：项目太大了。

如果把所有接口都写在一个 `api.ts` 里，会有几个问题：

- 文件会非常大
- 所有业务域耦合在一起
- `product`、`checkout`、`payment`、`order` 的边界会变差
- 不符合项目按业务域拆分的整体设计

所以项目采用的是：

- 先有一个公共 `base api`
- 然后每个业务域自己注入自己的 endpoints

这样做有几个好处：

第一，保留统一底座。  
比如认证、重试、base url、tag types 这些还是统一的。

第二，业务边界清楚。  
`checkout` 的接口放在 checkout 域，`order` 的接口放在 order 域。

第三，仍然享受 RTK Query 的统一能力。  
虽然 endpoint 分散定义，但底层还是同一个 `api`，tag invalidation、cache 行为都能统一工作。

所以你可以把 `injectEndpoints()` 理解成：

“公共框架统一，具体业务各自挂接。”

---

**5. 监听器 listener 又是什么？为什么也看起来像 RTK 的一部分？**

对，它就是 `Redux Toolkit` 官方提供的能力。

store 里直接创建了 listener middleware，在  
[libs/shared/redux/store/src/store.ts](/Users/lidonghai/castlery/joyboy/libs/shared/redux/store/src/store.ts:1)

这里有：

- `createListenerMiddleware(...)`
- `startAppListening`

然后在 store middleware 里挂进去。

这说明 listener 不是项目自己造的轮子，而是 RTK 官方方案。

你可以把 listener 理解成：

“某个 action 发生之后，系统自动做后续动作的地方。”

它和组件里直接写 `useEffect`、或者在点击事件里一路串逻辑，最大的区别是：

- 组件不用知道所有后续动作
- 副作用可以集中管理
- 多个入口触发同一个动作时，更容易复用

---

**6. listener 在这个项目里主要解决什么问题？**

它主要解决“副作用管理”问题。

最典型场景就是：

一个接口成功了，但事情还没结束。

比如：

- 更新地址成功后，要刷新 checkout 信息
- 加/删 coupon 后，要重新拉 checkout
- 支付完成后，要埋点
- 订单更新后，要重新拉 order

这些后续动作如果都写在组件里，会很乱：

- 页面逻辑会越来越重
- 同一个接口在不同入口被调用时，副作用容易不一致
- 很难统一维护

所以项目把这些“后续动作”放到 listener 里。

比如 checkout listener 在  
[libs/modules/checkout/services/src/lib/checkout.listener.ts](/Users/lidonghai/castlery/joyboy/libs/modules/checkout/services/src/lib/checkout.listener.ts:1)

你能看到很多类似这种逻辑：

- 某个 matcher 命中
- 然后 `dispatch(getCheckoutInfo.initiate(...))`
- 或者重新拉 shipment options
- 或者更新本地 state

这说明 listener 在这里更像“业务编排器”。

---

**7. `matchFulfilled`、`matchRejected` 这些是什么？**

这是 RTK Query endpoint 自动带出来的 matcher。

比如一个 mutation 叫 `updateCheckoutAddressZipcode`，它会天然带这些：

- `matchPending`
- `matchFulfilled`
- `matchRejected`

意思分别是：

- 请求中
- 请求成功
- 请求失败

listener 可以直接监听这些状态。

比如：

- `updateCheckoutAddressZipcode.matchFulfilled`
- `addCouponToCart.matchFulfilled`

这样就能在接口成功或失败之后做后续处理。

这比你在每个组件里手写 `try/catch + dispatch` 更统一。

---

**8. 那 listener 和 RTK Query 的关系到底是什么？**

你可以这么记：

- `RTK Query` 负责“请求和缓存”
- `listener` 负责“请求之后的业务动作”

也就是：

- 谁发请求？`RTK Query`
- 谁存接口缓存？`RTK Query`
- 谁自动生成 hook？`RTK Query`
- 谁在成功后刷新别的数据、补埋点、做业务联动？`listener`

它们是配套关系，不是替代关系。

---

**9. 项目里还有一种更细的设计：直接监听 endpoint，还是监听 domain event**

这个在文档里讲得很清楚：  
[docs/redux-listener-event-design-pattern.md](/Users/lidonghai/castlery/joyboy/docs/redux-listener-event-design-pattern.md:1)

这里的核心思想是：

如果一个接口只有一个简单场景，可以直接监听 `matchFulfilled`。

但如果：

- 同一个 API 会被多个入口调用
- 每个入口上下文不同
- 副作用有共性也有差异

那就不要只监听接口成功，而是让组件在成功后再 `dispatch` 一个更明确的业务事件，也就是 `Domain Event`。

文档举的例子很典型：

- 同样是加车
- 可能来自 PDP、推荐位、收藏夹、搜索页
- API 一样，但 source 不一样
- 埋点和后续动作也不完全一样

所以更合理的做法是：

- API 成功
- 组件 dispatch 一个 `addedToCartEvent`
- listener 监听这个业务事件，而不是猜来源

这就是为什么项目里你会看到两种写法：

- 直接监听 `xxx.matchFulfilled`
- 监听一个自定义 event

这不是混乱，而是按场景选方案。

**10. 你怎么理解这一整套设计？**

我帮你总结成一句非常好记的话：

这个项目里，

- `Redux` 是状态容器
- `RTK Query` 是接口和缓存层
- `Listener` 是副作用编排层

它们组合起来，目标就是把复杂电商项目里的：

- 状态
- 请求
- 联动逻辑

拆开管理，不让所有东西都堆在组件里。

**11. 面试里怎么回答比较合适？**

你可以这样说：

“我们项目里用 Redux Toolkit，不只是为了替代传统 Redux，而是把状态、本地业务逻辑、接口缓存和副作用编排放到一套统一方案里。  
具体来说，`slice` 负责本地业务状态，`RTK Query` 负责接口定义、缓存和自动生成 hooks，`listener middleware` 负责请求成功或失败之后的一些联动逻辑，比如刷新 checkout、刷新订单、处理埋点或者错误处理。

`injectEndpoints()` 的作用，是在一个统一的 base api 上按业务域拆接口。这样既保留了统一的认证、重试和 cache/tag 能力，又不会把所有接口堆到一个超大的文件里，比较符合我们 monorepo 按业务域拆分的设计。

至于 listener，它本质上是 Redux Toolkit 官方提供的副作用能力。我们主要用它来处理‘接口完成之后还要做什么’，避免把这些联动逻辑全部写在组件里。尤其在 checkout、order、payment 这种复杂链路里，这种设计会更清晰。”
