这块不是单纯“登录了就带 token 请求”这么简单，而是已经形成了一套比较完整的治理思路：

`认证状态存储 -> 请求头注入 -> 401 自动续期 -> App Router 中间件兜底 -> HTTP 错误分层处理`

而且很真实的一点是，你们项目里同时存在“当前线上通用实现”和“更理想的最佳实践方案”。这反而很适合面试，因为它说明你理解的不只是结果，还理解这套体系为什么要继续演进。

**1. 这块的核心目标是什么**

这套设计本质上想解决 3 个问题：

1. 用户不要因为 token 刚过期就被直接踢下线。
2. 同一类 HTTP 错误要在固定层处理，不能每个业务自己乱写。
3. App Router 下客户端、服务端、中间件的职责要分清，不然很容易出现重复刷新、死循环、状态不一致。

所以它不是“鉴权功能”，更像“前端请求基础设施”。

**2. 认证与鉴权在你们项目里怎么落地**

你们项目里有一个统一的 persistence 体系，不同 token 放的位置其实是按场景分的。

对于 Web，`access token` 和 `refresh token` 主要是 cookie handle；对于 POS，更多是 localStorage。除了登录态 token，还有 `xCartToken`、`xCheckoutSessionToken` 这类业务 token。也就是说，你们不是只有一个“用户 token”，而是把“身份凭证”和“业务会话凭证”分开了。

请求发出去之前，会统一走 `prepareHeaders`。这里会自动注入：

- `X-Channel`
- `X-Access-Token`
- Web 的 `X-Cart-Token` / `X-Checkout-Token`
- POS 的 `salesId`、`retailStoreId`、`customerId` 这类上下文

这个点面试里很好讲，因为它说明你们不是在业务组件里手写 header，而是把鉴权注入沉到了基础设施层。

**3. Token 刷新为什么不能只说“401 了就 refresh”**

真正难的不是 refresh 本身，而是并发。

比如页面首屏同时发 6 个请求，token 过期了，这 6 个请求会一起收到 401。如果每个请求都自己去调 refresh，就会出现 refresh 风暴，甚至相互覆盖。

你们现在的共享 baseQuery 已经用了 `async-mutex` 来做并发去重，逻辑就是：

- 第一个 401 请求拿锁，去调 `/oauth/token`
- 其他 401 请求先等锁释放
- refresh 成功后，排队请求再带新 token 重放
- refresh 失败就清 token，交给上层处理

这个是这块最值得你吃透的点，因为它比“会用 RTK Query”更能体现工程深度。

**4. App Router 下为什么还需要 Middleware**

这是第二个很容易被追问的点。

客户端发 API 请求时，可以在 RTK Query 里 silent refresh；但 Server Component 不一样，它本质上只适合“读 cookie、发请求、决定渲染结果”，不适合自己做 token 刷新。

原因很简单：Server Component 自己不好承担完整的“刷新后写回 cookie 再继续渲染”的职责，尤其在 App Router 里，这件事更适合放在 Middleware 前置做。

你们项目里实际上已经有这个思路了：

- 中间件会先判断用户是否已登录
- 还会解析 JWT 的 `exp`
- 如果 access token 过期，就优先在 middleware 里尝试 refresh
- refresh 失败，再根据路由决定是否跳登录页

所以职责分工是：

- 浏览器请求接口：RTK Query 处理 401 refresh
- 页面进入前：Middleware 处理 cookie 级别的 refresh
- Server Component：只读 cookie，不自己刷新，拿到 401 就走 redirect 或降级

这句话你面试里一定要会说，因为这就是 App Router 场景下的边界感。

**5. HTTP 错误治理的核心不是“拦截所有错误”，而是“分层”**

你们这块最成熟的认知其实在文档里已经写得很清楚了：不是所有错误都应该在同一层处理。

大致可以这么理解：

- 基础设施层处理“和业务无关”的错误
比如 401、429、5xx、网络错误、重试、错误上报
- 业务层处理“和业务语义强相关”的错误
比如优惠券失效、库存不足、地址不完整、表单字段校验失败

这里你们有一个很关键的约定：不是所有失败都用 HTTP status 表达，很多业务失败还是 `HTTP 200 + 业务错误码`。

这就引出一个典型坑：伪 200 业务错误，HTTP 拦截器其实是感知不到的，所以不能只靠 `res.status`。你们最佳实践文档里已经明确要把这种响应统一转成 `BizError` 再交给业务层处理。

这个很像真实大厂项目，不是教科书式“所有错误都 RESTful”。

**6. 401、403、404、5xx 在你们体系里分别怎么理解**

这个最好面试时一定讲清：

`401`
不是“立刻跳登录”，而是“先尝试 silent refresh，失败再交给上层跳登录”。

`403`
你们想把它收敛成“账号封禁/风控拦截”这类账号级问题，而不是泛化成所有权限不足。因为一旦 403 语义不清，很容易误清登录态。

`404`
要区分两类：

- 页面路由不存在，走全局 404
- 资源不存在，不一定跳 404，很多时候是业务空状态

`5xx / 408 / 429`
这些属于基础设施层问题，应该由统一重试、退避、上报来处理，不要每个业务自己手写一遍。

**7. 你们当前实现里有哪些真实的坑**

这一段特别适合面试，因为会显得你不是在背方案，而是真的理解项目。

第一，当前共享 baseQuery 是“有刷新能力的可用实现”，但还不是最理想形态。  
比如它外层直接用了 RTK 的 `retry(maxRetries: 3)`，这意味着 4xx 的重试控制还不够精细，所以文档里才会继续提出 `smartBaseQueryFn + retry.fail()` 的分层方案。

第二，当前 403 处理偏激进。  
现有逻辑里很多 403 会直接清 token，POS 还可能 `location.reload()`。这在早期能快速兜底，但长期看语义太粗，容易把“权限不足”和“账号失效”混在一起。

第三，fake 200 是天然坑。  
如果后端失败还返回 200，那基础设施层如果不主动识别 `code !== 0`，业务层就会误以为请求成功。

第四，Web 和 POS 其实是两套运行时。  
Web 偏 cookie，POS 偏 localStorage；甚至 POS 还处理了不同 host 的 API 兼容。这意味着你面试里不能把 token 刷新讲成一个单一方案，而要强调“同一套治理思想，不同运行时各自适配”。

第五，cookie path / 区域隔离也是个隐性坑。  
你们 persistence handle 里默认按国家 path 做隔离，这对多站点是必要的，不然不同站点 token 可能串掉。

**8. 我建议你怎么把这个模块串成自己的理解**

你可以把它记成一句话：

“我们把认证和错误处理当成请求基础设施，而不是业务组件能力。”

然后展开成 4 层：

1. 持久化层：token 和业务会话分开存
2. 请求层：统一注入 header
3. 续期层：401 自动 refresh，Mutex 防并发风暴
4. 治理层：HTTP 错误按基础设施层和业务层分开处理

这样一讲，逻辑会很顺。

**9. 面试里怎么答**

如果面试官问你“你们认证和 token 刷新怎么做”，你可以这样回答：

“我们项目里把认证和 HTTP 错误治理做成了统一基础设施。登录后 token 不会让业务组件自己带，而是统一通过 prepareHeaders 注入请求头。401 时不会直接踢用户下线，而是先在 RTK Query 的 baseQuery 里做 silent refresh，并且用 Mutex 保证并发请求只会触发一次 refresh，其他请求等刷新完成后再重放。  
在 Next.js App Router 场景下，我们又把页面进入前的 token 校验和刷新前移到 middleware，因为 Server Component 更适合读 cookie 和决定渲染，不适合自己承担刷新和写回 cookie 的职责。  
HTTP 错误治理上，我们把 401、429、5xx 这类通用问题放在基础设施层处理，把优惠券失效、库存不足、表单校验失败这类业务问题留在业务层处理。这样好处是用户体验更稳定，错误语义也更清晰。”  

如果面试官继续追问“这里最难的点是什么”，你就答：

“最难的不是 refresh，而是并发 401 的去重，以及 App Router 下 middleware、server component、client request 三者的职责边界。”
