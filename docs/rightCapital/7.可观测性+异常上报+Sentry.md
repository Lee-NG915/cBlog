## Sentry 在你们系统里负责什么？

Sentry 在我们系统里负责线上错误发现、业务上下文归因、第三方噪音过滤和发布后回归追踪。

它不是只看 stack trace，而是帮助我们判断错误发生在哪个业务域、影响多大、是不是需要马上处理。

## Datadog 在你们系统里负责什么？它和 Sentry 的边界是什么？

**Sentry 负责错误归因。**  
比如某个页面抛了 JS error、Server Action 抛错、接口异常没有被兜住，Sentry 会告诉我们 stack、版本、页面、业务域、用户环境。它适合回答：**哪里报错了？是不是新版本引入的？影响范围多大？**

**Datadog 负责性能和链路。**  
比如 RUM 里看 LCP、INP、CLS、资源加载、长任务、页面加载耗时；APM / trace 看一次请求从前端到 BFF、后端、第三方服务哪一段慢。它适合回答：**用户为什么觉得慢？慢在前端、网络、服务端还是第三方？**

Sentry 是错误监控和错误归因工具，Datadog 是性能、RUM 和链路观测工具。Sentry 回答“哪里报错、为什么报错、影响谁”，Datadog 回答“哪里慢、慢多久、慢在哪一段”。线上排障时两者会一起用，但主职责不一样。

## 你们怎么做错误分桶？error_bucket 是怎么定义的？

我们的错误分桶是在Sentry beforeSend 里统一做，通过 error_bucket 这个 Sentry tag 把错误按处理优先级和归因类型先分类，方便告警、dashboard 和排查。

1. 从 event 里取 message、errorName、stack frames、statusCode
2. 判断是不是主动上报的 captureStructuredError
3. 调 classifyErrorBucket
4. 先写入 event.tags.error_bucket 和 bucket_confidence
5. 再继续走过滤规则，过滤第三方噪音、浏览器插件、无价值错误等

Sentry 错误先按 error_bucket 做业务化分类

- 客户端区分 hydration、network、api_5xx、api_timeout、third_party、app_error、js_fatal，
- 服务端区分 upstream_5xx、upstream_timeout、app_error、js_fatal。

Sentry tag 在 beforeSend 里统一计算和写入，业务代码只负责带 domain、pageType、priority 等上下文，不重复实现分桶逻辑。这样告警规则可以按 bucket + domain + priority 组合，而不是所有错误都一股脑报警。

## 为什么要做 10 条互斥告警？

本质上还是为了防止错误重复上报，导致排查的时候效率低，让每个错误只进入一个最合适的告警通道。也方便后续按 page_type、domain、error_bucket 做归因和统计。

## 如何避免告警风暴？

不能只关注减少上报的数量，要保证错误上报的质量，依靠 分层、过滤、聚合、阈值 这几种方式提升错误上报的多质量，保障错误都是有效且便于排查的，合理设计之后仅靠网络或者第三方的一些错误是不会达到把系统打爆的一个层次

- beforeSend 先过滤低价值噪音
- error_bucket 做错误分桶，互斥告警保证一个错误只进一个通道
- 再用 priority、阈值和时间窗口控制触发条件

## Ownership 自动分配怎么做？

我们不是靠人肉分配 Sentry issue，而是在错误上报时统一打 domain、page_type、error_bucket、priority 这些标签，

在Sentry Action控制台中配置错误上报时的自定义规范，根据我们带来的标签自动assigne给对应开发

## Runbook 里包含哪些内容?

Runbook 里要包含告警含义、影响范围、触发条件、排查入口、处理步骤、降级方案、owner 和升级路径、以及修复后的验证指标。

它的目标是让任何接手的人不用重新理解系统，也能按步骤把问题定位和止血。

## AI 自动修复链路

我做的是 AI 辅助自动修复链路。Sentry / Datadog 负责自动监控和告警，告警里带 issue id、domain、page_type、error_bucket 等结构化上下文；Codex 这类 coding agent 被触发后，通过 Sentry API 拉取 issue 详情，根据 runbook 定位代码，生成最小修复并补测试，最后产出 PR。低风险问题可以很快闭环，关键业务链路仍然需要 owner review 和发布 gate。

n8n 做告警编排，用 Dify 做 AI 分类和决策，用 Codex 这类 coding agent 做代码修复。Sentry/Datadog 告警进 n8n，n8n 拉 issue 上下文后交给 Dify 判断风险和 runbook；如果命中低风险白名单，就触发 coding agent 创建分支、修复、测试并开 PR。人工只需要 review PR，高风险问题只生成诊断报告，不自动改代码。

## 错误上报时客户敏感数据如何脱敏？金融/交易场景下有什么边界？

我们会把脱敏放在统一 observability / logger 层，而不是让业务代码各自处理。策略是字段白名单优先，黑名单正则兜底；

默认不上报完整请求体和响应体，只保留 error code、status、domain、orderId、traceId 这类排查字段。金融和交易场景边界更严格，卡号、CVV、支付 token、cookie、authorization、完整联系方式和支付 SDK 原始 payload 都不能进日志或 Sentry。
