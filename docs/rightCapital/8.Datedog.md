Web 项目里的 Datadog 分成 **客户端 RUM** 和 **服务端 APM** 两层，都是 SDK 自动采集，业务代码里没有手写 `datadogRum.addAction` 这类自定义埋点。

---

## 整体架构

```
浏览器 (用户访问 castlery.com)
    │
    ├─ @datadog/browser-rum  ← apps/web/app/datadog-init.tsx
    │     采集：点击、Long Task、前后端 trace 关联
    │
    └─ HTTP 请求 → Next.js Server
                        │
                        └─ dd-trace  ← apps/web/instrumentation.js
                              采集：HTTP/Next 自动 trace、日志 trace_id 注入
```

和 **Sentry** 的分工：Sentry 负责错误/告警；Datadog 负责性能 trace、RUM 体验数据。两者并行，不互相替代。

| 配置项                                      | 含义                               |
| ---------------------------------------- | -------------------------------- |
| `sessionSampleRate: 50`                  | 50% 用户会话被采样                      |
| `trackUserInteractions: true`            | 自动记录点击等交互                        |
| `trackLongTasks: true`                   | 记录主线程长任务（和 INP 分析相关）             |
| `trackResources: false`                  | 不采集静态资源加载详情                      |
| `defaultPrivacyLevel: 'mask-user-input'` | 输入框内容脱敏                          |
| `allowedTracingUrls`                     | 仅对 castlery 域名 API 做前后端 trace 关联 |

## 2. 服务端 APM 配置

**入口文件**：`apps/web/instrumentation.js`（Next.js `instrumentationHook` 启用）

```3:18:apps/web/instrumentation.js
export async function register() {
  if (process.env.NEXT_RUNTIME === 'nodejs' && process.env.NODE_ENV !== 'development') {
    const tracerLib = await import('dd-trace');
    const tracer = tracerLib.default;

    tracer.init({ logInjection: true, runtimeMetrics: true });
    tracer.use('http', {
      blocklist: [
        `/${EcEnv.NEXT_PUBLIC_COUNTRY.toLowerCase()}/health-check`,
        `/${EcEnv.NEXT_PUBLIC_COUNTRY.toLowerCase()}/api/health-check`,
        '/ping',
      ],
    });
    tracer.use('next');
```


| 配置                                | 作用                                 |
| --------------------------------- | ---------------------------------- |
| 仅 `production` + `nodejs` runtime | 本地 dev 不启动 dd-trace                |
| `logInjection: true`              | 日志里自动注入 `trace_id`，方便日志 ↔ trace 关联 |
| `runtimeMetrics: true`            | Node 运行时指标（CPU、内存等）                |
| `tracer.use('http')` + blocklist  | 自动 trace HTTP 请求，health-check 排除   |
| `tracer.use('next')`              | 自动 trace Next.js 请求处理              |

## 3. 在什么场景使用

### 当前代码里的使用方式：**全自动，无业务埋点**

全仓库搜索结果显示，`datadogRum` 只出现在两个 init 文件里，**没有**业务组件手动调用 Datadog API。也就是说：

- 不需要在页面里写 Datadog 代码
- 用户访问任意页面 → layout 加载 → SDK 自动开始采集

### 实际监控场景


| 场景              | 用的 Datadog 能力         | 典型用途                                                                                                                              |
| --------------- | --------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| **页面性能 / 用户体验** | RUM（Long Task、交互）     | 分析 INP、卡顿、rage click                                                                                                              |
| **前后端链路追踪**     | RUM + APM trace 关联    | 一次用户操作对应后端哪个 API 慢                                                                                                                |
| **服务端 API 耗时**  | dd-trace APM          | SSR、API Route、下游 HTTP 瀑布图                                                                                                         |
| **日志排查**        | logInjection          | 在 Datadog Logs 里用 `trace_id` 串日志                                                                                                  |
| **运维/压测分析**     | Datadog APM Dashboard | 如 Confluence [Cart&Checkout 耗时分析](https://castlery.atlassian.net/wiki/spaces/EC/pages/3602743519) 里用 trace 瀑布图定位 Lisa/Maggie 跨洋调用 |


### 不使用的场景


| 场景                           | 用什么代替                                        |
| ---------------------------- | -------------------------------------------- |
| **错误告警 / 异常分类**              | Sentry（`captureStructuredError` 等）           |
| **CLS 精确定位**                 | 自研 `web-vitals.tsx` → Sentry                 |
| **交易链路监控（checkout/payment）** | 正在迁移到 OTel → Grafana/Tempo，**明确不再用 Datadog** |
| **POS 端**                    | RUM 已在 layout 里**注释掉**，相当于关闭                 |

## 4. 运行条件小结


| 组件            | 何时启用                                          | 何时不启用                         |
| ------------- | --------------------------------------------- | ----------------------------- |
| **RUM (web)** | 有 `DATADOG_APP_ID` + `CLIENT_TOKEN`，layout 加载 | 缺 env → 只打 console.error，不初始化 |
| **dd-trace**  | `NODE_ENV=production` 且 Node runtime          | 本地 `pnpm web:us` dev 模式不启动    |
| **POS RUM**   | 代码存在但 layout 已注释                              | 当前 POS 实际未启用                  |

## 5. 和团队规划的关系（重要）

根据 `docs/transaction-related/transaction-observability/`，Datadog 处于 **逐步下线** 阶段：

- **P3 目标**：移除 `@datadog/browser-rum`、`dd-trace`、`datadog-init.tsx` 及 DD env
- **替代方案**：OpenTelemetry → Grafana Tempo/Prometheus/Loki
- **RUM 体验指标**（Web Vitals、rage click）可能单独立项「Web Experience milestone」，尚未确定新方案

当前状态：**代码还在、Web RUM 仍开着，但新 observability 规范以 Sentry + OTel 为主，Datadog 不再是长期方向。**

## 一句话总结

Web 项目的 Datadog 配置很「薄」：**RUM(客户端用户初始化) 在根 layout 初始化一次，APM（服务端初始化） 在** `instrumentation.js` **里自动 trace 服务端请求**。使用场景主要是 **性能监控、链路追踪、日志关联**；错误用 Sentry，交易监控正在迁到 OTel。业务代码不需要也不应该直接调 Datadog API。
