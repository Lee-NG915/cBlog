**我们项目里的性能优化，不是一次单点优化，而是分成了三类问题分别处理：**

1. 页面和接口慢不慢
2. bundle 大不大
3. 搜索这种高频交互场景有没有重复请求和重复计算

---

**1. 性能问题是怎么发现的**

我们不是只靠主观感觉“页面有点慢”，而是靠几种方式一起发现。

第一类是用户侧指标。  
像前面讲过的 Datadog RUM、Web Vitals、页面交互时延，可以帮助我们看到首屏、交互、请求链路是不是异常。

第二类是工程侧分析。  
项目里有专门的打包分析脚本，在 [package.json](/Users/lidonghai/castlery/joyboy/package.json:1) 里有 `pnpm web:analyze`，而 `next.config.js` 里也接了 `@next/bundle-analyzer`，见 [apps/web/next.config.js](/Users/lidonghai/castlery/joyboy/apps/web/next.config.js:16)。  
这类分析主要用来回答：到底是哪个包、哪个组件、哪个第三方依赖把首屏 JS 变大了。

第三类是业务链路里的现象反推。  
比如搜索页是最典型的。用户在前进后退、切换筛选、重复输入时，如果每次都重新打请求，就会感觉“搜索很肉”。这个不是 Lighthouse 能完全说明的，需要结合真实交互去看。项目里后来专门做了搜索结果缓存文档，见 [docs/search-result-cache.md](/Users/lidonghai/castlery/joyboy/docs/search-result-cache.md:1)。

**2. 我们主要看的指标是什么**

这块你面试里不要只说“优化了性能”，最好说清楚你们看的是哪类指标。

我建议你记这几类：

- 页面级：TTFB、首屏速度、交互响应时间
- 体验级：重复进入页面时是否秒开，筛选切换是否顺滑
- 请求级：同一类请求有没有重复发
- 构建级：bundle 体积、哪些 chunk 过大、哪些依赖不该进首屏

在搜索场景里，项目里最明确的指标其实是：

- 单次搜索响应耗时
- 缓存命中后返回耗时
- 请求数是否下降

**3. bundle analyzer 看到了什么**

项目里虽然没有一份“最终 bundle 分析报告”直接把结论全写出来，但从代码和文档能看到几个明确方向。

第一，团队明确把 bundle 作为一个专项治理对象。  
`next.config.js` 里已经接了 bundle analyzer，说明它不是临时排查，而是一个标准能力，见 [apps/web/next.config.js](/Users/lidonghai/castlery/joyboy/apps/web/next.config.js:16)。

第二，针对大包问题，项目已经做过一些明确动作。  
比如：

- 使用 `optimizePackageImports`
- 尽量避免内部 monorepo 包被错误优化导致 HMR 和依赖膨胀问题
- 通过 dynamic / conditional load 延后重型模块
- Sentry server config 里甚至有“移除 Pino integration 以减少 bundle 大小”的说明，

第三，团队已经意识到“开发慢”和“线上 bundle 大”不是一回事。  
这点在 [docs/LOCAL_DEV_PERFORMANCE_PLAN.md](/Users/lidonghai/castlery/joyboy/docs/LOCAL_DEV_PERFORMANCE_PLAN.md:483) 写得很清楚：不能把所有问题都归因到 bundle，要区分开发期重编译和线上首屏负载。

**它帮助我们识别首屏不该加载的第三方依赖、过重的 barrel import、以及一些应该做 dynamic import 的 client 模块。它不只是看总大小，更重要是看哪些依赖进入了错误的 chunk。**

**4. 搜索双层缓存是怎么做的**

这是项目里最清楚、也最适合拿来讲的性能优化 case。

它不是简单做一个 `Map`，而是做了两层缓存：

**第一层：结果缓存**
作用是缓存“已经成功返回的搜索结果”。

配置大概是：

- `RESULT_CACHE_TTL = 5 分钟`
- `MAX_RESULT_CACHE_SIZE = 50`

命中逻辑在这里：  
[search-view-client.tsx](/Users/lidonghai/castlery/joyboy/libs/modules/search/components/src/lib/search-view/search-view-client.tsx:564)

如果相同请求 key 已经有结果，并且没过期，就直接返回缓存数据，不再重新打接口。

**第二层：进行中请求缓存**
作用是请求去重，也就是避免同一时刻对同一个查询发多次请求。

配置大概是：

- `MAX_PENDING_REQUESTS = 10`

逻辑在这里：  
[search-view-client.tsx](/Users/lidonghai/castlery/joyboy/libs/modules/search/components/src/lib/search-view/search-view-client.tsx:593)

如果某个 request key 对应的请求已经在飞，就直接复用同一个 Promise，而不是再发一次。

**所以这两层分别解决的是两个不同问题：**

- 结果缓存：解决“重复看同一个搜索结果”
- 请求去重：解决“同一时刻重复发同一个请求”

这个设计非常合理，因为很多人只做第一层，不做第二层，结果快速切换筛选时还是会打爆请求。

---

**5. 200ms → <5ms 的统计口径是什么**

这个口径你要讲严谨一点，不然面试官容易追问。

它不是说“所有搜索都从 200ms 到 5ms”，而是：

**在缓存命中的场景下，原本需要走一次网络请求，大约要 200ms；现在直接从客户端结果缓存返回，耗时降到小于 5ms。**

这个场景通常包括：

- 用户从 PDP 返回搜索结果页
- 用户切回之前用过的筛选组合
- 重复搜索同一个 query

也就是说，这是**命中缓存后的交互耗时对比**，不是后端接口性能本身提升了 40 倍。

文档里就是这样描述的，见  
[docs/search-result-cache.md](/Users/lidonghai/castlery/joyboy/docs/search-result-cache.md:8)

所以面试里最好说：

**这个数字的统计口径是“客户端已命中缓存时的返回耗时”，不是所有请求的平均耗时。**

---

**6. 30% 请求减少是怎么算的**

这个也不能说得太虚。

更合理的理解是：

**把原来用户在重复筛选、返回页面、重复搜索时本来会重新发出的请求，变成了缓存命中或进行中请求复用，所以总请求数下降。**

典型会减少请求的场景：

- 搜索结果页返回后再次展示
- 同一个筛选条件来回切换
- 用户短时间内触发相同搜索
- 同时发出的相同请求被 pending cache 合并

所以这个“30%”本质上是：

**优化前后的同类搜索 API 请求总量对比**  
不是单次请求变少，而是用户在一个典型浏览流程里，总共少发了约三成请求。

文档里有这个结论，见  
[docs/search-result-cache.md](/Users/lidonghai/castlery/joyboy/docs/search-result-cache.md:10)

**7. 这种缓存有没有副作用**

有，而且这是你面试里讲出来会很加分的地方。

### 第一类副作用：缓存一致性

最直接的问题就是，缓存不可能永远是最新的。

比如：

- 商品库存变化
- 价格变化
- 地区变化
- 用户登录态变化
- 地址变化影响可售结果

项目里的解决方式不是“完全避免过期”，而是做**有限时间缓存 + 手动失效**。

比如文档里明确写了这些场景要手动清缓存：

- 登录 / 登出
- 国家切换
- 语言切换
- 货币变化
- 配送地址变化

见 [docs/search-result-cache.md](/Users/lidonghai/castlery/joyboy/docs/search-result-cache.md:56)

### 第二类副作用：内存占用

客户端缓存如果不控大小，会越来越大。

项目里专门做了限制：

- 结果缓存最多 50 条
- pending request 最多 10 条
- 过期自动清理
- 超限淘汰

实现见：

- [search-view-client.tsx:95](/Users/lidonghai/castlery/joyboy/libs/modules/search/components/src/lib/search-view/search-view-client.tsx:95)
- [search-view-client.tsx:198](/Users/lidonghai/castlery/joyboy/libs/modules/search/components/src/lib/search-view/search-view-client.tsx:198)
- [search-view-client.tsx:220](/Users/lidonghai/castlery/joyboy/libs/modules/search/components/src/lib/search-view/search-view-client.tsx:220)

### 第三类副作用：错误重试 / 风暴请求

如果搜索请求一直失败，又没有保护，可能会反复打接口。

项目里还做了一个小型熔断器：

- 连续错误累计
- 达到阈值后打开 circuit breaker
- 防止无限重试

这块实现也在 `search-view-client.tsx` 里，见  
[search-view-client.tsx:131](/Users/lidonghai/castlery/joyboy/libs/modules/search/components/src/lib/search-view/search-view-client.tsx:131)

这说明团队并不是只考虑“快”，也考虑了“出问题时不要更糟”。

**8. 面试里怎么回答**

你可以直接用这一版：

“我们项目里的性能优化不是只做 Lighthouse 分，而是按实际业务链路拆开看。首先会结合 RUM 和页面体验看慢点在哪，比如首屏、交互、搜索切换；然后会用 bundle analyzer 看哪些依赖进了不该进的 chunk；最后针对高频交互场景做专项优化。

比较典型的 case 是搜索页。我们做了双层缓存：第一层是结果缓存，用来复用已经成功返回的搜索结果；第二层是进行中请求缓存，用来合并同一时刻的重复请求。这样用户在前进后退、重复筛选、切回旧 query 的时候，不需要重新打接口。项目文档里的统计口径是：缓存命中的场景下，返回耗时从大约 200ms 降到 5ms 以内，同时整体搜索请求量下降了大约 30%。

当然这种方案也有副作用，主要是缓存一致性和内存占用问题。所以我们做了 5 分钟 TTL、最大缓存数量限制、过期清理，以及在登录态、国家、语言、地址变化时主动清缓存。另外在错误场景下还加了 circuit breaker，防止搜索接口异常时不断重试，把问题放大。”
