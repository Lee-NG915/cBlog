## Concurrent Rendering 是什么？它解决什么问题？是自动启用的吗？

Concurrent Rendering 是 React 18 引入的一种新的渲染模型，它的核心不是“并行渲染”，而是让渲染变成可中断、可恢复、可按优先级调度。

它主要解决的问题是：

以前 React 更新一旦开始，往往会同步占住主线程直到整次渲染完成，复杂页面里很容易导致输入卡顿、点击延迟、列表更新不流畅。
有了 Concurrent Rendering 之后，React 可以优先处理高优先级任务，比如用户输入或点击，把低优先级更新延后，所以交互体验会更平滑。
它也是 startTransition、useDeferredValue、Suspense、Streaming 这些能力的底层基础。

如果放到业务场景里，最典型的就是搜索页这种“大列表更新 + 高频交互”的页面：输入框更新应该优先，结果列表重渲染可以稍微延后，这样用户感知会明显更顺。

框架层如果用了 App Router、Suspense、streaming，这类收益会自然吃到一部分；但像 startTransition、useDeferredValue 这种交互优先级优化，如果业务代码没有显式使用，是不会自动帮你把重更新降优先级的。

我们项目中已经吃到了 React 18 在 Suspense + RSC + streaming 这部分的收益，但不是所有并发渲染优化都自动打开、更不是所有页面都会自动变快。

## useTransition 和 useDeferredValue 的区别

useTransition：你主动告诉 React，这次 state 更新优先级低一点
useDeferredValue：你告诉 React，这个值可以晚一点跟上，不用立刻同步

useTransition 延后的是更新操作，useDeferredValue 延后的是值。

## React Server Components 和普通组件有什么边界？

React Server Components 运行在服务端,普通组件 通常指 Client Component，运行在浏览器

Server Component 不能用 useState、useEffect、不能绑定事件、不能访问 window/document；Client Component 才能处理这些交互逻辑。
Server Component 可以渲染 Client Component，但一旦某个高层模块加了 'use client'，下面整棵树都会更偏客户端，所以边界要尽量压低。

## 页面加载整体时序

HTML → CSS/JS 并行下载 → 首屏绘制 → JS hydration → 客户端请求数据 → 局部更新

在 Next.js 这种 SSR / RSC 场景里，服务端会先返回 HTML，所以页面结构和一部分内容其实先出来了；浏览器拿到 HTML 后开始并行拉 CSS 和当前路由需要的 JS chunk。CSS 先参与首屏绘制，页面会先显示出来；等 JS chunk 执行后，React 再做 hydration，把这份 HTML 接管成可交互组件，最后客户端组件再发自己的数据请求，做局部更新。

所以整体时序我一般会概括成：
服务端返回 HTML → 浏览器并行加载 CSS 和 JS chunk → 首屏绘制 → React hydration → 客户端请求数据 → 页面局部更新。
如果放到你现在看的 account/orders 页，就是：
layout 壳先 SSR 出来
account/orders 对应 chunk 下载
account layout 和 orders page 先后 hydrate
然后 orders 页里的 query 再去拉订单数据，最后把列表渲染出来。

## 结合页面加载时序，useEffect 的执行时机是什么？依赖数组怎么判断？

useEffect 一定是客户端时机，不会在服务端执行，所以它一定是在客户端组件、水合、挂载加载完之后运行

如果结合 SSR / hydration 的页面流程来说，就是： 服务端先返回 HTML，浏览器下载 JS，React 完成 hydration 之后，useEffect 才开始跑。

依赖数组的判断方式不是深比较，而是 React 对数组里的每一项做 Object.is 的浅比较（只比较引用，不会深入比较引用的值）。

所以：不写依赖数组：每次渲染后都执行

[]：首次挂载后执行一次
[deps]：首次挂载执行一次，之后只有依赖项相对上一次发生变化才重新执行
最容易踩坑的是对象、数组、函数这类引用值，因为即使内容一样，只要引用变了，effect 也会重新触发。
所以在真实项目里，判断 effect 会不会重跑，关键不是“看起来一样”，而是“这一项的引用和上一次是不是同一个”。

## 如何处理组件卸载后的异步请求？

这里是说当前组件已经被卸载了，但是某些异步操作还待完成，比如useEffect里的定时器还未完成、fetch请求还未返回、事件监听器未清楚等等，就有可能出现内存泄漏、请求返回的数据过期这些未知副作用

第一层是清理副作用本身，像定时器、事件监听、observer、subscription 这些，一定放到 useEffect cleanup 里释放。
第二层是对网络请求优先做真正取消，最常见就是 AbortController，组件卸载时直接 abort()，这样可以避免请求回来后再做无意义更新。
第三层是处理竞态，因为很多时候问题不只是组件卸载，而是旧请求晚于新请求返回，所以我会配合 requestId 或 ignore flag，只接受最后一次请求结果。

能取消就取消，不能取消就忽略旧结果，同时一定处理竞态，不只是处理 unmount。

## useMemo / useCallback 什么时候有用，什么时候反而没必要？

昂贵的计算 (useMemo)： 当进行大规模数据过滤、排序、复杂数学运算时 。例如，执行一个 O(n²) 的复杂逻辑，使用缓存可避免每次状态更新时的重复计算 。

保持引用稳定 (useCallback)： 当函数需要作为依赖项传递给 useEffect，或作为 props 传递给使用 React.memo 包裹的子组件时。这能防止因函数引用地址改变而引发的连锁渲染。

大型组件树 (useCallback)： 父组件包含庞大的子组件树，缓存函数能确保子组件不会因父组件的无关状态变化而盲目重新渲染。

## React.memo 失效的常见原因有哪些？最佳使用场景是什么

`React.memo` 最常见的失效原因，其实是 props 虽然值没变，但引用变了，比如父组件每次都传新的对象、数组、函数，这种浅比较是过不去的。还有两类情况它本来就拦不住，一个是组件自己的 state 更新，一个是它依赖的 context 更新。

我一般会把它用在列表 item、卡片、纯展示组件这类场景，前提是 props 比较稳定，而且组件本身渲染成本值得优化。如果只是普通小组件，或者父层传参本身就不稳定，那包 `React.memo` 往往收益不大，甚至会让代码更复杂。

## 复杂表单如何设计状态？如何避免整页重渲染？

复杂表单我一般会把状态拆成四类：字段值、校验状态、派生状态、业务流程状态。

字段值通常交给 react-hook-form 这类表单库管理，校验用 schema，像可提交、价格汇总这类尽量做派生，不再额外存一份，提交中、步骤流转、接口报错这类再单独管理。

避免整页重渲染的核心是局部订阅。也就是不要在页面根组件 watch 整个表单，而是让每个字段区块只订阅自己关心的值；表单按模块拆开，能用非受控就不用全受控，重计算和异步联动也尽量放在局部处理。这样一个字段变化，影响的只是对应区块，不会把整个页面都带着重刷。

## 你封装过哪些 hooks？怎么保证可测试性？

我封装过的 hooks 主要分四类：
交互类，比如 useInfiniteScroll、useDebounce；
业务编排类，比如 checkout step、促销可用性判断这类；
环境能力类，比如 useBreakpoints、useFeatureFlag；
还有数据聚合类，把搜索页的筛选、分页、loading 这些状态统一收口。

可测试性上，我会控制 hook 不要过重。一般是让 hook 负责编排，把状态流转和判断逻辑拆成纯函数，把 window、router、observer、埋点、请求这些外部依赖隔离出去。这样测试时一部分直接测纯函数，另一部分用 renderHook 测行为，比如状态切换、异步完成、卸载 cleanup，而不是死盯内部实现。这样 hook 才不会越封越难维护。

## RSC 和 client component 的边界

RSC 和 Client Component 的边界，核心是：Server Component 负责拿数据和生成静态 UI，Client Component 负责交互和浏览器能力

**Server Component 能做什么**

- 可以直接在服务端 `await` 数据，比如查 API、读 CMS、读配置。
- 不会打进客户端 JS bundle，适合大块静态内容、SEO 内容、商品详情、CMS 区块。
- 可以引入 Server Component，也可以引入 Client Component。
- 不能用 `useState`、`useEffect`、事件监听、浏览器 API。

**Client Component 能做什么**

- 文件顶部加 `'use client'`。
- 可以用 `useState`、`useEffect`、`onClick`、`window`、`localStorage`、Redux hook、表单交互。
- 会进入客户端 bundle，需要 hydration。
- 可以接收 Server Component 传下来的可序列化 props。
- 不能直接 import Server Component，也不能直接访问服务端资源。

**边界怎么划**
一般是外层 Server、内层 Client

项目里像 `page.tsx`、`layout.tsx` 常常作为 Server Component 负责拿初始数据和拼页面；真正有交互的 `page.client.tsx`、`layout.client.tsx`、按钮、表单、弹窗、购物车操作才放到 Client Component。
