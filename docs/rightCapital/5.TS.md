## interface 和 type 的区别？

interface 更适合描述对象结构，尤其是对外暴露的契约，比如 props、API response、class 实现，它支持声明合并，扩展性更强。
type 更通用，除了对象，还可以定义联合类型、交叉类型、字面量类型、元组、函数类型这些，表达能力更强。

实际使用上我的习惯是：
对象 shape、可扩展公共契约：优先 interface
联合类型、工具类型组合、复杂类型运算：优先 type

## 泛型解决了什么问题？项目里怎么用？

泛型本质上解决的是 类型复用和类型约束 的问题。

如果不用泛型，通常只有两种结果：

- 要么你把类型写死，复用性很差
- 要么你写成 `any`，灵活了，但类型安全没了

所以我一般会说，泛型的价值有三点：

- **让一套逻辑适配多种数据结构**
- **在复用的同时保留类型推导**
- **把输入类型和输出类型关联起来**

项目里常见用法我会分几类答：

- **接口请求层**
  - 比如 `ApiResponse<T>`、`PaginationResult<T>`，同一个响应壳子可以套商品、订单、用户不同数据
- **工具函数**
  - 比如 `pick<T, K extends keyof T>`、`merge<T>`，保证传什么类型，返回结果还能跟着推出来
- **组件封装**
  - 比如表格、Select、FormField 这类通用组件，用泛型约束 item 或 option 的结构，避免组件只能服务一种数据
- **Hooks**
  - 比如 `useForm<FormValues>()`、自定义 `useRequest<TData, TParams>()`，让 hook 返回值和入参都有明确约束
- **状态管理 / RTK Query**
  - 像 `builder.query<Result, Params>` 这种就是很典型的泛型场景，接口返回值和请求参数都能在调用链里保持一致

## Pick / Omit / Partial / Required / Record 的实现原理？

这些工具类型本质上都是基于 **`keyof` + mapped types + 索引访问** 做出来的，核心思想就是：**先拿到 key，再按 key 去重新构造一个新类型**。

- `Partial<T>`
  - 原理是遍历 `T` 的所有 key，然后把每个属性变成可选
  - 类似：

  ```ts
  type Partial<T> = {
    [K in keyof T]?: T[K];
  };
  ```

- `Required<T>`
  - 跟 `Partial` 相反，把所有可选属性变成必填
  - `-?` 表示移除可选标记

  ```ts
  type Required<T> = {
    [K in keyof T]-?: T[K];
  };
  ```

- `Pick<T, K>`
  - 从 `T` 里挑一部分 key 组成新类型
  - `K` 一般约束为 `keyof T`

  ```ts
  type Pick<T, K extends keyof T> = {
    [P in K]: T[P];
  };
  ```

- `Omit<T, K>`
  - 本质不是直接“删”，而是先算出“剩下的 key”，再 `Pick`
  - 通常结合 `Exclude`

  ```ts
  type Omit<T, K extends keyof any> = Pick<T, Exclude<keyof T, K>>;
  ```

- `Record<K, T>`
  - 原理是用一组 key 去构造一个对象类型，并且每个 key 的 value 类型都一样

  ```ts
  type Record<K extends keyof any, T> = {
    [P in K]: T;
  };
  ```

- 它们的价值在于 **不重复写类型**，还能让类型跟着原始结构自动演进，尤其适合 DTO、表单、接口入参、组件 props 裁剪这类场景。

## 如何给组件库组件设计 Props 类型？

我一般会从这几个原则来设计组件库的 Props 类型：

- **先定义稳定的核心能力**
  - 先想清楚这个组件对外真正需要暴露什么，比如 `value`、`onChange`、`disabled`、`size`、`variant`
  - 不把内部实现细节直接暴露出去

- **区分通用 props 和组件私有 props**
  - 通用能力比如 `className`、`style`、`children`
  - 组件特有能力单独定义，比如 `options`、`loading`、`placement`

- **优先复用原生元素类型**
  - 比如按钮我通常会基于 `React.ButtonHTMLAttributes<HTMLButtonElement>` 或 `ComponentProps<'button'>`
  - 这样原生属性不会漏，也不用手写一遍

- **通过联合类型约束互斥关系**
  - 例如 `href` 和 `onClick`、`single` 和 `multiple` 这种场景，可以用联合类型把非法组合限制掉
  - 这样比只靠文档约束更稳

- **用泛型提升复用性**
  - 比如下拉、表格、表单项这类组件，`Option<T>`、`SelectProps<T>` 这种方式可以保留业务数据类型

- **控制扩展性，不做过宽的 any**
  - 组件库最怕一开始图省事全写松，后面就很难收口
  - 我会尽量让 props 既能扩展，又有明确边界

如果面试官追问一句“总结一下”，我会答：

- 组件库 Props 设计的核心就是 **对外 API 稳定、类型约束清晰、原生能力可继承、复杂场景可扩展**。
