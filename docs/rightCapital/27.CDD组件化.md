这一块在你们项目里本质上不是“做了一套组件样式”，而是搭了一条完整的设计系统交付链路：`Token -> Theme -> Component API -> Storybook -> Chromatic -> PR 校验`。这样做的目标，是让品牌一致性、组件复用和回归验证都能工程化，而不是靠人肉对页面。

1. `Token` 分层模型

你们现在更像是 3 层 Token。

第一层是原始 Token，也就是最底层的设计值，比如颜色板、字号、字重、间距、圆角这些。这里会有品牌色、灰阶、字体体系，V1/V2 也各有一套，说明它不是写死在组件里的，而是先沉到底层。

第二层是语义 Token。比如 `primary / neutral / danger`、`text.primary / text.secondary`、`background.surface / level1 / level2` 这种。它的意义不是“这个颜色值是多少”，而是“这个值在界面里扮演什么角色”。这样以后品牌升级时，改的是语义映射，不是全项目搜颜色。

第三层是组件 Token。也就是把语义 Token 再落到具体组件状态上，比如按钮的 `outlinedBorder`、`plainHoverBg`、Chip 的 padding、Icon 的尺寸变量。这一步很关键，因为真正稳定复用的不是单个颜色，而是“某个组件在某种状态下的表现规则”。

你面试里可以说：我们不是直接在组件里写样式，而是先有基础 Token，再抽语义 Token，最后落到组件级 Token，这样主题切换和品牌升级成本会低很多。

1. Storybook 的 CDD 流程

你们这套很典型是 CDD，也就是先把组件当成独立产品单元来开发，而不是永远嵌在业务页面里调。

实际流程我理解是这样：先定义组件 API 和状态，再补 stories，把不同主题、不同尺寸、不同交互态都铺开。Storybook 里不仅有展示，还有 controls、autodocs、design addon，甚至能挂 Figma 链接，所以它不只是“看效果”，而是设计、开发、测试共享的组件说明书。

比较真实的一点是，你们不是只测静态展示，Storybook 里还接了 test runner 和截图快照，所以一个 story 本身就能承载文档、交互测试、视觉回归三件事。这个比“写完组件再去页面里试”成熟很多。

面试里你可以说：我们做组件时会先在 Storybook 里把 API、状态、主题和交互跑通，再集成到业务页面，这样能把问题前置到组件层解决。

1. Chromatic 怎么接入 PR

你们是把 Chromatic 直接接进了 GitHub PR 流程里，而且是单独给 Fortress 这条链路做了工作流。PR opened、reopened、synchronize 的时候都会触发，只要 Fortress 相关内容有变化，就会跑 Chromatic。

这里有个很值得讲的细节：配置上虽然支持 `onlyChanged`，但实际 workflow 里把它关掉了，而且还显式关了 TurboSnap 相关优化。这个说明团队在稳定性和速度之间做了取舍，最后选的是“慢一点，但视觉回归结果更可信”，尤其是在 Monorepo 和 Storybook 8 这种场景里，这个决定很真实。

还有一个工程细节是，Storybook 预览里有主题模式、viewport 和截图延迟控制，test runner 里还会等页面资源稳定后再截图。也就是说，你们不是“接了 Chromatic 就完了”，而是针对误报、抖动、资源未加载完这些典型坑做了处理。

面试里可以说：Chromatic 不是装个插件，而是作为 PR 门禁的一部分，保证组件改动在合并前就能看到视觉差异。

1. `Joy UI -> Tailwind` 的 ADR 取舍

从现在工程现状看，你们最终没有往 Tailwind 迁，核心原因不是 Tailwind 不好，而是当前设计系统已经深度建立在 Joy UI 的主题能力上了。

Joy UI 这套对你们的价值很明确：有 `CssVarsProvider`、主题扩展、语义色板、组件级 override、`sx` 响应式能力，和现有 Token 体系是紧耦合的。尤其你们已经有 V1/V2 主题、组件状态变量、slot 扩展，这些如果迁到 Tailwind，不是“换个写法”，而是要把整套设计语言重新编码一遍。

Tailwind 的优点是更轻、更直接、RSC 时代也更自然，因为大部分样式在构建期就展开了，运行时成本小；但它的代价是，如果你们已经有成熟的主题系统和大量组件 override，迁移成本会非常高，而且会破坏现有回归基线。

所以更像一个务实 ADR：如果从零开始做偏内容站、营销站，Tailwind 可能是好选择；但对于你们这种已有组件库、主题体系、品牌升级历史包袱的项目，继续用 Joy UI 包一层 Fortress 更合理。

1. RSC 兼容问题

这块是个很好的面试加分点，因为它不是“能不能跑”，而是“怎么优雅地跑”。

你们现在是 Next.js App Router 场景，Joy UI 又依赖 Emotion/CSS-in-JS，所以最大的坑就是 SSR/RSC 下样式注入时机。如果处理不好，会出现样式闪烁、服务端和客户端 className 不一致、hydration warning 这些问题。

你们的处理思路很典型，也比较标准：用 Theme Registry 管 Emotion cache，再借助 `useServerInsertedHTML` 在服务端把样式提前塞进去。这样页面虽然是 Server First，但样式系统不会在 hydration 后才补上。

不过这里也要诚实一点说，Joy UI 这类方案天然没有 Tailwind 那么“RSC 纯粹”。所以你们实际上走的是一种折中路线：页面尽量保持服务端渲染，交互组件和主题提供层接受一定 client boundary。这个回答很真实，面试官一般会认可。

1. 组件 API 设计原则

你们这套组件 API 我觉得有几个明显原则。

第一是语义优先，不让业务方直接操作底层实现细节。比如业务传的是 `primary / secondary / tertiary` 这类语义，而内部再映射到 Joy 的 `solid / outlined / plain`。这样业务代码更稳定，底层库以后换也更可控。

第二是可扩展，但不失控。你们保留了 `sx`、`slotProps`、`component/as` 这种扩展口，说明不是做死板组件；但又没有把所有底层能力原样暴露出去，而是包了一层自己的约束。

第三是主题驱动，而不是局部 hardcode。很多尺寸、颜色、icon 大小、spacing 都通过 theme 或 CSS var 落下去，这样组件不会随着业务迭代越写越散。

第四是兼容演进。Storybook 里能同时切 V1/V2 theme，这说明组件 API 在设计时考虑了版本迁移，不是每次升级都推倒重来。

1. 面试里怎么答

如果面试官问这块，你可以用这个版本回答：

“我们组件库这块本质上是 Fortress 这一层，把 Joy UI 作为底座做了品牌化封装。设计上我们先做 Token 分层，先有颜色、字体、间距这些基础 Token，再抽成语义 Token，最后落到按钮、表单、Badge 这类组件级 Token，所以品牌升级时不会全项目改样式。开发流程上我们走的是 Storybook 的 CDD，组件先在 Storybook 里把状态、主题和交互跑通，再接业务页面，同时接了 Chromatic 做 PR 级别的视觉回归。为什么没有转 Tailwind，主要是因为现有主题系统、组件 override 和视觉基线已经建立在 Joy UI 上了，迁移成本会很高。RSC 兼容上，最大的坑是 Emotion 的服务端样式注入，所以我们通过 Theme Registry 和服务端样式插入去解决，整体是 server-first，但接受必要的 client boundary。”
