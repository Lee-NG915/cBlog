## Next 14有什么特性？Next.js App Router 和 Pages Router 的差异是什么？

**解释版本**

### `Next.js 14` 比较关键的点有这几个：

- `Server Actions` 稳定了  
  以前做表单提交、mutation，通常要自己写 API Route 再 `fetch`。  
  到 14 之后，`Server Actions` 成为稳定能力，可以直接在组件里声明 server action，而且和 `revalidatePath`、`revalidateTag`、`redirect`、`cookies` 这些能力整合得更紧。  
  这会明显改变你写数据变更逻辑的方式。
- 在开发体验上继续推进  
  官方在 `Next.js 14` 的发布里重点提了 dev 性能，比如本地启动和 Fast Refresh 更快。
- `Partial Prerendering` 进入 preview  
  这个能力的目标可以简单理解成：先给用户一个很快的静态 shell，再把动态部分通过 streaming 补进来。  
  它本质上是在 App Router + Suspense + Streaming 的模型上继续往前走。
- Metadata 能力做了调整  
  比如 `viewport`、`themeColor` 这类元信息的处理方式更明确了，一部分老写法开始被替代。

### 再说 `App Router` 和 `Pages Router` 的差异。

我会把它理解成：**Pages Router 是传统 Next.js 模型，App Router 是基于 React Server Components 的新架构模型。**

核心差异主要有这几类：

1. 路由和布局模型不同

- `Pages Router` 以 `pages/` 为核心，一个文件对应一个路由，布局通常靠 `_app.tsx`、`_document.tsx` 或页面级手动包裹。
- `App Router` 以 `app/` 为核心，天然支持 `layout.tsx`、嵌套路由、route groups、parallel routes、intercepting routes。  
  也就是说，App Router 的布局和路由组织能力更强，更适合复杂应用。

1. 渲染模型不同

- `Pages Router` 更偏传统：`SSR / SSG / ISR / CSR`，主要依赖 `getServerSideProps`、`getStaticProps`、`getStaticPaths`。
- `App Router` 默认就是 `Server Components`，而且 Next 会根据你用了什么 API、数据是否缓存，自动在 static、dynamic、streaming 之间选择策略。  
  这个思路比 Pages Router 更“数据和组件一体化”。

1. 数据获取方式不同

- `Pages Router` 里数据获取通常是 page 级别的。
- `App Router` 里数据获取可以直接放在 Server Component 里，离 UI 更近，也更容易按组件边界拆分。  
  再加上缓存和 revalidate，粒度会更细。

1. mutation 方式不同

- `Pages Router` 常见模式是：前端 -> API Route -> 后端逻辑。
- `App Router` 则可以直接用 `Server Actions`，减少很多样板代码。

1. 客户端 bundle 控制不同

- `Pages Router` 默认页面就是客户端 React 组件思维。
- `App Router` 默认服务端组件，只有交互部分才显式加 `'use client'`。  
  这样通常更容易减小 client bundle，但也要求你更清楚地划分 server/client 边界。

1. 迁移成本和适用场景不同

- `Pages Router` 更成熟、心智负担更低，老项目很多还是它。
- `App Router` 更现代，也更适合新项目，但团队需要适应 `Server Components`、缓存、streaming、server/client 边界这些新心智。  
  官方也明确说了：新应用推荐 App Router，老应用可以增量迁移，而且两个 router 可以共存。

所以如果把这题真正讲透，核心就一句话：

**Next.js 14 的重点不是多了几个 API，而是把 Next.js 从“传统 page-based SSR framework”进一步推进成了“基于 Server Components、Streaming、Server Actions 的全栈 React 框架”。**

---

**面试回答版本**

“如果按 Next.js 14 的关键特性来看，我会重点提四个点：第一是 `Server Actions` 在 14 里稳定了，表单提交和数据 mutation 不一定再需要单独写 API Route；第二是 `Turbopack` 继续推进，开发阶段的启动和热更新性能更好；第三是 `Partial Prerendering` 进入 preview，本质上是在 App Router 和 Streaming 模型上进一步优化首屏体验；第四是一些 metadata 和构建层面的能力做了升级和调整。

至于 App Router 和 Pages Router 的差异，我会把它理解成两代架构模型。Pages Router 是传统的 `pages/` 路由体系，主要围绕 `getServerSideProps`、`getStaticProps` 这些 page 级数据获取方式来组织。App Router 则是基于 `app/` 目录和 React Server Components，默认服务端渲染组件，天然支持嵌套 layout、streaming、server actions、以及更细粒度的数据获取和缓存控制。

所以简单说，Pages Router 更像传统 Next.js，适合存量项目；App Router 更像 Next.js 的新一代架构，尤其适合新项目和复杂应用。”

## 什么是server action

**解释版本**

`Server Action` 可以简单理解成：**把原来你需要通过 API Route 才能完成的服务端写操作，直接以内联函数的方式挂到 React / Next.js 组件树里。**

以前在 Next 里，如果你要做一个“提交表单、更新数据库、下单、修改用户资料”这样的动作，常见链路是：

- 前端组件触发提交
- `fetch('/api/xxx')`
- API Route 接收请求
- 服务端执行业务逻辑
- 前端再根据结果刷新 UI

`Server Action` 的思路是把这条链路收短。  
你可以在服务端定义一个带 `'use server'` 的函数，然后在表单提交或交互里直接调用它。  
它还是在服务端执行，只是调用方式更像“直接调函数”，而不是你手写一个 REST 接口再去请求它。

所以它的本质不是“前端能直接操作数据库”，而是：

- 框架帮你把调用封装了
- 让 mutation 更贴近组件和页面
- 减少很多样板 API 代码

一个很典型的理解方式是：

- `Server Component` 解决“服务端读数据”
- `Server Action` 解决“服务端改数据”

它比较适合这些场景：

- 表单提交
- 新增 / 编辑 / 删除
- checkout 这类写操作
- 提交后顺手做 `revalidatePath`、`revalidateTag`
- 提交成功后 `redirect`

**Server Action = Next.js 在 App Router 时代提供的“服务端 mutation 函数”，用来替代一部分传统 API Route 表单提交流程。**

---

**面试回答版本**

“Server Action 可以理解成 Next.js App Router 里的一种服务端 mutation 能力。以前我们做表单提交或者写操作，通常是前端先调用 API Route，再由服务端处理；Server Action 则是把这层调用收短成一个带 `'use server'` 的服务端函数，让组件可以更自然地触发服务端写逻辑。

它的核心价值是减少样板代码，并且和 `revalidatePath`、`redirect` 这类能力整合得更好，所以很适合表单提交、资料更新、下单这类场景。但我不会把它理解成可以替代所有 API，因为如果涉及开放接口、跨端复用、标准 BFF(Backend-for-Frontend 中间层) 边界，API Route 还是有必要保留。”

## 我不理解你说的API Route，提交表单不是直接提交到后端服务吗，为什么还需要先调用前端的服务，提交之后也可以直接在客户端组件展示提交的结果，这里哪些场景需要用到前端项目的服务端？

**解释版本**

你这个问题问得对，我刚才那种说法如果没先把架构层次讲清楚，确实容易让人误会。

关键点先说清楚：

**表单当然可以直接提交到后端服务，不是所有场景都需要先走前端项目自己的服务端。**

真正要区分的是，一个 Next.js 项目里通常可能同时存在三层：

- 浏览器里的客户端组件
- 前端项目自己的服务端，比如 `Next.js server`、`API Route`、`Server Action`
- 真正的后端业务服务，比如 Java / Node / Go 的订单、用户、支付服务

所以你可以把问题理解成：

**客户端为什么有时不直接调用后端，而要先经过 Next.js 自己这一层服务端？**

答案是：**不是必须，而是某些场景这样更合适。**

常见有这几类。

1. 需要保护服务端密钥或内部接口
   比如：

- 调第三方支付
- 调 CMS 私有 token
- 调内部订单服务的私有 endpoint
- 生成 S3 上传签名

这些东西不能放到浏览器里，所以客户端不能直连，只能先打到 Next 服务端，再由它去调真实后端或第三方。

1. 需要做 BFF(Backend-for-Frontend 中间层) 聚合
   有些页面不是只调一个接口，而是要把：

- 用户信息
- 购物车
- 优惠券
- 推荐商品
- 库存状态

拼成一个前端更好用的结果。

这时候如果全让客户端自己调，会有很多请求、很多编排逻辑、很多错误处理。  
更常见的做法是让 Next 这一层先聚合，再把结果给页面。  
这就是典型的 BFF(Backend-for-Frontend 中间层) 思路。

1. 需要基于 cookie / session 做安全处理
   很多时候鉴权信息不希望暴露给浏览器 JS，比如：

- `httpOnly cookie`
- server session
- 内部 token exchange

这时 Next 服务端更容易安全地读取 cookie、校验身份，再代替客户端去调用后端。

1. 需要服务端渲染或首屏就拿到结果
   如果是 App Router 场景，页面希望在服务端就把数据准备好，而不是浏览器加载后再慢慢请求，那就会在服务端拿数据。  
    这和传统客户端提交表单不完全一样，但本质也是“前端项目自己的服务端”在参与业务流程。
2. 需要统一缓存、重刷和数据一致性控制
   比如一个提交动作之后，你希望：

- 更新数据库
- 立刻刷新 `/cart`
- 刷新某个 tag cache
- redirect 到成功页

这种在 App Router + Server Action 下会很顺，因为服务端 mutation 和 revalidate 是连在一起的。

---

但反过来说，**哪些场景不一定需要前端项目自己的服务端？**

也有很多。

1. 纯公开读接口
   比如商品列表、公开文章、公开搜索，客户端直接请求后端就可以。
2. 你本身就有成熟的后端 BFF(Backend-for-Frontend 中间层)
   如果公司已经有专门给前端用的后端层，那 Next 不一定还要再包一层。
3. 简单表单、简单交互、没有敏感凭据
   比如一个普通留言接口，如果安全模型允许，浏览器直接提交到后端完全没问题。

---

所以真正准确的理解应该是：

**API Route 或 Server Action 不是“因为表单不能直接提交后端才存在”，而是因为前端项目有时也需要一层自己的服务端能力，用来做安全、聚合、鉴权、缓存和渲染协作。**

**面试回答版本**

“表单并不是一定要先提交到前端项目自己的服务端，很多场景下客户端可以直接调用后端。关键不是‘能不能直连’，而是前端项目这一层服务端是否有额外价值。

我一般会在几种场景下使用 Next.js 的服务端能力，比如 API Route 或 Server Action：第一是需要保护服务端密钥或调用内部接口；第二是需要做 BFF(Backend-for-Frontend 中间层) 聚合，把多个后端结果整理成更适合前端消费的数据；第三是要基于 cookie、session 做安全鉴权；第四是需要和 SSR、缓存刷新、revalidate 这些能力协同。

如果只是简单公开接口或者公司已经有成熟 BFF，客户端直接请求后端完全可以。所以我不会把 API Route 或 Server Action 当成必须层，而是把它看成前端项目在安全、聚合和服务端渲染场景下的一层服务端能力。”

## 你们的Server Actions 用在哪里？有什么安全风险

我们项目里 Server Actions 主要不是拿来替代所有接口，而是集中用在几类比较适合做 BFF 边界的场景。
第一类是支付编排，这是最核心的，比如 initiatePaymentAction 和 capturePaymentAction，它们负责把前端支付 UI 和后端支付服务、provider strategy 串起来。
第二类是CMS 和 DY 这类第三方代理，比如 Storyblok 内容获取、DY campaigns 获取，这样可以避免把服务端 token 和调用细节暴露到浏览器。
第三类是一些服务端取数封装，比如 review 和 product 的 server-side action，用来适配 App Router/RSC 的调用方式。
安全风险我会重点看四个。
第一是信任客户端传参过多，尤其支付 action 里像 orderId、amount、paymentConfig 这类字段，如果服务端不二次校验，就有篡改风险。
第二是权限绕过，因为 Server Action 本质上也是服务端入口，不能只靠前端页面流程保护，必须做登录态和资源归属校验。
第三是重放和重复提交，特别是支付 initiate/capture，要做幂等，不然会有重复调用风险。
第四是数据泄漏和代理滥用，像 CMS/DY 这种 action 如果做成通用代理，可能暴露敏感信息，甚至变成 SSRF 入口。

所以我对 Server Action 的理解是：
它适合做服务端编排和敏感能力收口，但它不是天然安全，真正关键的是鉴权、参数校验、幂等和最小暴露。

## 客户端页面什么时候容易和 SSR 页面出现冲突，可能会出现什么问题

客户端页面和 SSR 页面最容易冲突的场景，本质上是服务端首屏渲染的结果，和客户端 hydration 后计算出来的结果不一致。

常见原因有几类：
第一是页面依赖 window、localStorage、屏幕宽度、时区这类客户端独有信息；
第二是渲染用了 Date.now()、随机数、实时数据这类不稳定输入；
第三是服务端和客户端拿到的用户态、缓存数据或者 feature flag 不一致。

这类冲突常见的问题就是 hydration warning、首屏闪烁、登录态跳变、列表或价格抖动，严重一点会导致某些交互状态错乱。

我的处理思路一般是让 SSR 首屏只依赖稳定数据，把客户端独有判断延后到 hydration 之后，同时尽量保证服务端和客户端共享同一套 source of truth，比如统一登录态、实验分桶和缓存策略。

怎么避免，核心就几条：
SSR 首屏只依赖两端都稳定可得的数据
客户端独有信息放到 hydration 后再介入
保证服务端和客户端的 source of truth 一致
首屏数据和客户端缓存策略要对齐
不要在 render 里用随机值、当前时间这类不稳定输入
登录态、实验、feature flag 最好服务端先算好，再传给客户端

## RSC 解决了什么问题？它和 SSR 有什么区别？

`SSR` 的核心价值是：**首屏 HTML 先在服务端生成，再发给浏览器。**

它主要解决的是：

- 首屏白屏时间太长
- SEO 不友好
- 弱网下用户要等 JS 全下载完才看到内容
- 某些页面需要服务端拿数据再渲染

但是 `SSR` 有一个没彻底解决的问题：

**虽然 HTML 是服务端先渲的，但页面最终还是要把整棵客户端 React 组件树 hydrate 一遍。**

也就是说：

- 组件代码还是要发到浏览器
- 浏览器还是要执行很多 JS
- 数据和 UI 逻辑很多还是在客户端承担

所以 SSR 主要优化的是“先看到”，不一定优化“发多少 JS”和“客户端负担有多重”。

---

再说 `RSC`，也就是 `React Server Components`。

`RSC` 真正解决的问题是：

**不是所有组件都应该发送到客户端，也不是所有渲染逻辑都应该在浏览器执行。**

它想解决的核心痛点有几个：

1. 降低客户端 JS 体积
   很多组件本质上只是：

- 查数据
- 拼模板
- 输出 UI

这些组件其实不需要在浏览器里重新执行。  
RSC 允许它们只在服务端运行，浏览器拿结果，不拿完整组件逻辑。

1. 让数据获取更靠近组件
   以前经常是 page 级别先取数据，再一层层传 props。  
    RSC 允许组件自己在服务端拿数据，组件边界和数据边界更贴近。
2. 减少客户端的计算和 hydration 成本
   SSR 页面虽然先出了 HTML，但 hydration 仍然重。  
    RSC 可以把很多纯展示组件留在服务端，客户端只 hydrate 真正需要交互的那部分。
3. 更自然地访问服务端资源
   比如：

- 数据库
- 私有 API
- server-only secrets
- 文件系统

这类东西在 RSC 里可以直接用，不需要先绕到客户端再请求一次。

---

所以两者的本质区别可以这么理解：

`SSR`
是**服务端先把页面 HTML 渲出来**。

`RSC`
是**把组件本身划分成“只在服务端运行的组件”和“需要在客户端运行的组件”**。

也就是说：

- SSR 关注的是“渲染发生在哪儿”
- RSC 关注的是“哪些组件根本不该下发到客户端”

---

一个更直观的对比：

传统 SSR：

- 服务端把页面 HTML 渲出来
- 客户端下载整页组件 JS
- 客户端 hydrate 整页

RSC：

- 服务端渲染 server components
- 客户端只拿交互组件对应的 JS
- hydrate 范围更小

---

什么时候 RSC 特别有价值？

- 内容型页面
- 数据密集但交互不重的页面
- 后台系统、配置页、详情页这类“读多写少”场景
- 需要减少 bundle size 的复杂应用

什么时候它不是万能的？

- 强交互组件还是得是 client component
- 团队需要适应 server/client 边界
- 共享 hooks、context、第三方前端库时会有额外心智成本

---

一句话总结：

**SSR 解决的是“页面更早展示出来”；RSC 进一步解决的是“哪些组件根本没必要发到浏览器，以及如何减少客户端 JS 和 hydration 成本”。**

---

**面试回答版本**

“我会把 SSR 和 RSC 分开理解。SSR 主要解决的是首屏渲染和 SEO 问题，也就是服务端先把 HTML 渲出来，让用户更早看到内容。但 SSR 并没有消除客户端负担，因为页面最后通常还是要把对应的 React 组件 JS (需要交互的组件)下载下来并 hydrate 一遍。

RSC 解决的是另一层问题：不是所有组件都应该发送到客户端。很多组件只是读数据然后渲染 UI，这类组件完全可以只在服务端运行，不把组件逻辑下发到浏览器。这样可以减少客户端 JS 体积、降低 hydration 成本，也让数据获取更贴近组件边界。

所以我会总结成一句话：SSR 解决的是更早看到页面，RSC 解决的是更少把不必要的代码发到客户端。两者不是替代关系，而是不同层面的优化。”

## RSC 解决了什么问题？它和 SSR 有什么区别？什么场景适合 RSC，什么场景必须 CSR？简单解释

**解释版本**

可以把它简单理解成：

`SSR` 解决的是：**页面先在服务端渲出来，让用户更早看到内容。**

`RSC` 解决的是：**不是所有组件都值得发到浏览器，有些组件只在服务端运行就够了。**

所以两者差别是：

- `SSR`：重点是“先把 HTML 生成出来”
- `RSC`：重点是“哪些组件根本不用下发到客户端”

---

更直白一点：

传统 SSR：

- 服务端先生成页面 HTML
- 浏览器还是要下载整页很多 JS（JS本质是用来处理交互逻辑的，如果是纯展示的组件，就不需要JS，在服务端生成好HTML在客户端直接展示即可，所以这里下载全量的JS就会影响性能）
- 然后 hydrate

RSC：

- 服务端组件直接在服务端跑
- 浏览器只拿结果，不拿这部分组件代码
- 客户端只处理真正需要交互的部分

---

`RSC` 适合的场景：

- 读多写少的页面
- 商品详情、文章详情、后台详情页
- 主要是展示数据，交互不重
- 想减少 bundle size、减少 hydration 成本的页面
- 需要直接访问数据库、私有 API、服务端密钥的组件

`必须 CSR` 的场景：

- 要用 `useState`、`useEffect`
- 要响应点击、输入、拖拽、弹窗、动画
- 要访问 `window`、`localStorage`、`navigator`
- 强交互表单
- 图表、富交互筛选器、客户端编辑器
- 依赖很多浏览器 API 的组件

一句话记忆：

**RSC 适合“展示型、服务端数据型组件”，CSR 适合“强交互、浏览器驱动型组件”。**

---

**面试回答版本**

“SSR 主要解决的是首屏展示问题，也就是服务端先把 HTML 渲出来，让用户更早看到内容。RSC 解决的是另外一层问题：很多组件其实只是读数据然后渲染 UI，这些组件没必要把代码发到浏览器，所以可以只在服务端运行，减少客户端 JS 和 hydration 成本。

场景上，RSC 更适合数据展示型页面，比如详情页、内容页、后台读多写少页面；而必须 CSR 的通常是强交互组件，比如表单、弹窗、拖拽、图表、依赖 `window` 或本地存储的组件。简单说，RSC 偏展示和服务端数据，CSR 偏交互和浏览器能力。”

## ISR 的原理是什么？和 SSR / SSG / CSR 怎么取舍？

`ISR` 可以理解成：**先按静态页面的方式缓存起来，但过一段时间允许它在运行时重新生成。**

它的核心原理不是“每次都重新渲”，而是：

1. 页面先被静态生成出来
2. 后续请求先直接吃缓存，速度接近静态站
3. 到了 `revalidate` 时间后，下一次请求先拿旧页面
4. 同时后台触发一次重新生成
5. 新页面生成成功后，后续请求再拿新版本

Next 官方文档对 ISR 的描述很直接：它可以“更新静态内容而不需要全站重建”，并且“多数请求直接返回预渲染的静态页面”。

所以 ISR 的本质是：**静态为主，增量更新。**

---

怎么和 `SSR / SSG / CSR` 取舍？

可以这样记：

### 1. SSG

构建时就把 HTML 生成好。  
适合：

- 基本不常变的页面
- 营销页、关于页、帮助页、博客静态内容页

优点：

- 快
- CDN 友好
- 成本低

缺点：

- 内容更新要重新构建  
  官方对 SSG 的定义就是：页面 HTML 在 build time 生成，然后每次请求复用。

---

### 2. SSR

每次请求都在服务端重新生成 HTML。  
适合：

- 强依赖实时数据
- 用户态很强
- 每次请求结果都可能不同
- 个性化内容、权限内容、实时价格/库存、订单页

优点：

- 数据最新
- 首屏完整

缺点：

- 每次请求都有服务端成本
- TTFB 一般比静态方案高  
  官方定义也很明确：SSR 是 page HTML “generated on each request”。

---

### 3. CSR

先给一个很轻的 HTML，数据和主要内容靠浏览器 JS 再拉。  
适合：

- 强交互后台
- 登录后控制台
- SEO 不敏感页面
- 页面内容本来就高度依赖客户端状态

优点：

- 前后端职责清晰
- 交互灵活

缺点：

- 首屏更慢
- SEO 风险更高  
  官方文档也明确提到 CSR 可能影响 SEO，因为一些爬虫只会看到初始空态或 loading 状态。

---

### 4. ISR

介于 SSG 和 SSR 之间。  
适合：

- 内容不是实时到毫秒级，但也不能老靠重新 build
- 商品详情页
- 分类页
- CMS 页面
- 读多写少、更新频率按分钟或小时算的页面

优点：

- 大部分请求像静态页一样快
- 内容又能自动更新
- 不需要每次更新都全站重构

缺点：

- 不是强实时
- 会存在“短时间读到旧内容”的窗口

---

所以一个简单取舍思路是：

- **几乎不变**：`SSG`
- **会变，但不要求每次请求都最新**：`ISR`
- **每次请求都必须拿最新结果**：`SSR`
- **SEO 不重要、强交互很重**：`CSR`

如果让我再压缩一句：

**SSG 看静态，ISR 看更新频率，SSR 看实时性，CSR 看交互强度。**

---

**面试回答版本**

“ISR 的原理可以理解成静态页面的增量更新。页面先像 SSG 一样被预生成并缓存，后续请求大部分直接返回缓存；等到 `revalidate` 时间到了，下一次请求先拿旧页面，同时后台重新生成新版本，生成成功后再替换缓存。所以它本质上是静态为主、运行时更新。

取舍上我一般这么看：内容几乎不变的页面用 SSG；内容会更新但不要求每次请求都绝对最新的页面用 ISR，比如商品详情、CMS 页面；实时性要求高、强依赖用户态或动态数据的页面用 SSR；而 SEO 不敏感但交互很重的后台类页面更适合 CSR。简单说，SSG 看静态，ISR 看更新频率，SSR 看实时性，CSR 看交互强度。”

## Redis有什么特性？ 你们为什么要在 ISR 外再加 Redis缓存？

Redis 本质上是一个 内存型的 key-value 存储，适合做缓存。

它最关键的特性有这几个：

- 快：数据主要在内存里，读写延迟很低。
- 支持 TTL：很适合缓存场景，数据可以自动过期。
- 可共享：不是某一台应用实例本地独享，而是多个服务实例都能读同一份缓存。
- 支持持久化和高可用扩展：虽然它常被当缓存，但不是一次进程重启就全没办法管的那种简单内存对象。
- 数据结构丰富：除了 string，还有 hash、set、zset，不过我们当前这个场景主要用的是 string cache。

Redis 的核心特性我会概括成四点：快、支持 TTL(生存时间 (TTL) 是指数据包在被路由器丢弃之前可存在于网络内部的时间或“跃点”数。TTL 还用于其他上下文中，包括 CDN 缓存和 DNS 缓存。)、可被多实例共享、适合做高并发缓存。
我们在 ISR 外再加 Redis，主要不是因为 ISR 不够用，而是因为 ISR 解决的是“页面何时生成和再生”，不天然解决“多实例之间怎么共享缓存”。
如果只靠 ISR，在多 pod 部署下，每个实例可能各有一份缓存，命中率不稳定，还会出现重复 regeneration 和重复回源。
所以我们项目里是把 Next 的 cache handler 接到 Redis，上生产时走共享缓存；这样所有实例读的是同一份缓存，命中率更稳定，也能减少重复渲染和重复 API 调用。
另外我们还用 buildId + env 做 key prefix，避免新旧版本缓存串掉。Redis 挂了再 fallback 到本地 LRU，保证服务可用，但那只是兜底，不是主路径。

## Next.js 自带缓存有多大？是依赖应用内存吗？自带缓存不够吗？

Next.js 自带缓存不是无限的。按官方当前自托管文档，默认会把生成缓存放在内存和磁盘里，内存默认 50MB，但这个缓存是实例级的，不是分布式共享的。
所以如果是单实例部署，自带缓存通常够用；但如果是多 pod / Kubernetes 部署，就会有几个问题：每个实例各有一份缓存、命中率不稳定、revalidateTag 默认也不能自动同步到所有实例。
这也是为什么我们项目里把 cacheMaxMemorySize 设成 0，再通过自定义 cacheHandler 接 Redis。这样缓存就从“单机内存缓存”变成“跨实例共享缓存”，命中率和一致性都会更稳定。

## Redis 挂了怎么办？你说的 LRU 降级具体怎么做？

开发环境不走 Redis；生产走 cache-handler.js，连不上 Redis 时降级为进程内 LRU(Least Recently Used)。

LRU 是一种常见的缓存淘汰策略：

- 最近被读/写的条目排在前面
- 缓存满了就删掉最久没被用到的条目
- 目标是在有限内存里尽量保留「热数据」

假设 web 服务有 3 个 Pod：

Redis：Pod A 缓存了首页，Pod B/C 也能读到同一份
进程内 LRU：Pod A 有一份，Pod B/C 各有一份；某个 Pod 重启后它的缓存没了；ISR 再验证可能在 3 个 Pod 上各跑一遍
所以生产环境才需要 Redis；进程内 LRU 主要是兜底，保证 Redis 挂了服务仍能工作，只是缓存效果和一致性会差一些。

## 如何避免缓存击穿、缓存雪崩、缓存污染？

缓存击穿是单个热点 key 过期，大量请求同时回源；
缓存雪崩是大量 key 同时失效，或者缓存服务整体不可用；
缓存污染是key 维度设计不对，或者把不该共享的数据缓存了，命中后返回错误结果。

对应的治理方式我一般这样做：
击穿：对热点 key 做 singleflight 或分布式锁，只让一个请求回源；同时配合逻辑过期和后台刷新，优先返回旧值，避免热点瞬间打穿后端。
雪崩：TTL 做随机抖动，避免同一时间批量过期；缓存层做高可用；再加本地 LRU 这类二级缓存和限流降级，防止 Redis 挂掉后流量直接压垮源站。
污染：重点是 key 设计和边界控制。缓存 key 要带完整业务维度，比如 region、locale、currency、buildId；用户态数据和公共缓存要隔离，像 cart、coupon、member price 不能进共享缓存；另外要有版本化和精准失效机制。

如果结合我们项目，Redis 共享缓存就是在解决多实例下的雪崩和重复回源问题，buildId + env 的 key prefix 是防缓存污染，本地 LRU fallback 是缓存层故障时的降级手段。

## Webhook revalidate 如何保证准确？如果 CMS 连续发布怎么办？

结合我们项目，Storyblok webhook 不是粗暴全站清缓存，而是按依赖关系做精准失效。
普通页面按 full_slug 打 tag，blog 这种列表型内容按 content type 级 tag 失效，PLA layout 这类特殊页面再补 revalidatePath。

这样单页更新不会误伤全站，但聚合页也能被带上。这个在代码里是对齐的，因为 CMS 读取缓存时本身就挂了对应的 full_slug tag 和 content_type tag。

连续发布时，我会重点做两件事。

第一，webhook 只做 invalidate，不直接写缓存，这样顺序风险会小很多，因为真正写入缓存是在下一次请求回源时拉最新 CMS 数据。
第二，再加一层幂等和合并，比如同一个 story 在几秒内连续 publish，只保留一次失效任务；如果 webhook 带 revision 或 published_at，就只处理最新版本，旧版本直接丢弃。

所以我的原则是：
精确 tag 保证失效范围准，invalidate-only 保证不会把旧数据反写回缓存，revision 去重和短窗口合并保证连续发布时系统不抖。

## 搜索页、商品详情页、Checkout 页分别用什么渲染策略？为什么？

- 搜索页 用的是动态 SSR + CSR 交互。
  页面层明确 force-dynamic，服务端先把首屏需要的分类、筛选上下文、zipcode、DY context 准备好，再交给客户端做筛选、排序、翻页这类高交互能力。因为搜索页既要首屏可读，也要高频交互，纯 SSR 或纯 CSR 都不合适。

- 商品详情页 用的是server-first 的混合渲染，可以理解成 SSR/RSC + Suspense streaming。
  核心商品数据、SEO 信息、结构化数据在服务端先拿，保证首屏和 SEO；规格选择、加购这类交互再放到客户端。PDP 是典型既要 SEO 又要交互的页面，所以最适合这种模式。

- Checkout 页 则是CSR-first。
  外层只有一个很薄的 layout，真正的地址、运费、支付步骤基本都在客户端跑，而且取数时很多地方直接 noCache: true。因为 Checkout 是强登录态、强 session、强实时状态页面，不需要 SEO，重点是状态一致性和支付流程控制，所以更适合客户端主导。

## 你们RSC 场景下组件库有什么兼容问题？

我们在 RSC 场景下，组件库兼容问题主要有五类。
第一类是很多基础组件本身已经是 client component，比如我们自己的 Fortress 里 Button、Grid 这类基础组件顶部就是 'use client'。这样做虽然兼容性简单，但代价是 server/client 边界会上移，RSC 减少 hydration 的收益会被吃掉。
第二类是组件库里有很多浏览器依赖，比如 useMediaQuery、document.createElement、document.querySelector、表单聚焦、上传、Portal 这类能力，它们天然不能直接跑在 RSC，只能下沉到叶子 client component。
第三类是响应式策略的问题。如果组件库内部大量依赖 JS breakpoint 来决定结构，服务端拿不到真实 viewport，就容易出现 hydration mismatch 或首屏跳变。所以我们后来会更强调 CSS 响应式，像 responsive sx 这类 SSR 友好的方案。
第四类是样式系统兼容。我们基于 MUI Joy 和 Emotion，在 App Router/RSC 下必须自己做 ThemeRegistry + useServerInsertedHTML 这套样式注入，否则就会有 FOUC、className 不一致这类问题。
第五类是API 形态偏传统 CSR。像 render props、函数 props、表单控制这类老组件库 API，在 RSC 边界下不能直接传，迁移时往往要重构调用方式，而不只是加个 'use client' 就结束。

## 踩过的坑

像 CookieYes 这种会在 hydration 前直接改 DOM 的第三方脚本，会直接把 SSR/RSC 页面搞坏

虽然它不是 Fortress 组件库本身的问题，但它说明了一个事实：
RSC 场景下，任何会提前碰 DOM 的组件/脚本集成，都会放大兼容风险。
