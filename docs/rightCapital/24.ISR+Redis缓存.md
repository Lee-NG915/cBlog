我们这套 `ISR + Redis`，本质上是在解决 **Next.js 默认缓存只适合单实例，但我们生产是多 Pod 场景** 这个问题。  
如果不把缓存从本地内存搬到 Redis，多台实例之间缓存就不会共享，ISR 的收益会被打很多折，甚至出现数据不一致和重复回源。

---

## 1. 为什么 Next.js 自带缓存不够

### 先讲默认情况

Next.js 自带的 ISR / Full Route Cache，本身没问题。  
问题在于它默认更偏“当前实例可用”，也就是缓存天然更靠近单机进程。

如果你只有一台服务，这没什么问题。  
但我们线上不是单机，是多 Pod。

### 多 Pod 下会出现什么问题

假设有 3 个 Pod：

- 用户第一次打到 Pod A，A 生成并缓存页面
- 第二次用户打到 Pod B，B 本地没有缓存，又要重新生成
- 第三次用户打到 Pod C，C 也没有缓存，还得再生成一次

这样会有几个后果：

- 同一个页面会被多个 Pod 重复生成
- 缓存命中率会下降
- 回源压力会变大
- 不同 Pod 看到的新旧数据可能不一致

所以不是 Next.js 缓存“不能用”，而是 **默认缓存不适合横向扩容后的共享场景**。

### 项目里怎么体现这个判断

在 [apps/web/next.config.js](/Users/lidonghai/castlery/joyboy/apps/web/next.config.js:212) 里做了两个关键配置：

- `cacheHandler: production 下使用 ./cache-handler.js`
- `cacheMaxMemorySize: 0`

这个意思非常明确：

- 生产环境不靠默认内存缓存
- 用自定义 `cacheHandler`
- 把默认 in-memory cache 关掉，避免“本地一份、Redis 一份”造成行为不清晰

这个决策本身就说明团队已经意识到：**默认缓存层不够，需要共享缓存层**。

---

## 2. 多 Pod 场景到底要解决什么

你可以把它理解成两个问题：

### 问题一：缓存要共享

A Pod 生成的页面，B Pod 也能直接用。  
这就是 Redis 的第一价值。

### 问题二：缓存失效要一致

如果 CMS 内容更新了，只清掉 A Pod 的缓存没意义，必须所有 Pod 都感知到失效。  
这就是 Redis + `revalidateTag/revalidatePath` 的第二价值。

所以 Redis 在这里不是“为了快”，更重要是为了：

- **共享**
- **一致**
- **减少重复生成**

## 3. Redis key 怎么设计

这块项目里最直接的实现是在  
[apps/web/cache-handler.js](/Users/lidonghai/castlery/joyboy/apps/web/cache-handler.js:1)

这里 Redis handler 的 key prefix 是：

```js
keyPrefix: `${process.env.APPLICATION_ENV}-${buildId}:`
```

### 这个设计有什么含义

它至少解决了两个问题：

#### 1. 环境隔离

比如 test / uat / prod 不能混缓存。  
`APPLICATION_ENV` 把不同环境隔开。

#### 2. 构建版本隔离

`buildId` 会让不同版本发布后天然切换到新的缓存命名空间。  
这样可以避免：

- 新代码读到旧结构缓存
- 回滚 / 发布后缓存格式不兼容

所以这类 key 设计的核心原则不是“越短越好”，而是：

- 环境可隔离
- 版本可隔离
- 避免脏缓存污染

## 4. revalidate 是怎么触发的

项目里其实有两种触发方式。

---

### 第一种：时间驱动

也就是常见的 ISR 方式。

比如：

- [apps/web/app/[deviceTheme]/[region]/[locale]/home/page.tsx](/Users/lidonghai/castlery/joyboy/apps/web/app/[deviceTheme]/[region]/[locale]/home/page.tsx:13)  
`export const revalidate = 600;`
- blog、help-center、contact-us 这些页面也有类似配置
- Storyblok 服务里很多 fetch 也带 `revalidate`

这种适合：

- 内容型页面
- 更新频率不高
- 可以容忍几分钟延迟

---

### 第二种：事件驱动 / 主动失效

这是更关键的。

项目里有 Storyblok webhook 路由：  
[apps/web/app/api/storyblok/route.ts](/Users/lidonghai/castlery/joyboy/apps/web/app/api/storyblok/route.ts:1)

这里在 CMS 内容变更后会调用：

- `revalidateTag(...)`
- `revalidatePath(...)`

比如：

- layout 改了，会 revalidate 对应 path
- blog 内容改了，会 revalidate 某类 tag
- 普通 Storyblok 内容更新，会按 `full_slug` 去 revalidate tag

### 为什么要 tag

因为很多缓存不是单个页面，而是“按内容资源组织的”。

比如在 Storyblok service 里：  
[libs/modules/cms/services/src/lib/sbBaseService.ts](/Users/lidonghai/castlery/joyboy/libs/modules/cms/services/src/lib/sbBaseService.ts:147)

能看到它缓存时会打这些 tags：

- `storyblok`
- `sb-all-${content_type}`
- `cacheKey`
- 外部传入 tags

这意味着后面 CMS 内容变更时，可以精准失效，而不是整站清缓存。

## 5. Redis 故障怎么降级

#### 1. 先尝试连 Redis

如果连得上，就用 Redis handler。

#### 2. 连不上就退回本地 LRU

这里直接 fallback 到：

```js
createLruHandler()
```

也就是说：

- 最优方案：共享 Redis 缓存
- 退化方案：当前实例本地内存 LRU

虽然降级后多 Pod 不共享了，但至少服务还能跑，不至于因为 Redis 异常直接影响整站可用性。

### 这里还有几个很细的防坑设计

#### `timeoutMs: 1000`

Redis handler 设置了超时，避免 Redis 卡住时把页面请求也拖死。

#### `reconnectStrategy`

只有 ready 后才允许持续重连，避免冷启动阶段无限重连导致服务卡住。

#### 启动失败会主动 disconnect

如果连接失败，不让 Redis client 一直无意义重试。

这些都很工程化，说明团队考虑过“缓存系统本身不能反过来拖垮业务”。

## 6. 如何防缓存击穿 / 缓存雪崩

---

### 第一层：避免同一时刻所有请求都变成硬 miss

项目里有个非常关键的设计：

在 `cache-handler.js` 里：

```js
ttl: {
  estimateExpireAge: (staleAge) => 3600,
}
```

这个值很重要。

### 为什么重要

业务上的 `revalidate` 可能只有 600 秒，但物理 Redis key 没有在 600 秒就立刻删掉，而是保留更久，比如 3600 秒。

这样做的好处是：

- 600 秒后，缓存只是“变 stale”
- 但 Redis 里还有旧值
- 系统可以先返回 stale，再触发再生
- 不会一下子所有请求都打成彻底 miss

这本质上就是在防止缓存击穿。

如果你把物理过期时间设置得和业务 revalidate 一样，一到点缓存就直接消失，多 Pod 和高并发下很容易瞬间回源。

---

### 第二层：请求级去重

项目里在某些数据层还做了“请求级去重”。

比如 PDP Data Bucket：  
[libs/modules/cms/domain/src/lib/api/server-api/pdp-data.ts](/Users/lidonghai/castlery/joyboy/libs/modules/cms/domain/src/lib/api/server-api/pdp-data.ts:1)

这里用了两层缓存：

- `unstable_cache` 做持久缓存
- `react cache` 做同一请求内的去重

这解决了一个很实际的坑：

同一个页面渲染过程中，`generateMetadata`、页面主体、某些 server component 可能会重复读同一份数据。  
如果不去重，同一个请求里都可能打多次后端。

所以这里不仅要防“跨请求击穿”，也要防“单次渲染内部重复取数”。

---

### 第三层：对热点接口有专门的本地缓存策略

虽然这不完全属于 ISR + Redis，但项目在搜索里也用了类似思想。  
在 [docs/search-result-cache.md](/Users/lidonghai/castlery/joyboy/docs/search-result-cache.md:1) 里能看到：

- 结果缓存
- 进行中请求复用

它本质上也是在防止短时间同类请求把后端打爆。

## 7. 哪些页面不能缓存

这个问题不能简单回答“动态页面都不能缓存”，更准确的说法是：

**不能做 Full Route Cache，或者不能缓存核心数据的页面，通常有这些特征：**

- 强个性化
- 强实时性
- 跟用户态强相关
- query 组合过多、变化频繁
- 依赖 cookie / location / auth / inventory / payment state

### 项目里几个典型例子

#### 1. Search / PLP / Collections / Sales

很多是 `force-dynamic`

比如：

- [apps/web/app/[deviceTheme]/[region]/[locale]/(PLP)/search/page.tsx](/Users/lidonghai/castlery/joyboy/apps/web/app/[deviceTheme]/[region]/[locale]/(PLP)/search/page.tsx:7)
- [apps/web/app/[deviceTheme]/[region]/[locale]/(PLP)/categories/[...permalink]/page.tsx](/Users/lidonghai/castlery/joyboy/apps/web/app/[deviceTheme]/[region]/[locale]/(PLP)/categories/[...permalink]/page.tsx:11)

因为这些页面：

- 筛选组合太多
- 排序、分页、query 变化频繁
- 还可能跟推荐系统、动态投放联动
- 很容易造成缓存碎片化或命中率很差

所以这里更倾向 dynamic，而不是硬上 ISR。

> 在 Next.js App Router 里，页面默认会尽量静态化，也就是说如果不声明 force-dynamic，框架会根据数据获取方式判断能不能走缓存。
> 但不是说没写 force-dynamic 就一定缓存整个页面，因为页面内部某些 fetch 可能是 no-store，某些数据也可能单独用了 revalidate 或 unstable_cache。所以我们实际理解是：force-dynamic 是强制动态，而不声明时 Next 会自动推断，最终缓存行为取决于页面和数据层的组合。
> 缓存粒度其实有好几层：
>
> - 页面 route segment 级别
> - fetch 请求级别
> - unstable_cache / cache 级别

#### 2. 产品实时数据

产品页不是所有内容都缓存。

像产品 API 在  
[libs/modules/product/domain/src/api/product.api.ts](/Users/lidonghai/castlery/joyboy/libs/modules/product/domain/src/api/product.api.ts:296)

直接用了：

```ts
cacheOption: 'no-store'
```

因为产品详情往往受这些因素影响：

- zipcode
- city
- region
- 库存 / 配送 / lead time
- 个性化配置

这些数据如果缓存错了，用户看到的就可能不是自己真实可买的结果。

#### 3. checkout / payment / cart / account 这类用户态页面

这类页面通常不适合做页面级共享缓存，因为天然和当前用户会话强绑定。

即使外层壳子能静态一点，核心数据也必须是：

- 用户私有
- 实时
- 不可共享

注意，像  
[apps/web/app/[deviceTheme]/[region]/[locale]/wishlist/page.tsx](/Users/lidonghai/castlery/joyboy/apps/web/app/[deviceTheme]/[region]/[locale]/wishlist/page.tsx:6)  
虽然写了 `revalidate = 600`，但这里更多像是页面壳和 metadata 层的缓存；真正用户的 wishlist 数据，通常不会靠共享 ISR 来做。

所以面试里最好说：

“有些页面 route shell 可以缓存，但用户核心数据不能共享缓存。”

## 8. 这套方案里踩过什么坑，以及怎么解决

我帮你总结几个最像真实工程里的坑。

### 坑一：以为 Next.js 默认 ISR 上线就够了

实际多 Pod 下会发现：

- 每个实例各自回源
- 缓存命中率不稳定
- 实例间不一致

**解决：**
上 Redis 共享缓存，自定义 `cacheHandler`，并关掉默认内存缓存。

---

### 坑二：Redis 挂了以后整站性能抖动

如果 Redis 是强依赖，Redis 不稳定时页面请求会被拖住。

**解决：**

- Redis 超时控制
- 启动失败主动断开
- fallback 到本地 LRU
- 服务继续可用，只是退化成非共享缓存

---

### 坑三：缓存刚过期，大量请求一起回源

这就是典型缓存击穿。

**解决：**

- 物理过期时间大于业务 stale 时间
- 让 stale 数据还在 Redis 里
- 优先 stale-while-revalidate，而不是直接硬 miss

---

### 坑四：内容更新了，但页面还不刷新

只靠定时 `revalidate`，CMS 改完可能要等几分钟才生效。

**解决：**

- Storyblok webhook
- `revalidateTag`
- `revalidatePath`
- 按资源 tag 精准失效，而不是全站清理

---

### 坑五：有些数据不该缓存却被缓存了

比如 location-sensitive 的 PDP 数据、搜索结果、checkout 状态。

**解决：**

- 对强实时/强个性化数据显式 `no-store`
- 对 PLP/search 这类页面直接 `force-dynamic`
- 不把“缓存”当默认，而是按场景选

---

## 9. 面试里怎么回答比较合适

你可以这样答：

“我们做 `ISR + Redis` 的核心原因，是 Next.js 默认缓存更适合单实例，但我们线上是多 Pod 场景。如果还只用本地缓存，同一个页面会在不同 Pod 上重复生成，缓存命中率和数据一致性都会比较差。

所以我们在生产环境里接了自定义 `cacheHandler`，把 Next 的缓存后端换成 Redis，同时把默认 in-memory cache 关掉。Redis key 前缀会带环境和 buildId，这样可以做环境隔离和版本隔离。内容失效一部分靠 `revalidate` 定时更新，一部分靠 Storyblok webhook 主动触发 `revalidateTag` 和 `revalidatePath`。

Redis 故障时我们不会让服务直接挂掉，而是自动降级到本地 LRU。为了防缓存击穿，我们让物理 TTL 大于业务 stale 时间，这样缓存过期后还能先返回 stale，再后台再生，而不是让所有请求一起回源。至于不能缓存的页面，主要是 search、PLP、checkout、payment 这类强实时或强个性化页面，它们要么直接 force-dynamic，要么核心数据显式 no-store。”
