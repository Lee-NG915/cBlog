# Payment Error Classification

Reference for `classify-payment-error.ts`. Each enum category maps to `paymentProcessingError.<category>` in the error i18n resource, except order-status categories, which are consumed by payment wallet special modal branches.

## Handling Summary

1. Payment result failures show the fixed payment-failed title, category-specific explanatory text, and raw third-party `failureCode` / `failureInfo`.
2. `usePaymentErrorHandler` calls `classifyPaymentError`, resolves `paymentProcessingError.<category>`, and writes the modal `PaymentError`.
3. `payment_intent_authentication_failure` is special-cased as `threedsAuthError` for the 3DS authentication failed popup.
4. `paypalError` is a dedicated PayPal-only category. Identification covers both entry points and both paths produce the same field shape (`failureCode` + `errorMessage` as `[SHORT]` + `failureInfo` as `[LONG]`) so the modal renderer can treat them uniformly:

   - **BFF path**: failure surfaces with `failureCode === '10703046'` (`ErrPayPalShortError`). BFF already wraps PayPal `[SHORT]` text into `errorMessage` and `[LONG]` text into `failureInfo`. No FE-side normalization needed.
   - **PayPal JS SDK path**: failure surfaces from `paypal-payment-element` `onError` with `failureCode === 'PAYPAL_ERROR'`. The JS SDK's `Error` object is **not** a PayPal `[SHORT]` text — it is a developer-level message (e.g. `"Window closed before response"`). To match the PRD row 19 modal template, the SDK callback **normalizes** the fields:
     - `errorMessage` → fixed `[SHORT]` text (default `"Payment was interrupted."`, awaiting final PRD copy) — placeholder constant exported from the PayPal element module.
     - `failureInfo` → `String(err)` (the raw SDK error string) so Sentry / DataDog / the modal's `[LONG]` slot still surface the technical detail for debugging.
   - This requires extending the SDK `onError` callback contract (`paypal-payment-element.tsx`) and the `onSdkError` callback in `use-payment-execution.ts` to include an optional `failureInfo?: string` field.

   The FE does **not** maintain a local enumeration of PayPal NVP/SOAP codes (reference: <https://developer.paypal.com/api/nvp-soap/errors/>); two synthetic codes (`10703046`, `PAYPAL_ERROR`) are sufficient. Modal text follows PRD row 19 template: `<[SHORT] from PayPal> You can make the payment again from the order history.`

5. `paymentProcessingError` (new) is the **dedicated** category for PRD Failure Reason row 14 "Payment Processing Failures". Stripe / 2C2P codes previously bucketed as fallback are now classified into this category and resolve to `paymentProcessingError.paymentProcessingError` i18n copy.
6. `genericPaymentError` is **only** the last-resort fallback when an unmatched code (or empty code) is encountered. Modal text follows the PRD fallback template: `Oops, your payment didn't go through. Please retry the payment from your order history.`
7. `paymentSuccessOrderCanceled` is a BFF-only category for code `10703048` (`ErrPaymentSuccessButOrderCanceled`) — third-party payment succeeded but the BFF subsequently canceled the order. Modal text is customized to inform the user that a refund will be processed (final copy TBD pending PRD update).

## Classifier Priority

Steps are executed top-to-bottom; first match wins.

1. Order status codes -> `orderExpired` / `orderCanceled` / `paymentSuccessOrderCanceled`
2. `payment_already_paid` -> `orderAlreadyPaid`
3. PayPal entry codes (`10703046` / `ErrPayPalShortError` BFF; `'PAYPAL_ERROR'` PayPal JS SDK) -> `paypalError`
4. `PENDING_PAYMENT_CODES` -> `paymentPending`
5. `CANCELED_OR_EXPIRED_CODES` -> `canceledOrExpired`
6. Server/client action codes -> `serverError` / `clientError`
7. HTTP status fallback: `4xx` -> `clientError`, `5xx` -> `serverError`
8. Provider-specific code groups in the order below
9. `PAYMENT_PROCESSING_CODES` (PRD row 14 Stripe + 2C2P codes) -> `paymentProcessingError`
10. Fallback -> `genericPaymentError`

Notes:

- PayPal step (3) runs before `paymentPending` / cancellation buckets to guarantee PayPal short-error copy is shown even if a numeric subcode would otherwise collide.
- `paymentProcessingError` (9) runs **after** all provider-specific buckets so a code that has a more specific category still wins; only PRD row 14 codes with no specific bucket fall here.
- `genericPaymentError` (10) is reserved for codes that match nothing at all (empty code, unknown provider code, etc.).

## Generic Fallback Behaviour

Only truly unmatched codes (empty `failureCode`, unknown provider codes, codes that the BFF has not bucketed) fall through to `genericPaymentError`. The modal copy is fixed:

> Oops, your payment didn't go through. Please retry the payment from your **order history**.

"order history" MUST be rendered as a clickable link (route: `/account/orders`, same as other error messages that reference order history). Implement via i18n placeholder (e.g. `<orderHistoryLink>order history</orderHistoryLink>`) and resolve the link in the modal renderer — do not hard-code the anchor inside the translation string.

BFF code `10703030` (`ErrPaymentFailedGeneric`) is the BFF-side representative of this fallback.

## Implementation Plan (FE)

The following code-side changes are required after this plan is approved. Listed here for traceability; **no code is changed in this commit.**

1. `classify-payment-error.ts`
   - Add two new entries to `PaymentErrorCategory`: `'paypalError'`, `'paymentProcessingError'`.
   - Add a new constant `PAYPAL_ERROR_CODES = new Set([String(TransactionApiErrorCode.ErrPayPalShortError), 'PAYPAL_ERROR'])` — first entry covers the BFF path, second covers the PayPal JS SDK `onError` synthetic code. PayPal NVP/SOAP codes themselves are intentionally not enumerated on the FE.
   - Add a new constant `PAYMENT_PROCESSING_CODES` containing the Stripe + 2C2P codes from the "Payment Processing Codes (PRD Row 14)" table, plus `String(TransactionApiErrorCode.ErrPaymentProcessingFailure)`.
   - In `classifyPaymentError`:
     - After the order-status / `orderAlreadyPaid` checks (current step 2), add a PayPal short-circuit: `if (PAYPAL_ERROR_CODES.has(code)) return 'paypalError';`.
     - Before the final `return 'genericPaymentError'`, add: `if (PAYMENT_PROCESSING_CODES.has(code)) return 'paymentProcessingError';`.
   - Update the JSDoc block at the top of `classifyPaymentError` to reflect the new priority list.
2. `classify-payment-error.spec.ts`
   - Add specs: `10703046` → `paypalError`; `PAYPAL_ERROR` → `paypalError`; `10703042` → `paymentProcessingError`; each Stripe / 2C2P code from the new set → `paymentProcessingError`; an unrecognised code → `genericPaymentError`.
3. `paypal-payment-element.tsx` — SDK error normalization (Plan B)
   - Export a placeholder constant (e.g. `PAYPAL_SDK_SHORT_FALLBACK = 'Payment was interrupted.'`). Final copy TBD with PRD; keep the constant in one place so future copy changes are one-line edits.
   - Change the SDK `onError` callback to:
     ```ts
     onError={(err) => {
       pendingRef.current = null;
       if (initiateFailedRef.current) {
         initiateFailedRef.current = false;
         return;
       }
       onError({
         failureCode: 'PAYPAL_ERROR',
         errorMessage: PAYPAL_SDK_SHORT_FALLBACK, // → [SHORT]
         failureInfo: String(err),                // → [LONG]
       });
     }}
     ```
   - Extend `PaypalPaymentElementProps['onError']` signature: `(error: { failureCode?: string; errorMessage: string; failureInfo?: string }) => void`.
   - Apply the same `failureInfo` extension to the inline `onError({ errorMessage: 'Payment initiation data is missing' })` call (failureInfo can remain undefined here; that path is an integration bug, not a PayPal error).
4. `use-payment-execution.ts` — propagate `failureInfo`
   - Extend `onSdkError` signature to accept optional `failureInfo` and forward it to `handlePaymentError`.
   - Verify other SDK error callers (Affirm element, etc.) compile — adding an optional field is backwards-compatible but worth a build check.
5. i18n keys under `paymentProcessingError.*` in the error i18n resource
   - Add `paypalError` with template `{{shortMessage}} You can make the payment again from your <orderHistoryLink>order history</orderHistoryLink>.` (`{{shortMessage}}` resolves to `errorMessage`).
   - Add `paymentProcessingError` mirroring whatever copy the PRD specifies for row 14 (default to the current generic copy if PRD has not differentiated). Same `<orderHistoryLink>` slot if copy references order history.
   - Update `genericPaymentError` copy to `Oops, your payment didn't go through. Please retry the payment from your <orderHistoryLink>order history</orderHistoryLink>.`
6. `usePaymentErrorHandler` (or equivalent modal-text resolver)
   - For `paypalError`, feed `errorMessage` into the `{{shortMessage}}` slot. Render `failureInfo` in the modal's `[LONG]` / error info area (same slot used by BFF `[LONG]`).
   - For `genericPaymentError`, no longer render `failureCode` / `failureInfo` lines (PRD fallback only contains the fixed copy).
   - Resolve `<orderHistoryLink>` to an `<a>` / `Link` pointing at `/account/orders` (or the existing constant used by other error messages that link to order history — reuse, don't introduce a new path).
7. Markets / regression
   - Verify SG / AU / CA / US / UK all surface the new copy; PayPal availability differs per market, so PayPal modal needs at least an SG/AU smoke test.
   - Cover both PayPal entry points in QA: (a) BFF returns `10703046` (mock failed initiate, expect BFF `[SHORT]` from `errorMessage`), (b) PayPal JS SDK `onError` fires (cancel popup / force network error, expect normalized `[SHORT]` constant + raw SDK error in `failureInfo`).
