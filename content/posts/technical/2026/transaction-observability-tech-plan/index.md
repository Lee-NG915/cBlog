# 交易链路可观测性建设：从故障定位到实时预警

## 一、背景与痛点

### 1.1 交易链路的重要性

- checkout / payment 作为核心业务闭环
- 故障直接等同于营收损失

### 1.2 现有监控的不足

- Sentry 与 Datadog 已完成平台级初始化
- 但缺乏交易语义层的统一观测模型
- 人工翻日志定位问题效率低下

## 二、方案目标

### 2.1 核心收益定义

- 按交易 ID 串联前后端上下文
- 系统故障、业务失败、第三方波动的精准分诊
- 基于成功率的实时告警体系
- 发布后 30 分钟内回归检测

## 三、统一事件模型设计

### 3.1 事件 schema 定义

- 交易生命周期内的关键事件节点
- 上下文字段标准化（userId, orderId, sessionId 等）

### 3.2 埋点与采集机制

- 前端 SDK 嵌入
- 后端中间件拦截与日志输出

## 四、前后端上下文串联

### 4.1 Trace ID 透传

- 从用户点击到支付回调的全链路追踪
- 跨服务边界（Gateway / BFF / Core Service）的传递规范

### 4.2 数据关联查询

- Sentry issue 与 Datadog trace 的关联
- 单交易维度的时序事件视图

## 五、故障分诊机制

### 5.1 故障分类体系

- 系统故障（500 / 超时 / 资源耗尽）
- 业务失败（库存不足 / 地址异常 / 风控拦截）
- 第三方波动（支付网关异常 / 物流接口超时）

### 5.2 自动化分诊规则

- 基于错误码与响应模式的分类逻辑
- 分诊结果的通知路由（On-call 响应组）

## 六、告警与回归检测

### 6.1 告警体系设计

- 成功率阈值与滑动窗口配置
- 告警分级（P0 / P1 / P2）与升级策略

### 6.2 发布回归检测

- 发布前后成功率对比基线
- 30 分钟观察期与自动回滚触发条件

## 七、落地成果与优化方向

- 平均故障定位时长（MTTR）变化
- 告警信噪比优化
- 后续向全链路追踪（OpenTelemetry）的演进
