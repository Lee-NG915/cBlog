# 技术文档迁移（去企业化 → 个人博客）

将企业内部技术文档迁移至个人博客时，必须遵循以下规范，确保文档**通用、可复制、体现个人专业能力，且不暴露任何企业项目痕迹**。

## 核心原则

1. **彻底去品牌化**：删除所有企业名称、品牌名、商标、内部项目代号。
2. **彻底去项目痕迹化**：禁止出现任何真实目录名、文件路径、CI 工作流内部名称、Git 分支命名规范、内部包名、仓库组织名、真实密钥或 Token。
3. **场景通用化**：将企业专属业务场景抽象为行业通用场景（如跨境电商、SaaS、中台系统）。
4. **人称个人化**：突出个人在技术方案中的角色——主导者、推进者、核心实现者。
5. **技术细节保留**：保留架构思路、数据指标、技术选型对比、实现原理，这是文档的价值所在。
6. **可复制性**：方案应让读者感觉可以套用到自己的项目中，而非依赖某个特定企业的内部环境。

## 绝对禁止清单

以下类型的内容**绝对禁止**出现在迁移后的博客中：

- 企业名称、品牌名、产品商标
- 内部项目代号（无论大小写）
- 真实文件目录路径（如 `libs/xxx/.storybook`、`apps/xxx`）
- 真实文件名（尤其是内部定制的配置文件名）
- CI/CD 工作流内部名称、Pipeline 名称、Job 名称
- Git 分支命名规范（如 `master_brand_refresh_web`、`release-please--branches--master`）
- 内部 npm scope、包名、模块路径（如 `@company/xxx`）
- 真实密钥、Token、Project ID、Bucket 名、Region 名等基础设施标识
- 内部服务名、数据库名、缓存实例名
- 企业专属的工具平台名（除非该技术方案的核心就是介绍这个开源工具）

## 去企业化映射表

| 原表述 | 迁移后表述 | 说明 |
|--------|-----------|------|
| 企业品牌名 | 删除，改为"某跨境电商品牌"或直接去除 | 绝对禁止出现 |
| 内部项目代号 | 企业级电商前端平台 / 核心电商系统 / 组件库 | 统一替换为通用技术定位 |
| 消费者端产品名 | 消费者端电商平台 / C端电商系统 | 保留技术含义，去除代号 |
| 门店端产品名 | 门店端 / 导购端 / O2O线下系统 | 根据上下文选择 |
| 组件库品牌名 | 企业级电商组件库 / 内部设计系统 | 去除品牌名，保留组件库定位 |
| Storyblok / Knight 等内部服务 | CMS系统 / 商品数据源 / 内部BFF服务 | 抽象为通用服务类型 |
| 真实目录路径（如 `libs/fortress/.storybook`） | 组件库包内 / Storybook 配置目录 | 用通用位置描述替代 |
| CI 工作流名称 | 品牌刷新分支的 CI 流程 / 自动化发布流水线 | 用通用概念描述替代 |
| Git 分支规范 | 版本发布分支 / 功能开发分支 | 用通用分支类型描述替代 |
| 内部 npm scope（如 `@castlery/fortress`） | 组件库包 / 内部私有包 | 用通用包概念替代 |
| 真实密钥/Token | `[YOUR_TOKEN]` 或直接删除 | 绝对禁止出现 |
| "我们团队" / "公司决定" | "我主导推进" / "我负责设计" / "团队共识" | 突出个人角色 |
| 具体国家/市场名（如 US/SG/AU） | 多国家市场 / 海外区域 | 除非技术必要性 |

## 人称与角色规范

**必须体现个人主导的项目**（用户明确声明）：
- 组件库重构 → **完全由我主导并推进**

**其他项目的人称转换**：
- 如果是架构方案的发起人或核心撰写者 → "我提出并主导了..."
- 如果是核心开发者 → "我负责设计与实现..."
- 如果是参与者 → "我参与了...，主要负责..."
- 避免使用"我们""公司"等模糊集体主语

## 代码示例规范

代码示例必须遵循以下规则：

1. **import 路径**：禁止引用企业内部包或内部目录结构。应使用通用包名或相对路径占位符。
   - ❌ `import { Button } from '@castlery/fortress'`
   - ✅ `import { Button } from '@company/design-system'`
   - ✅ `import { Button } from '../components/Button'`

2. **配置文件名**：通用的配置文件名（如 `vite.config.ts`、`tsconfig.json`）可以保留；企业内部定制的配置文件名必须改为通用描述。
   - ❌ "在 `vite.config.storybook.ts` 中配置"
   - ✅ "在 Storybook 的 Vite 构建配置中调整"

3. **环境变量与密钥**：代码示例中出现的环境变量应使用通用占位符。
   - ❌ `CHROMATIC_PROJECT_TOKEN: chpt_xxx`
   - ✅ `CHROMATIC_PROJECT_TOKEN: ${{ secrets.CHROMATIC_TOKEN }}`

4. **CI/CD 配置示例**：如果需要在文中展示 CI 配置，应使用通用化的 YAML 结构，删除企业专属的工作流名称、分支名、Bucket 名等。

## 内容结构调整

### 保留的内容
- 技术问题定义与分析
- 架构设计与方案对比
- 性能指标与数据
- 实现原理与核心思路
- 踩坑记录与复盘

### 删除的内容
- 企业内部组织架构
- 非技术性的项目管理细节（如会议记录、审批流程）
- 企业专属工具/平台名（除非有技术学习价值）
- 具体的业务运营数据（GMV、订单量等敏感数据）
- 任何真实项目痕迹（目录、路径、工作流名、分支名）

### 替换的内容
- 企业背景 → 行业通用背景（跨境电商、DTC品牌、O2O零售）
- 内部代号 → 通用技术术语
- 集体成果 → 个人技术贡献与思考
- 真实项目痕迹 → 通用技术描述
